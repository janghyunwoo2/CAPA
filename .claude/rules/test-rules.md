# 테스트 규칙 (Test Rules)

## 테스트 결과 문서화 규칙

테스트(pytest 자동화 또는 수동)를 실행한 후, 반드시 해당 테스트 계획서에 대응하는
결과 문서(`*-test-result.md`)에 결과를 기록한다. 결과 생략이나 구두 요약으로 대체하는 것은 금지한다.

### 결과 기록 테이블 형식 (필수)

테스트 케이스별 결과는 반드시 아래 테이블 형식으로 기록한다:

| TC | Step | 스텝 역할 | 인풋 | 아웃풋 (실제값) | assert 단언 | 판정 | 왜 이렇게 나왔나 |
|----|------|-----------|------|----------------|-------------|------|-----------------|
| TC-XX-01 | Step N | 의도 분류 등 | 질문 또는 요청값 | status=200, intent="DATA_QUERY" 등 | `assert status == 200` | ✅ PASS | 정상 분류 |
| TC-XX-01 | Step N | ... | ... | error_code="SQL_GENERATION_FAILED" | `assert error_code in (...)` | ❌ FAIL | LLM 타임아웃 발생 |

### 컬럼 정의

- **TC**: 테스트 케이스 ID (예: `TC-A-01`, `TC-EX-05`)
- **Step**: 파이프라인 단계 번호 (예: `Step 1`, 해당 없으면 `-`)
- **스텝 역할**: 해당 Step이 하는 일 (예: 의도 분류, SQL 생성, Redash 실행)
- **인풋**: 해당 Step에 넘긴 실제 입력값 (질문 문자열, 환경변수 값 등)
- **아웃풋 (실제값)**: 실제로 받은 응답 (status code, error_code, body 핵심 필드)
- **assert 단언**: 검증에 사용한 조건식 (코드 그대로 또는 요약)
- **판정**: `✅ PASS` 또는 `❌ FAIL`
- **왜 이렇게 나왔나**: PASS면 정상 동작 이유, FAIL이면 원인 분석

### 적용 규칙

1. 테스트 실행 완료 후 결과 문서 업데이트 없이 대화를 종료하지 않는다.
2. `@pytest.mark.skip`으로 건너뛴 케이스는 "수동 테스트 필요" 또는 수동 결과를 별도 기재한다.
3. FAIL 케이스는 원인과 수정 방향을 "왜 이렇게 나왔나" 컬럼에 반드시 기재한다.
4. 결과 문서가 없으면 테스트 계획서(`*-test-plan.md`) 기준으로 신규 생성한다.
5. 테스트 결과는 결과 문서 기록과 함께, 대화창에도 동일한 테이블 형식으로 출력한다.
