# Report Generator vs Redash 공존 논의 기록

**작성일**: 2026-03-04
**목적**: CAPA 플랫폼의 보고서/대시보드 아키텍처 결정

---

## 1. 시작점: ANALYSIS_REPORT.md 검토

팀 에이전트가 작성한 분석 보고서에서:
- **추천**: Report Generator + Redash 둘 다 유지, 역할 분리
- **Report Generator**: 정기 공식 보고서 (온디맨드)
- **Redash**: 일상 모니터링 (자동 갱신)

---

## 2. 제기된 의문점

### 2.1 그래프 겹침 문제
```
Report Generator의 보고서
  - 일별 트렌드 그래프
  - 카테고리별 성과 시각화
  - Shop별 성과 테이블

Redash 대시보드
  - 일별 트렌드 그래프
  - 캠페인별 성과 시각화
  - Shop별 순위 테이블

→ 같은 데이터/그래프 중복 제공
```

### 2.2 기술 중심 vs 실무 중심
- 보고서는 "데이터가 있으니까" 기준으로 설계됨
- 실제 마케팅 실무에서 필요한 것이 뭔지 불명확
- 마케터의 관점이 부족

### 2.3 토이 프로젝트의 특수성
- 실제 회사가 아니어서 마케터와 상의 불가
- 데이터베이스 구조도 대충 구성되어 있음
- "뭐가 맞는지" 판단 기준이 없음

---

## 3. 발견 사항

### 3.1 순환 논리 문제
```
아키텍처 결정 (Report Generator + Redash 공존?)
  ↓ (결정하려면)
실제 마케팅 팀의 요구사항 파악
  ↓ (근데)
토이 프로젝트라서 마케터가 없음
  ↓ (그래서)
뭐부터 시작해야 할지 불명확
```

### 3.2 해결 방법: 역순서 접근
```
일반적 접근: 데이터 설계 → 분석 설계 → 구현

토이 프로젝트 최적 접근:
  1. 실제 광고 플랫폼/마케팅 보고서 사례 조사
  2. "마케터들이 실제로 필요한 것" 파악
  3. 거기서 필요한 데이터/지표 역산
  4. DB에 필요한 것만 추가
  5. 아키텍처 결정
```

---

## 4. CAPA의 실제 목적 명확화

### 4.1 광고주 서비스 여부
- **아니다**: CAPA는 광고주(가게 사장)에게 개별 보고서 제공하는 서비스 아님

### 4.2 실제 목적
- **맞다**: 내부 마케팅팀/경영진을 위한 **전체 광고 플랫폼 성과 분석**
- 배달앱에 노출된 광고의 로그를 수집/분석
- 내부 의사결정을 위한 데이터 제공

---

## 5. 다음 단계: 시나리오 정의

### 목표
CAPA 내부에서 이 분석 플랫폼을 **어떻게 사용하는가**를 구체적인 시나리오로 정의

### 시나리오 작성 기준
각 시나리오에 포함할 것:
- 누가 (역할/직급)
- 언제 (매일/주/월)
- 뭘 확인하고
- 그걸로 뭘 결정하는가

### 예시
```
[시나리오 1] 매일 아침 마케팅팀장의 KPI 모니터링
- 사용자: 마케팅팀장
- 시간: 매일 아침 9시
- 확인: 어제 광고 매출, CTR, 비용
- 결정: 입찰가 조정, 캠페인 일시중지 여부

[시나리오 2] 월말 경영진 보고
- 사용자: CEO
- 시간: 월말
- 확인: 월간 광고 매출, ROI, 카테고리별 성과
- 결정: 광고 예산 조정, 다음 달 전략
```

## 6. 현재 상황 정리

| 항목 | 상태 |
|---|---|
| **아키텍처 결정** | 보류 중 (시나리오 정의 필요) |
| **데이터 설계** | 임시 상태 (필요 항목 파악 후 조정) |
| **Report Generator** | 기본 구현 완료, 버그 수정 필요 |
| **Redash** | 배포 완료, 쿼리/대시보드 미구성 |
| **마케터 요구사항** | 미정의 (시나리오로 대체) |

## 7. 진행 방향

### Phase 1: 시나리오 정의 ← **현재 단계**
- CAPA에서 이 플랫폼을 어떻게 쓸지 3-5개 시나리오 작성

### Phase 2: 필요 항목 파악
- 각 시나리오를 위해 필수/선택 지표 정의
- 필요한 데이터 컬럼 확인

### Phase 3: 아키텍처 최종 결정
- Report Generator vs Redash 공존 여부 결정
- 각각의 역할 명확화

### Phase 4: 구현
- DB 구조 조정
- 쿼리/보고서/대시보드 구성
