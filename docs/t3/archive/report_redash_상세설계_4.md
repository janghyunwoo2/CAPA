# CAPA 광고 분석 플랫폼 - 상세 설계 명세서

**작성일**: 2026-03-05
**버전**: 1.0
**상태**: 초안

---

## 1. 프로젝트 개요

### 1.1 목적
배달앱 광고 성과를 분석하고 모니터링할 수 있는 플랫폼 - 팀의 누구나 쉽게 접근 가능

### 1.2 범위
- 비개발자도 데이터에 접근할 수 있는 3가지 도구 제공
- 실시간 모니터링부터 정기 보고까지 전방위 지원

### 1.3 핵심 가치
- SQL 없이도 데이터 조회 가능
- 자동화된 정기 보고서
- 실시간 대시보드 모니터링

---

## 2. 목표 및 범위

### 2.1 주요 목표
```
1. 데이터 접근성 향상
   - 비개발자가 자유롭게 데이터 분석 가능
   - "데이터팀에 SQL 요청" 시간 80% 감소

2. 의사결정 가속화
   - 실시간 KPI 모니터링으로 빠른 대응
   - 정기 보고서로 전략 수립 지원

3. 운영 효율성 증대
   - 자동화된 보고서로 수작업 제거
   - 일관된 형식의 분석 데이터 제공
```

### 2.2 범위 제외
- 광고주에 대한 개별 보고서 (향후 검토)
- 이상탐지 및 자동 알림 (향후 검토)
- 예측 분석 (향후 검토)

## 4. 유스케이스 (상세)

### UC1: 광고 운영팀장의 일일 KPI 모니터링

**사용자 정보**:
- 직책: 광고 운영팀장
- 경험: 비개발자, 마케팅 배경
- 업무: 일일 광고 성과 모니터링 및 전략 조정

**전제 조건**:
- Redash가 정상 작동 중
- 어제 광고 로그가 Athena에 저장됨
- 팀장이 Redash 접근 권한 있음

**시나리오**:

| 단계 | 활동 | 입력 | 출력 | 시간 |
|---|---|---|---|---|
| 1 | Redash 접속 | 웹 브라우저 | Redash 로그인 페이지 | 30초 |
| 2 | 대시보드 선택 | "오늘 현황" 클릭 | Redash 대시보드 로딩 | 1분 |
| 3 | KPI 확인 | 페이지 로딩 완료 | KPI 카드 3개 표시 (매출, CTR, ROI) | 30초 |
| 4 | 추이 분석 | 라인 차트 확인 | 최근 7일 추이 표시 | 2분 |
| 5 | 의사결정 | 분석 결과 | "입찰가 조정" 결정 | 1분 |

**성공 기준**:
- ✅ Redash 접속부터 의사결정까지 5분 이내
- ✅ 모든 KPI 정확하게 표시
- ✅ 실시간 데이터 반영

**실패 케이스**:
- ❌ Redash 접속 불가: 관리자 문의
- ❌ 데이터 없음: "데이터 준비 중" 메시지 표시
- ❌ 차트 로드 실패: 새로고침 또는 관리자 문의

### UC2: 마케팅 분석가의 특정 데이터 조회

**사용자 정보**:
- 직책: 마케팅 분석가
- 경험: 비개발자, SQL 미경험
- 업무: 카테고리별 성과 분석

**전제 조건**:
- Vanna API가 정상 작동 중
- 광고 로그 데이터가 Athena에 존재
- 분석가가 Vanna 접근 권한 있음

**시나리오**:

```
분석가: "지난 30일간 분식 카테고리 일별 CTR 데이터"

Vanna: SQL 자동 생성
  SELECT
    DATE(from_unixtime(timestamp/1000)) as date,
    COUNT(CASE WHEN event_type='impression' THEN 1 END) as impressions,
    COUNT(CASE WHEN event_type='click' THEN 1 END) as clicks,
    ROUND(100.0 * clicks / NULLIF(impressions, 0), 2) as ctr_pct
  FROM ad_events_raw
  WHERE campaign_id = 'food'
    AND year || '-' || month || '-' || day
        BETWEEN '2026-02-04' AND '2026-03-04'
  GROUP BY date
  ORDER BY date DESC

결과: 테이블 형식으로 30개 행 반환
분석가: "2월 25일에 CTR이 급락했네. 뭐가 있었나?"
```

**성공 기준**:
- ✅ SQL 요청 없이 자연어로 질문
- ✅ 5분 이내 결과 반환
- ✅ 정확한 데이터 제공

## 5. 기능 명세 (상세)

### 기능 1: Redash "오늘 현황" 대시보드

**ID**: F1.1
**기능명**: 오늘 현황 KPI 대시보드
**카테고리**: 실시간 모니터링

**요구사항**:

| 항목 | 상세 |
|---|---|
| **접근** | 웹 UI (Redash 메인 페이지) |
| **사용자** | 광고 운영팀장, 팀장 |
| **갱신** | 5분마다 자동 (또는 수동 새로고침) |
| **대상 데이터** | 어제 00:00 ~ 23:59 UTC |
| **성능** | 페이지 로딩 < 5초 |

**표시 항목**:

```
1. KPI 카드 3개 (상단)

   카드 1: 어제 매출
   - 수치: 어제 매출액 (만원 단위)
   - 목표: 일 평균 목표 매출
   - 달성률: (실제 / 목표) × 100 %
   - 색상: 달성 시 초록, 미달 시 노랑/빨강
   - 예: "1,250만원 / 1,000만원 목표 (125%)"

   카드 2: 어제 CTR
   - 수치: 어제 평균 CTR (%, 소수점 2자리)
   - 전일 대비: (어제 - 전일) / 전일 × 100
   - 증감 화살표: ↑초록 (상향), ↓빨강 (하향)
   - 예: "7.45% (↑0.23%)"

   카드 3: 어제 ROI
   - 수치: 어제 ROI (%, 소수점 1자리)
   - 목표: 일반적 목표 ROI (예: 400%)
   - 달성률 바 차트
   - 예: "425.6% (목표 400% 달성)"

2. 라인 차트 (하단)
   - 제목: "최근 7일 KPI 추이"
   - X축: 날짜 (YYYY-MM-DD)
   - Y축: 값 (3개 라인)
   - 라인 1: 매출 (파란색, 좌측 Y축)
   - 라인 2: CTR (주황색, 우측 Y축)
   - 라인 3: ROI (초록색, 우측 Y축)
   - 범례: 각 라인별 범례 표시
   - 상호작용: 마우스 오버 시 값 표시
```

**데이터 출처**:
```sql
SELECT
  SUM(CASE WHEN event_type='impression' THEN 1 END) as impressions,
  SUM(CASE WHEN event_type='click' THEN 1 END) as clicks,
  SUM(CASE WHEN event_type='conversion' THEN total_amount ELSE 0 END) as revenue,
  SUM(CASE WHEN event_type='click' THEN cpc_cost ELSE 0 END) as cost,
  ROUND(100.0 * clicks / NULLIF(impressions, 0), 2) as ctr_pct,
  ROUND(revenue / NULLIF(cost, 0) * 100, 1) as roi_pct
FROM capa_db.ad_events_raw
WHERE year = '2026' AND month = '03' AND day = '04'
```

**에러 처리**:
- 데이터 없음: "데이터 준비 중입니다. 잠시만 기다려주세요."
- 쿼리 타임아웃: "쿼리 실행 시간 초과. 범위를 좁혀주세요."
- 권한 없음: "이 대시보드에 접근할 수 없습니다."

### 기능 2: Redash "상세 분석" 대시보드

**ID**: F1.2
**기능명**: 카테고리/Shop별 성과 분석 대시보드
**카테고리**: 분석

**표시 항목**:

**섹션 A: 카테고리별 성과 (테이블)**
```
컬럼:
  1. 순번 (자동 증가)
  2. 카테고리명 (문자, 정렬 가능)
  3. 매출 (숫자, 만원, 소수점 0)
  4. Impression (숫자, 천 단위 구분)
  5. Click (숫자)
  6. CTR (%, 소수점 2자리)
  7. ROI (%, 소수점 1자리)

기본 정렬: ROI 내림차순
페이지네이션: 전체 카테고리 표시 (보통 6~8개)
행 색상: ROI 기준 색상 코딩
  - 400% 이상: 초록
  - 300% ~ 400%: 노랑
  - 300% 미만: 빨강
```

**섹션 B: Shop별 순위 (탭형)**
```
탭 1: "Top 10 (높은 매출순)"
  컬럼: Shop ID, Shop 이름, 매출, ROI
  정렬: 매출 내림차순
  행 수: 10개

탭 2: "Bottom 10 (낮은 매출순)"
  컬럼: Shop ID, Shop 이름, 매출, ROI
  정렬: 매출 오름차순
  행 수: 10개
```

**데이터 출처**: ad_daily_summary (최근 7일 합산)
**필터**: 기간 선택 (7일, 30일, 90일)

**ID**: F2.1
**기능명**: 자동 생성 주간 성과 보고서

빈도: 매주 월요일

**보고서 구성**:

```
1. 제목 페이지
   - 제목: "주간 광고 성과 보고서"
   - 기간: "2026-02-24 ~ 2026-03-02"
   - 생성일: 2026-03-03 (월요일 아침)

2. Executive Summary (1페이지)
   - 주요 수치 요약
     * 지난주 총 매출: OOO만원
     * 평균 CTR: O.OO%
     * 평균 ROI: OOO%
   - 주간 하이라이트 (3개)
   - 주의 사항 (2개)

3. 상세 분석 (2페이지)

   섹션 A: 카테고리별 성과 분석
   - 테이블: 모든 카테고리의 매출, CTR, ROI
   - Top 3 (강세)
   - Bottom 3 (약세)
   - 차트: 카테고리별 매출 비교 (바 차트)

   섹션 B: Shop별 성과
   - Top 10 가게 (ROI 기준)
   - Bottom 10 가게 (ROI 기준)

4. Claude LLM 자동 생성 인사이트 (1페이지)
   - "지난주 분식 카테고리가 전주 대비 8% 성장.
     이는 홍보 캠페인의 효과로 보임."
   - "카페 카테고리의 CTR이 평균 대비 15% 낮음.
     소재 개선이 필요함."
   - "신규 가게 'OOO'의 ROI가 예상을 초과함.
     성공 사례 분석 권장."

5. 추가 데이터 (선택)
   - 시간대별 트래픽 (라인 차트)
   - 카테고리별 추이 (라인 차트, 7주)
```

**품질 기준**:
- ✅ 모든 데이터 정확성 검증
- ✅ PDF 가독성 (최소 11pt 폰트)
- ✅ 한글 폰트 정상 표시
- ✅ 이미지 품질 (고해상도)

### 기능 4: Vanna API Text-to-SQL

**ID**: F3.1
**기능명**: 자연어 → SQL 변환 및 실행
**카테고리**: 분석

**입력**:
```
사용자 자연어 질문 예시:
- "지난 30일 분식 카테고리 일별 CTR"
- "어느 가게 ROI가 가장 높아?"
- "시간대별 impression이 어떻게 되나?"
```

**처리 흐름**:
```
1. 사용자가 자연어 질문 입력
   → Vanna로 전송

2. Vanna가 SQL 자동 생성
   - LLM이 질문 해석
   - 적절한 테이블 및 컬럼 선택
   - SQL 쿼리 생성

3. Athena에서 쿼리 실행
   - 권한 확인
   - 쿼리 최적화 (파티션 활용)
   - 데이터 반환

4. 결과를 사용자에게 표시
   - 테이블 형식
   - 또는 차트 형식 (자동 추천)
```

**응답 시간**: 5분 이내
**지원 질문 범위**:
- ✅ 단순 집계 ("전체 매출은?")
- ✅ 기간 조건 ("지난 30일?")
- ✅ 필터링 ("분식 카테고리만")
- ✅ 정렬/순위 ("TOP 10은?")
- ❌ 복잡한 조인 (향후)
- ❌ 머신러닝 분석 (향후)
