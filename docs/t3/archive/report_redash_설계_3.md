## 1. 여정 요약

### 1.1 처음 의문
- ANALYSIS_REPORT.md에서 "Report Generator + Redash 둘 다 유지" 추천
- **의문**: 이게 맞나? 겹치는 부분이 있지 않나?

### 1.2 발견한 문제
```
- 보고서가 "마케터 한정"으로 작성됨
- "기술 중심"이라 비즈니스 요구사항 불명확
- 토이 프로젝트라 실제 사용자 없음
→ 뭐가 필요한지 판단 불가능
```

### 1.3 핵심 깨달음
```
Report Generator의 원래 목적:
  - 일정 시간마다 자동 보고서 발송 (정기)
  - 요청받을 때 온디맨드(필요할때 즉석에서) 생성
  → "데이터팀에 SQL 요청하는 시간 줄이기"

배달앱 광고 플랫폼 (내부)의 특성:
  - 매 시간 트렌드가 변함
  - 운영팀/경영진이 빠른 판단 필요
  - 다양한 상황에 빠르게 대응 필요

→ 각 상황마다 다른 도구가 필요하고, 역할이 명확함
```

## 2. 최종 결정: 3가지 도구 조합

### 2.1 Vanna API (Text-to-SQL)
```
도구명: Vanna (Text-to-SQL 엔진)
목적: 자연어 → SQL 자동 변환
역할: 비개발자가 자유롭게 데이터 조회

사용 방식:
  - 비개발자: "지난주 분식 카테고리 CTR이 뭐야?"
  - Vanna: SQL 자동 생성
  - 결과: 쿼리 결과 반환

특징:
  ✅ 개발자 개입 없음 (데이터팀 요청 불필요)
  ✅ 자유로운 질문 가능
  ✅ 5분 내 답변

사용 예시:
  - "지난주 분식 카테고리 CTR이 뭐야?"
  - "어느 가게가 최고 ROI인가?"
  - "시간대별 트래픽이 언제가 피크인가?"


```

### 2.2 Report Generator
```
목적: 자동 정기 보고서 + LLM 분석
역할: 정기/온디맨드 보고서 자동 생성 및 발송

보고서 내용:
  ✅ 지난주/지난달 요약 (핵심 수치)
  ✅ 카테고리별 상세 분석
  ✅ Top/Bottom 성과 (최고/최저 카테고리, 가게)
  ✅ Claude LLM 인사이트
     - "CTR이 올라간 이유는..."
     - "카페 카테고리의 CVR이 하락한 원인..."
  ✅ ROI/ROAS 분석 (버그 수정 필수)

주기: 매주/매월 자동 발송 (설정 가능)
형식: PDF
사용자: CEO, 경영진, 마케팅팀장
(버그 수정 + 구조 유지)
```

### 2.3 Redash (웹 기반 BI 도구)
```
도구명: Redash
목적: 웹 기반 대시보드 + 모니터링
역할: 일일 모니터링, 빠른 의사결정

대시보드 형태:
  - 라인 차트 (일별 추이) ← 아드리엘 대시보드 참고
  - 테이블 (상세 데이터)
  - 필터 (기간, 카테고리 선택)

특징:
  ✅ 웹 UI (누구나 쉽게 접속)
  ✅ 자동 갱신 (5분/1시간/1일 설정 가능)
  ✅ 여러 시각화 한 화면에 배치

사용자: 팀장, 분석가, 운영팀
(쿼리 4개 + 대시보드 3개 + Alert 3개)
```

## 3. 역할 분담 매트릭스

| 항목 | Vanna API (Text-to-SQL) | Redash | Report Generator |
|---|---|---|---|
| **도구명** | Vanna | Redash | Report Generator |
| **목적** | 자유로운 탐색 | 일일 모니터링 | 정기 보고서 |
| **형식** | 웹 UI | 웹 대시보드 | PDF |
| **갱신 주기** | 온디맨드 | 5분~1시간 | 주/월 |
| **사용자** | 데이터팀, 분석가 | 팀장, 운영팀 | CEO, 경영진 |
| **특징** | 자연어 → SQL | 차트 | 자동 인사이트 |
| **사용 빈도** | 필요할 때마다 | 매일 | 정기적 (주/월) |

## 4. 유스케이스

### UC1: 광고 운영팀장의 일일 KPI 모니터링

**사용자**: 광고 운영팀장
**시간**: 매일 아침 9시
**목표**: 어제 성과 확인 → 오늘 마케팅 전략 결정
**도구**: Redash

**세부 흐름**:
```
1. Redash 웹 접속 (http://redash.capa.local)
2. "오늘 현황" 대시보드 열기
3. KPI 카드 확인
   - 어제 매출: 1,000만원
   - 어제 CTR: 7.2%
   - 어제 ROI: 450%
4. 라인 차트로 최근 7일 추이 분석
5. 의사결정 수행
   → "CTR이 떨어졌으니 입찰가 올려야겠다"
6. 마케팅팀에 지시사항 전달
```

**기대 결과**: 오늘의 광고 입찰가 조정 완료

---

### UC2: 마케팅 분석가의 특정 데이터 조회

**사용자**: 마케팅 분석가
**시간**: 필요할 때 (예: 오후 2시)
**목표**: "분식 카테고리가 왜 CTR이 낮을까?" 분석
**도구**: Vanna API (Text-to-SQL)

**세부 흐름**:
```
1. Vanna API 접속
2. 자연어로 질문
   → "분식 카테고리 최근 30일 일별 CTR 데이터"
3. Vanna가 SQL 자동 생성 및 실행
4. 결과 조회
   - 날짜별 CTR 추이 확인
   - 패턴 분석
5. 원인 파악
   → "특정 요일에 CTR이 떨어진다"
6. Report Generator 또는 경영진에게 인사이트 전달
```

**기대 결과**: SQL 요청 없이 5분 내 데이터 분석 완료

### UC3: CEO의 주간 성과 보고서 수신

**사용자**: CEO
**시간**: 매주 월요일 오전 8시
**목표**: 지난주 광고 플랫폼 전체 성과 파악
**도구**: Report Generator

**세부 흐름**:
```
1. 이메일/Slack에서 Report Generator 보고서 자동 수신
2. PDF 보고서 열기
3. 지난주 요약 섹션 확인
   - 총 매출: 7,000만원
   - 평균 CTR: 6.8%
   - 평균 ROI: 440%
4. 카테고리별 상세 분석 읽기
   - 분식: CTR 8.5%, ROI 500% (강세)
   - 카페: CTR 5.2%, ROI 300% (약세)
5. Claude LLM 자동 생성 인사이트 검토
   → "지난주 분식 카테고리 강세로 전체 성과 상승"
   → "카페 카테고리 소재 개선 필요"
6. 경영진 회의에서 보고 및 의사결정
   → "카페 카테고리 예산 재배분 검토"
```

**기대 결과**: 경영진 의사결정 필요 데이터 제공 완료

## 5. Redash 기능 명세

### 기능 1: "오늘 현황" 대시보드

**기능명**: 오늘 현황 KPI 대시보드
**사용자**: 광고 운영팀장
**접근**: 웹 UI (Redash)
**갱신**: 5분마다 자동 (또는 수동 새로고침)
**위치**: Redash 메인 대시보드

**표시 항목**:
```
1. KPI 카드 (3개)
   - 카드 1: 어제 매출 (단위: 만원)
   - 카드 2: 어제 CTR (단위: %, 소수점 2자리)
   - 카드 3: 어제 ROI (단위: %, 소수점 1자리)

2. 메트릭 (각 카드마다)
   - 값: 어제 실제값
   - 목표: 일반적 목표값 대비
   - 증감: 전일 대비 증감률 (색상: ↑초록, ↓빨강)

3. 데이터 소스
   - Athena 쿼리: ad_daily_summary 테이블
   - 기간: 어제 00:00 ~ 23:59
```

**에러 처리**:
- 데이터 없음: "데이터 준비 중..." 표시
- 쿼리 실패: "데이터 로드 실패. 관리자 문의"

---

### 기능 2: "상세 분석" 대시보드

**기능명**: 카테고리별/Shop별 성과 분석 대시보드
**사용자**: 마케팅 분석가
**접근**: 웹 UI (Redash)
**갱신**: 1시간마다 자동
**위치**: Redash 세부 분석 탭

**표시 항목**:

**섹션 A: 카테고리별 성과 (테이블)**
```
컬럼:
  1. 카테고리명 (문자열)
  2. 매출 (숫자, 만원)
  3. CTR (숫자, %)
  4. ROI (숫자, %)

정렬: ROI 내림차순 (높은 것부터)
행 수: 전체 카테고리 표시
```

**섹션 B: Shop별 순위 (테이블)**
```
컬럼:
  1. Shop ID (숫자)
  2. Shop 이름 (문자열)
  3. 매출 (숫자, 만원)
  4. 주문 건수 (숫자)

상위/하위: Top 10 + Bottom 10 (탭으로 구분)
정렬: 매출 내림차순
```

**데이터 소스**:
- Athena: ad_daily_summary, ad_events_raw
- 기간: 최근 7일 합산

---

### 기능 3: "트렌드" 대시보드

**기능명**: 추이 분석 대시보드
**사용자**: 경영진, 마케팅 리더
**접근**: 웹 UI (Redash)
**갱신**: 1시간마다 자동
**위치**: Redash 트렌드 탭

**표시 항목**:

**섹션 A: 일별 추이 (라인 차트)**
```
X축: 날짜 (최근 30일)
Y축: 값

라인 3개:
  1. 매출 (파란색)
  2. CTR (주황색)
  3. ROI (초록색)

범례: 각 라인별 범례 표시
```

**섹션 B: 주별 추이 (라인 차트)**
```
X축: 주 (최근 12주)
Y축: 값

라인: 매출, CTR, ROI

특징: 요일 변동성을 제거한 추이 확인
```

**데이터 소스**:
- Athena: ad_daily_summary
- 기간: 최근 30일 (필터로 조정 가능)

**필터**:
- 기간 선택 (7일, 30일, 90일)
- 카테고리 선택 (선택사항)

## 6. 당신의 할 일 (우선순위)

### 우선순위 1: Report Generator 버그 수정 ⭐⭐⭐
```
파일: services/report-generator/t3_report_generator/athena_client.py

[작업 1] ROAS 지표 추가
  - query_kpi_summary() 함수 수정
  - SQL: SUM(total_amount) / SUM(cpc_cost) × 100
  - 보고서에 "광고비 대비 수익률" 표시

[작업 2] shop_id 오류 수정
  - query_shop_performance() 함수에서
  - user_id → shop_id 변경
  - "Shop별 성과" 섹션 정상화

[작업 3] 테스트
  - Slack 봇으로 "/리포트" 입력
  - 생성된 PDF에서 ROAS, Shop 성과 확인

예상 소요시간: 1주
```

### 우선순위 2: Redash 대시보드 구성 ⭐⭐⭐
```
[작업 1] SQL 쿼리 작성 (4개)
  1. 일별 KPI 요약
     - 매출, impression, click, conversion
     - CTR, CVR, CPC, ROAS

  2. 카테고리별 성과
     - 카테고리명, 매출, CTR, ROAS
     - ROI 내림차순 정렬

  3. 시간대별 볼륨
     - 시간, impression, click
     - 피크 시간대 파악

  4. Shop별 순위
     - Shop ID, 매출, conversion
     - Top 20 + Bottom 10

[작업 2] Redash에서 시각화
  - 라인 차트: 일별 추이
  - 테이블: 상세 데이터
  - 바 차트: 카테고리별 비교

[작업 3] 대시보드 3개 구성
  1. "오늘 현황" (KPI 카드)
  2. "상세 분석" (카테고리별, Shop별)
  3. "트렌드" (7일, 30일 추이)

예상 소요시간: 2주
```

### 우선순위 3: 구조 정리
```
- 나중에 쿼리를 쉽게 바꿀 수 있는 구조
- 하드코딩 제거
- 설정값 분리

예상 소요시간: 1주
```

  세 가지 모두 필수적
