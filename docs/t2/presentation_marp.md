# 효율적인 데이터 저장 및 파이프라인 운영 전략
## 2026-02-19 멘토링 요약

---

# 1. 데이터 저장 전략: 핵심 원칙
- **이벤트별 독립 저장**: Impression, Click, Conversion 로그 분리
- **장점**: 
  - 수집 단계 복잡도 감소
  - 특정 로그 장애 시 전체에 미치는 영향 최소화
- **처리**: 배치 단계에서 주기적 **Join(병합)** 수행 (빅데이터 표준)

---

# 2. Columnar Storage (Parquet)
- **왜 컬럼 기반 저장인가?**
  - **압축 효율**: 동일 타입 데이터 반복으로 저장 공간 절약
  - **I/O 절감**: 쿼리에 필요한 컬럼만 디스크에서 읽기 가능
- **본 프로젝트 선택**: AWS Athena + Glue 환경에 최적화된 **Parquet**

---

# 3. 압축 알고리즘: zstd
| 알고리즘 | 특징 |
|---|---|
| **zstd** | **압축률과 속도의 균형 (본 프로젝트 채택)** |
| snappy | 매우 빠른 속도, 다소 낮은 압축률 |
| gzip | 높은 압축률, 느린 속도 |

---

# 4. Airflow DAG 운영 원칙
- **분리 기준**:
  1. 스케줄이 다르면 무조건 분리 (**Hourly** vs **Daily**)
  2. 독립적 운영(On/Off) 및 장애 전파 방지 고려
- **적용**: 
  - `ad_hourly_summary`: Impression + Click (매시간)
  - `ad_daily_summary`: Hourly + Conversion (매일 02:00)

---

# 5. 광고 도메인: CVR의 중요성
- **전환(Conversion)**: 실제 구매, 구매, 설치 등 최종 행동
- **광고주 관점**: 클릭(CPC)만 발생하고 전환이 없으면 비용만 낭비
- **비즈니스 임팩트**: CVR이 낮으면 광고주가 이탈하고 플랫폼 수익 감소

---

# 6. 현업 Summary 테이블 전략
- **집계 기간**: **15일치** 기준 (트렌드 반영 + 통계적 안정성)
- **CVR 딜레이**: 클릭 후 전환까지 약 **3시간 시차** 발생
- **분석**: 시간 단위보다는 일 단위 이상에서 CVR 산출 권장

---

# 7. 파티셔닝과 dt 설계
- **원칙**: `dt`는 **STRING** 타입이어야 함 (시간 단위 `-00` 표현 가능)
- **장점**: 사전순(Lexicographic) 범위 비교 최적화
- **쿼리 패턴**:
  ```sql
  WHERE dt >= '2026-01-25' AND dt < '2026-01-26'
  ```
