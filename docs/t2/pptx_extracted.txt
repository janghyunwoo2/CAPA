SQL 정확도 4라운드 개선 스토리
Exec (Execution Accuracy): 생성 SQL의 실행 결과값이 정답과 일치하는가
SQL 생성 정확도 변화 (Exec)
Round 1 (33.3%) — 기반 공사
QA 시딩 편향성 해결(21개) 및 날짜 동적 처리(Jinja2) 도입으로 기초 정확도 확보.
Round 2 (36.1%) — 사고 통제
CoT 6단계 프롬프트, Negative Rules, 계산식 공식화를 통해 생성 로직의 안정성 강화.
Round 3 (69.4%) — 검색 진화
한국어 특화 임베딩(ko-sroberta) 도입 및 Reranker 제거로 관련 지식 검색 효율 극대화.
Round 4 (80.6%) — 구조 돌파
QA 메타데이터 기반 DDL 동적 주입을 통해 테이블 오인식 한계를 극복하고 80%대 진입.

AS-IS vs TO-BE — 도입 전후 비교
AS-IS (과거)
데이터 조회 (Ask)
데이터팀에 요청 후 수 시간~수 일 대기. 의사결정 타이밍 지연.
정기 리포트 (Report)
매주/매월 담당자가 수기 엑셀 다운로드 및 집계. 반복적인 리소스 낭비.
이상 탐지 (Alert)
사람이 대시보드를 볼 때만 인지. 사후 발견으로 인한 예산 낭비 발생.
대시보드 (Dashboard)
지표 확인마다 매번 데이터팀 요청 반복. 실시간 셀프 확인 불가.
TO-BE (현재)
데이터 조회 (Ask)
슬랙에서 자연어로 질문 시 수십 초 내 즉시 답변 및 시각화.
정기 리포트 (Report)
Airflow 기반 완전 자동화. 정해진 시간에 슬랙으로 자동 전송.
이상 탐지 (Alert)
AI 모델이 5분 주기로 실시간 감지. 이상 발생 즉시 슬랙 알림.
대시보드 (Dashboard)
Redash 대시보드 원클릭 셀프 확인. SQL 없이 언제든 실시간 지표 조회.
데이터 추출 업무 소요 시간 99% 단축 (반나절 → 1분 이내)

Reranker 도입 후 롤백
동일 노드 내 Airflow 워커의 과부하 작업으로 인한 자원 부족(oom: out of memory)으로 서버가 다운되는 현상
Karpenter를 도입하여 작업 특성별로 노드를 분리 프로비저닝하여 해결
Reranker 도입으로 정확도 개선을 기대했으나, 정확도는 오르지 않고 50초 이상의 지연 시간이 발생하여 UX 저하
사용되는 테이블이 2개 뿐이라 Reranker 도입은 오버엔지니어링. 롤백하고, ChromaDB의 메타데이터 필터링을 고도화하는 방향으로 선회
신뢰구간 상단이 비즈니스 맥락 없이 모델 예측값만으로 결정되고 있었음
트래픽이 적은 시간대에 신뢰구간 하한이 음수로 예측됨
즉시 조회 시 집계가 완료되지 않아, 수집 시점에 따라 데이터가 누락될 수 있었음
신뢰구간 상단 차등 가중치 부여
Prophet 신뢰구간 내 음수
CloudWatch 집계 처리시간
생성된 Raw데이터가 하나의 파이프로만 연결되었으나,
데이터 특성에 맞게 각각의 파이프라인으로 연결되어야 함.
