## 핵심 메시지

> **"데이터가 필요한 사람과 데이터에 접근 가능한 사람이 다르다"**

```
[데이터가 필요한 사람]          [데이터에 접근 가능한 사람]
  마케터, 기획자, 운영자    ≠      데이터 엔지니어, 분석 전문가
  
  → 의사결정 권한은 있지만         → 기술적 접근은 가능하지만
    데이터 추출 능력이 없다           비즈니스 맥락을 모른다
```

이 불일치(mismatch)가 **병목 → 지연 → 기회 손실**로 이어진다.

---

## 1. 접근 가능한 사람이 얼마나 적은가 — 리터러시 격차

### 1-1. 데이터 리터러시 현황

| 수치 | 내용 | 출처 |
|------|------|------|
| **약 43%** | 디지털 네이티브 세대 중 데이터 리터러시를 보유한 비율 | Exasol / TechRepublic, 2021 |
| **약 83%** | CEO 중 데이터 기반 의사결정을 원하는 비율 | IDC / Tableau, 2021 |
| **약 43%** | 실제 현장 직원 중 데이터 역량을 갖춘 비율 | IDC / Tableau, 2021 |

> 💡 **CEO 83%가 원하지만 직원 43%만 가능** → **40%p 갭**이 조직 내 데이터 접근 장벽

### 1-2. 국내 실제 사례 — 우아한형제들

| 수치 | 내용 |
|------|------|
| **약 95%** | 업무에 데이터를 활용해야 하는 구성원 비율 |
| **50% 이상** | 그 중 기술적 역량 부족으로 접근이 막힌 비율 |

```
95%가 필요 → 50%+는 접근 불가
→ 해결책으로 자체 AI 데이터 분석 도구 "물어보새" 개발
```

> 출처: 우아한형제들 기술 블로그, "AI 데이터 분석가 '물어보새' 등장 – 1부", 2024.07.12

## 2. 데이터 병목 (Data Bottleneck) — 대기 비용

### 2-1. 데이터팀 의존성이 만드는 병목

```
비개발 직군(마케터, 기획자, 운영)
        ↓ "이 데이터 뽑아줘" 요청
   데이터팀 백로그
        ↓ 수 시간 ~ 수 일 대기
   결과 전달
        ↓
   이미 기회를 놓침
```

**Snowflake 사례 (ThoughtSpot 고객)**
- 셀프서비스 분석 도입 후 **IT 백로그 20% 감소** 효과
- 출처: ThoughtSpot 고객 케이스 스터디, 2026

**Avaya 사례 (Salesforce CRM Analytics)**
- 수작업 리포트 처리에 대규모 시간 소모  
- 도입 후: 전 세계 12,000개 리포트를 **15개 대시보드로 통합**  
- 출처: Salesforce CRM Analytics 고객 사례, 2022

---

### 2-2. 우아한형제들 — 국내 실제 사례

| 수치 | 내용 | 출처 |
|------|------|------|
| **약 95%** | 업무에 데이터를 활용하는 구성원 비율 | 우아한형제들 기술 블로그, "AI 데이터 분석가 '물어보새' 등장 – 1부", 2024.07.12 |
| **50% 이상** | 기술적 역량 부족으로 데이터 접근에 어려움을 겪는 구성원 비율 | 위와 동일 |

> 💡 **핵심 통계**: 95%가 필요하지만 절반은 접근 불가 → 사내 AI 데이터 분석 도구 "물어보새" 개발 동기

## 6. 셀프서비스 분석(Self-Service Analytics) 부재의 문제

### 6-1. 현재 구조적 의존 사슬

```
비개발 직군의 질문
    → 데이터팀 요청 (티켓)
    → 대기 (수시간~수일)
    → 오해로 인한 재요청
    → 실제 인사이트 획득
    → 그사이 기회 손실
```

### 6-2. 셀프서비스 분석 도입 효과 (사례)

| 기업 | 도입 기술 | 효과 |
|------|-----------|------|
| **Snowflake** | ThoughtSpot 셀프서비스 분석 | IT 백로그 20% 감소 |
| **Avaya** | Salesforce CRM Analytics | 12,000개 리포트 → 15개 대시보드 통합 |
| **우아한형제들** | 자체 AI 분석 도구 "물어보새" | 데이터 접근 병목 해소, 500+ A/B 테스트 검증 |
| **SK Planet** | InsightLens (LLM+RAG) | 수 일 소요 → 즉시 데이터 조회 |
| **데이블(Dable)** | DableTalk (Slack 봇) | 비개발자가 Slack에서 직접 데이터 조회 |

## 7. 요약: Slide 04 스토리 뼈대

### 핵심 프레임: "필요한 사람 ≠ 접근 가능한 사람"

```
[상황]
데이터가 필요한 사람     데이터에 접근 가능한 사람
─────────────────────   ────────────────────────
마케터, 기획자, 운영   ≠  데이터 엔지니어, 분석 전문가
(의사결정 권한 ○)        (기술 접근 ○, 비즈니스 맥락 ×)
```

```
① 데이터는 넘쳐난다
         ↓
② 그런데 "필요한 사람"이 직접 접근할 수 없다
   - 디지털 네이티브조차 43%만 데이터 리터러시 보유
   - CEO 83%가 원하지만 현장 직원 43%만 역량 보유 → 40%p 갭
   - 우아한형제들: 95%가 필요하지만 50%+가 기술 역량 부족으로 접근 불가
         ↓
③ 이 불일치가 병목을 만든다
   - 비개발 직군 → 데이터팀에 요청 → 수시간~수일 대기
   - 마케터 75%: 현재 측정 시스템이 속도·정확도·신뢰성에서 미달
         ↓
④ 병목은 돈으로 환산된다
   - $6.2B 생산성 손실 (AI 루틴 자동화로 회복 가능)
   - $26.3B 미디어 투자 가치 미회수 (의사결정 속도 지연)
         ↓
⑤ 해결책: 자연어 → 데이터 (Text-to-SQL)
   "필요한 사람"이 직접 묻고 직접 얻는다
   → CAPA Text-to-SQL의 존재 이유
```
