# Airflow ETL 시간대 처리 분석

## 요청사항
- Airflow(UTC)와 ETL(KST) 간 시간대가 다른데 정상 작동하는지 확인

## AS-IS (문서상 설명)
- "Airflow는 UTC로 동작"
- "etl_summary_t2는 KST로 동작"  
- "DAG에서 자동 변환 처리" (구체적 방법 불명)

## TO-BE (코드 분석 결과)

### 1. 시간대 변환 프로세스 (4단계)

#### 1단계: Airflow 스케줄러
```python
# UTC로 스케줄 실행
data_interval_end = context.get('data_interval_end')  # UTC datetime
# 예: 2026-03-12T06:00:00+00:00 (UTC)
```

#### 2단계: DAG 내부 변환 (핵심)
```python
# 03_ad_hourly_summary_test.py의 _run_hourly_etl 함수
dt_utc = context.get('data_interval_end')
dt_kst = pendulum.instance(dt_utc).in_timezone('Asia/Seoul')
# 예: 2026-03-12T15:00:00+09:00 (KST)
```

#### 3단계: ETL 패키지 실행
```python
# KST 시간으로 ETL 인스턴스 생성
etl = HourlyETL(target_hour=dt_kst)  # KST datetime 전달
etl.run()

# hourly_etl.py 내부
self.year = self.target_hour.strftime("%Y")   # "2026"
self.month = self.target_hour.strftime("%m")  # "03"
self.day = self.target_hour.strftime("%d")    # "12"
self.hour = self.target_hour.strftime("%H")   # "15" (KST)
```

#### 4단계: S3 파티션 생성
```
s3://capa-data-lake-827913617635/ad_combined_log/
├── year=2026/
│   └── month=03/
│       └── day=12/
│           └── hour=15/  # KST 15시 (UTC 06시)
│               └── data.parquet
```

### 2. 시간대 변환 방법 (3가지 백업)

```python
# 방법 1: pendulum 사용 (기본)
dt = pendulum.instance(dt_utc).in_timezone('Asia/Seoul')

# 방법 2: pytz 사용 (백업)
dt = dt_utc.astimezone(pytz.timezone('Asia/Seoul'))

# 방법 3: datetime timedelta 사용 (백업)
dt = dt_utc.astimezone(timezone(timedelta(hours=9)))
```

### 3. 실제 예시

| 구분 | UTC | KST | S3 파티션 |
|------|-----|-----|-----------|
| Airflow 실행 | 2026-03-12T06:00:00+00:00 | - | - |
| DAG 변환 후 | - | 2026-03-12T15:00:00+09:00 | - |
| S3 저장 | - | - | year=2026/month=03/day=12/hour=15/ |

### 4. Athena 쿼리 시 파티션 필터

```sql
WHERE imp.year = '2026'
  AND imp.month = '03'
  AND imp.day = '12'
  AND imp.hour = '15'  -- KST 15시 데이터
```

## 검증 결과
- ✅ **정상 작동 확인**: DAG에서 명시적으로 UTC→KST 변환 수행
- ✅ **데이터 정합성**: S3 파티션이 KST 기준으로 일관되게 생성
- ✅ **백업 방법**: pendulum 외에도 pytz, timedelta 사용 가능

## 주의사항
1. **ETL 패키지 입력**: 반드시 KST datetime 객체를 전달해야 함
2. **백필 시**: 동일한 시간대 변환 로직 적용 필요
3. **로그 확인**: Airflow 로그는 UTC, ETL 로그는 KST로 표시됨

## 결론
시간대 처리가 **정상적으로 작동**합니다. DAG에서 UTC→KST 변환이 명시적으로 수행되며, 모든 하위 프로세스는 KST 기준으로 동작합니다.
