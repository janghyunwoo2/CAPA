# 배포 전략 및 데모 관련 개념 정리

## 블루그린 배포 (Blue-Green Deployment)

블루그린 배포는 무중단 배포 전략 중 하나로, 안전하고 빠른 배포와 롤백을 가능하게 합니다.

### 핵심 개념
- **Blue 환경**: 현재 실행 중인 프로덕션 환경
- **Green 환경**: 새로운 버전을 배포할 대기 환경

### 동작 방식
1. Blue 환경에서 현재 버전이 실행 중
2. Green 환경에 새 버전 배포 및 테스트
3. 로드밸런서/라우터를 통해 트래픽을 Green으로 전환
4. 문제 발생 시 즉시 Blue로 롤백 가능

### 장점
- **무중단 배포**: 서비스 중단 없이 배포
- **빠른 롤백**: 문제 발생 시 즉시 이전 버전으로 복구
- **안전한 테스트**: 실제 환경과 동일한 환경에서 사전 검증

### 단점
- **리소스 비용**: 두 배의 인프라 필요
- **데이터 동기화**: 상태가 있는 애플리케이션의 경우 복잡
- **환경 관리**: 두 환경을 동일하게 유지해야 함

### 구현 예시
```
[사용자] → [로드밸런서] → [Blue (v1.0) - 현재]
                         ↘ [Green (v2.0) - 신규] ← 배포

# 전환 후
[사용자] → [로드밸런서] → [Blue (v1.0) - 대기]
                         ↘ [Green (v2.0) - 현재] ✓
```

### 지원 플랫폼
- Kubernetes
- AWS ECS
- AWS CodeDeploy
- Azure App Service
- Google Cloud Run

## 데모의 법칙 (Demo Effect/Murphy's Law for Demos)

"완벽하게 작동하던 것도 데모할 때는 반드시 실패한다"는 경험적 법칙입니다.

### 기본 법칙
- 리허설에서는 완벽했지만 실제 데모에서는 실패
- 중요한 사람 앞일수록 실패 확률 상승
- 가장 자신 있는 기능이 가장 먼저 고장

### 일반적인 실패 사례
- **네트워크 문제**: Wi-Fi 연결 끊김, API 응답 없음
- **환경 차이**: 로컬 vs 프로덕션 설정 불일치
- **데이터 문제**: 테스트 데이터 누락, 권한 오류
- **하드웨어 문제**: 프로젝터 연결 실패, 배터리 방전
- **타이밍 이슈**: 캐시 만료, 세션 타임아웃

### 예방책

#### 1. 백업 계획 준비
- 녹화된 데모 영상
- 스크린샷/슬라이드
- 오프라인 버전

#### 2. 사전 점검
- 동일한 환경에서 테스트
- 모든 계정/권한 확인
- 네트워크 대역폭 체크
- 배터리 충전 상태 확인

#### 3. 라이브 코딩 최소화
- 미리 준비된 코드 사용
- 복사-붙여넣기 준비
- 코드 스니펫 도구 활용

### 실패 시 대처법
1. **침착함 유지**: 당황하지 않고 상황 설명
2. **유머 활용**: 분위기 전환
3. **백업 활용**: 준비된 자료로 전환
4. **설명으로 대체**: "실제로는 이렇게 작동합니다" 설명

### 유명한 데모 실패 사례
- **Steve Jobs의 iPhone 첫 발표 (2007)**: Wi-Fi 문제로 백업 장비 사용
- **Windows 98 블루스크린 (1998 COMDEX)**: Bill Gates 발표 중 블루스크린
- **자율주행차 데모**: 여러 회사의 공개 시연 실패 사례들

### 실무 적용 팁
1. **환경 격리**: 데모 전용 환경 구축
2. **더미 데이터**: 안정적인 테스트 데이터 세트 준비
3. **오프라인 모드**: 네트워크 독립적 동작 구현
4. **상태 초기화**: 데모 시작 전 clean state 보장
5. **시간 여유**: 예상 시간의 1.5배 확보

## 결론

블루그린 배포는 안정적인 서비스 운영을 위한 필수 전략이며, 데모의 법칙은 모든 개발자가 명심해야 할 현실적인 교훈입니다. 두 개념 모두 **철저한 준비**와 **백업 계획**의 중요성을 강조합니다.
