# 2026-02-19 멘토링 회의록

## 1. 데이터 저장 전략

### 핵심 원칙
- 배치 프로세싱에 최적화된 형태로 데이터를 적재한다.
- 들어오는 로그 이벤트별로 **각각 따로 저장**한다 (impression, click, conversion 분리).
  - 처음부터 합쳐서 저장하면 수집 단계의 복잡도가 증가하고, 한쪽 로그 장애 시 전체에 영향을 준다.
  - 분리 저장 후 **배치에서 주기적으로 병합**(조인)하는 것이 빅데이터 처리의 표준 패턴이다.

### Columnar Storage Format (컬럼 기반 저장 포맷)
일반적인 행(Row) 기반 저장과 달리, **컬럼(Column) 단위로 데이터를 저장**하는 형식이다.

**왜 컬럼 기반이 압축 효율이 높은가?**
- 같은 컬럼의 값들은 데이터 타입과 분포가 유사하므로 압축 알고리즘이 더 잘 동작한다.
  - 예: `device_type` 컬럼 → `["iOS", "iOS", "Android", "iOS", ...]` → 같은 값이 반복되어 압축률이 극도로 높음
- 분석 쿼리는 보통 전체 컬럼 중 일부만 조회하므로, 필요한 컬럼만 읽어 I/O를 절감한다.
  - 예: `SELECT campaign_id, SUM(clicks) ...`는 campaign_id, clicks 컬럼만 디스크에서 읽음

**주요 포맷 비교:**

| 포맷 | 설명 | 주 사용처 |
|------|------|-----------|
| **Parquet** | Apache 생태계 표준. Spark, Athena, Glue 등 광범위 지원 | AWS 기반 데이터 레이크 (본 프로젝트 채택) |
| **ORC** | Hive 최적화 포맷. Parquet과 유사하나 Hive 생태계에 특화 | Hive 기반 온프레미스 환경 |

→ 본 프로젝트는 AWS Athena + Glue 기반이므로 **Parquet** 채택

**압축 알고리즘 비교:**

| 알고리즘 | 압축률 | 속도 | 특징 |
|----------|--------|------|------|
| **zstd** | 높음 | 보통 | 압축률과 속도의 균형이 좋음. **본 프로젝트 채택** |
| **gzip** | 높음 | 느림 | 전통적으로 많이 사용. 압축 해제 속도가 느림 |
| **snappy** | 보통 | 빠름 | 속도가 빠르지만 압축률이 다소 떨어짐. 실시간 처리에 적합 |
| **lz4** | 보통 | 매우 빠름 | 압축률보다 속도를 극단적으로 우선시할 때 사용 |

→ 배치 분석 용도에서는 **zstd**가 최적 (스토리지 비용 절감 + 합리적 처리 속도)

## 2. Airflow DAG 운영 원칙

### 분리 기준
1. **스케줄이 다르면 반드시 분리**: hourly DAG와 daily DAG를 한 파일에 넣지 않는다.
2. **스케줄이 같더라도 운영 상 분리를 고려**: 
   - 독립적으로 **on/off**(pause/unpause) 관리가 가능해야 한다.
   - 한 DAG의 장애가 다른 DAG에 영향을 주지 않아야 한다.
   - 예: impression 집계 DAG과 데이터 품질 체크 DAG이 같은 @hourly라도 분리

### 본 프로젝트 적용
- `ad_hourly_summary` (매시간): impression + click 조인 → hourly summary
- `ad_daily_summary` (매일 02:00): hourly summary 재집계 + conversion 조인 → daily summary
- 스케줄이 다르므로 별도 파일로 분리

## 3. 광고 도메인: 전환율(CVR)의 중요성

### 광고주 관점
- **전환(Conversion)이 광고주에게 가장 중요한 지표**이다.
  - 전환 = 실제 구매, 가입, 앱 설치 등 광고주가 원하는 최종 행동
  - 클릭만 발생하고 전환이 없으면 광고주 입장에서는 **비용만 나가는 손해**
- CPC(Cost Per Click) 과금 모델에서 클릭만 많고 전환이 없으면:
  - 광고주: 비용 대비 효과 없음 → 광고 중단 가능성
  - 플랫폼: 장기적으로 광고주 이탈 → 수익 감소

### CVR이 비즈니스에 미치는 영향
```
노출 → 클릭 → 전환 → 매출
       (CTR)   (CVR)
       
- CTR만 높고 CVR이 낮은 경우: 광고 소재는 매력적이지만 랜딩 페이지가 부실
- CTR이 낮고 CVR이 높은 경우: 타겟팅은 정확하지만 소재가 매력 없음
- 둘 다 높은 경우: 이상적인 광고 성과
```

## 4. Summary 테이블 생성 전략 (현업 사례)

### 멘토님 현업 기준
- **15일치 summary**를 기준으로 CTR을 분석한다.
  - 짧은 기간(1일)은 데이터 변동이 크고, 긴 기간(30일+)은 트렌드 변화를 놓칠 수 있음
  - 15일은 통계적으로 안정적이면서도 최근 트렌드를 반영하는 균형점
- **CVR은 약 3시간 딜레이를 가정**하여 집계한다.
  - 사용자가 광고를 클릭한 후 바로 전환하지 않고, 시간이 지난 후 전환하는 경우가 많음
  - 따라서 시간 단위(hourly) 집계에서는 CVR을 산출하지 않음 (데이터가 아직 불완전)
  - 일 단위(daily) 이상에서 CVR을 산출해야 의미 있는 수치

### Combined Log → AI 모델링
- impression + click을 조인한 **combined log**를 기반으로 AI 모델링 수행
  - 예: 어떤 유저가 어떤 광고를 클릭할 확률을 예측하는 CTR 예측 모델
  - RTB(Real-Time Bidding)에서 입찰가를 결정하는 데 활용

## 5. 파티셔닝과 쿼리 패턴

### dt 컬럼 설계 원칙

**`dt`는 DATE 타입이 아닌 STRING 타입이어야 한다.**

이유: 시간 단위 파티셔닝(`dt=2026-01-25-00`)에서 `-00` 같은 시간 부분은 DATE 타입으로 표현 불가.

### 쿼리 패턴별 dt 사용법

```sql
-- 1. 특정 일자 전체 데이터 조회
WHERE dt >= '2026-01-25' AND dt < '2026-01-26'

-- 2. 특정 시간 데이터 조회
WHERE dt = '2026-01-25-00'

-- 3. 월 단위 범위 조회 (문자열 비교이므로 가능)
WHERE dt >= '2026-01' AND dt < '2026-02'

-- 4. 여러 일자 범위 조회
WHERE dt >= '2026-01-05' AND dt < '2026-01-06'
```

**문자열 파티션의 장점:**
- 범위 비교(`>=`, `<`)가 사전순(lexicographic order)으로 동작하여 직관적
- 시간/일/월 단위를 하나의 컬럼으로 통일 가능
- Athena/Hive가 파티션 프루닝(pruning)을 효율적으로 수행

### S3 경로 기반 파티셔닝 (멘토님 방식)
```
s3://bucket/data/dt=2026-01-25-00/file.parquet
s3://bucket/data/dt=2026-01-25-01/file.parquet
...
```
- S3 경로 자체가 파티션 키가 됨
- `utc_basic_time` 컬럼명을 별도로 사용하는 경우도 있음 (Glue 테이블 정의에 따라)

## 6. MCP (Model Context Protocol) 활용

- **역할**: LLM이 코드베이스를 이해하고 코드 작성을 도와주는 도구
- **동작 방식**: 
  - DB 인덱스 기반이 아니라, **사람이 코드를 찾는 방식과 유사하게** grep, 파일 탐색 등을 수행

- **활용**: 대규모 코드베이스에서 관련 코드를 찾아 컨텍스트를 제공

  2. LLM이 이를 기반으로 Tech Spec 초안 작성

## 7. Plan 모드 (계획 수립)

### 개념
- **텍스트로 순서도를 그리는 기능**: 코드를 바로 작성하지 않고, 구현 계획을 먼저 수립
- `.md` 파일로 설계 컨셉/구현 단계를 미리 정의해두는 방식

### 왜 필요한가?
- **코드베이스가 클수록 필수**: 큰 프로젝트에서 바로 코딩하면 방향을 잃기 쉬움
- **작은 코드베이스에서도 사용**: 구현 방향을 먼저 확인하고 시작하면 불필요한 수정 감소
- **중요도가 높은 코드**: 여러 곳에서 호출되는 공통 모듈 등은 반드시 계획 후 구현

### 활용 패턴
```
1. Plan 모드로 구현 단계 작성 (MD 파일)
2. 단계별 구현
3. 단계 사이에 merge (중간 점검)
4. 반복
```
- 단계적 구현과 **단계 사이의 merge**가 핵심
- 한 번에 모든 것을 구현하려 하지 않고, 작은 단위로 나누어 검증하며 진행
