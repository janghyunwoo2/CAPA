## 현업에서 데이터 → 대시보드 사용방식

### 데이터 처리 패턴
- **배치 데이터**: 주기적으로 처리되는 데이터
- **BI(비즈니스 인텔리전스)**: 비즈니스에 최적화된 형태로 가공

### 데이터 예시
- 특정 시간 대의 노출 수
- 클릭 수
- CPC(클릭당비용) 계산값 - 시간 단위로 집계

### 대시보드 활용
- **기간 조회**: 1주일치, 1개월치 데이터 확인
- **경향 분석**: 2-3주 전 성과와 비교를 통한 추세 파악
- **데이터 구조**: fact 테이블, 디멘셔널 테이블, hourly 기반 `ad_summary` 테이블

### 반복되는 SQL 쿼리
- 자주 사용하는 쿼리는 Redash에 미리 저장
- 즐겨찾기(favorites) 기능으로 빠른 접근

### 시스템 흐름
1. **Athena**: 원본 데이터 쿼리
2. **Redash**: 쿼리 결과 시각화 및 관리
   - 데이터베이스 연결
   - SQL 쿼리 저장 및 관리
   - API 키를 통한 접근 제어
3. **대시보드**: 비즈니스 사용자용 시각화
4. **리포트**: AI 분석 결과 (PDF 형태)

### 계정 관리
- 관리자 계정: 시스템 설정 및 관리
- 사용자 계정: 대시보드 조회 및 보고서 요청

### 요청사항 처리
- Redash를 통해 요청사항 수집
- 필요시 SQL 쿼리 수정 및 재분석

### 작업 분담
- **현우**: 주기성 데이터 처리 (서버 분리)
- **병훈**: 비주기성 쿼리 처리 (SQL 기반)
- **목표**: 비주기 + 주기성 통합 처리 체계 구축

## 현업 업무환경

- **MCP(Model Context Protocol) 활용**
- **Plan 모드 → 검토 → 구현** 순환 구조

### 도구 활용 전략
- Q. 바이브코딩에 도움을 주는 도구?
  - A. 너무 많아서 필요한 대로 선택해서 사용 중

### 기술 선택 철학
1. **패스트팔로워 전략**: 다른 프로젝트의 입증된 방식 따라 적용
2. **메인스트림 선택**: 변화를 좇기보단 안착된 표준 기술 도입
   - 멘토님의 추천 기반

### 체계적인 개발 방식

- Markdown 파일로 prompt 문서화
- 체계적인 구조 정립

### 중요 원칙

#### Plan 문서의 중요성
> **커밋하기 전에 plan과 구현상태에 차이가 있는지 확인하고, plan에 업데이트가 필요하다고 판단되면 수정해라.**

- plan 내에 구현할 모든 내용이 포함되어야 함
- 동기화 오류 방지를 위해 plan을 적극 업데이트
- 코드 커밋 전 plan 검토 필수

#### 기록 관리
- 구두로 한 모든 내용도 Slack에 기록
- 어디서든 접근 가능한 형태로 유지
- 정보의 일관성 및 추적성 확보
