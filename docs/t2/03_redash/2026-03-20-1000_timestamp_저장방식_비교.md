# Timestamp 저장 방식 비교

## 저장 형식별 비교

### 1. Unix Timestamp (밀리초)
```
1773981663151
```
- **크기**: 8 bytes (int64)
- **정렬**: O(n log n)
- **비교**: O(1)
- **타임존**: UTC 기준

### 2. ISO 8601 문자열
```
"2026-03-20T13:41:03.151+09:00"
```
- **크기**: 32+ bytes
- **정렬**: O(n log n) × 문자열 길이
- **비교**: O(문자열 길이)
- **타임존**: 명시 필요

### 3. 사람이 읽기 쉬운 형식
```
"2026년 03월 20일 13시 41분 03초"
```
- **크기**: 50+ bytes
- **정렬**: 복잡한 파싱 필요
- **비교**: 거의 불가능
- **타임존**: 암시적

## 광고 로그 시스템에서의 실제 영향

### 데이터 규모 (일일 기준)
- 노출(impressions): 1,000만 건
- 클릭(clicks): 30만 건  
- 전환(conversions): 3만 건

### 저장 공간 비교
```python
# Unix timestamp (밀리초)
일일 데이터: 10,330,000 × 8 bytes = 82.64 MB

# ISO 8601 문자열
일일 데이터: 10,330,000 × 32 bytes = 330.56 MB

# 4배 차이!
```

### Athena 쿼리 성능
```sql
-- Unix timestamp (빠름)
SELECT * FROM impressions 
WHERE timestamp BETWEEN 1773900000000 AND 1773986400000;

-- 문자열 (느림)
SELECT * FROM impressions
WHERE timestamp BETWEEN '2026-03-20T00:00:00' AND '2026-03-20T23:59:59';
```

## 권장사항

1. **저장**: Unix timestamp (밀리초)
2. **전송**: Unix timestamp 또는 ISO 8601
3. **표시**: 사용자 로케일에 맞게 변환

이것이 대부분의 대규모 로그 시스템이 Unix timestamp를 사용하는 이유입니다.
