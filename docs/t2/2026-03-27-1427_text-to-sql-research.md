## 1. 개요

### 1.1 빅데이터·데이터 분석 시장의 폭발적 성장

오늘날 데이터는 조직이 보유한 가장 중요한 자산 중 하나로 자리잡고 있으며, 그 규모와 활용 수요는 매년 급격히 증가하고 있다.

#### 데이터 생성량의 기하급수적 증가

전 세계에서 하루에 생성되는 데이터는 **약 4억 2,700만 테라바이트(402.74 million TB)**에 달한다. 연간으로 환산하면 약 147 제타바이트(Zettabytes)에 해당한다. 2010년 전 세계 연간 데이터 생성량이 2 제타바이트에 불과했던 것과 비교하면, 불과 15년 만에 **약 74배** 증가한 수치다.

| 연도 | 연간 데이터 생성량 | 전년 대비 증가 |
|------|-----------------|--------------|
| 2010 | 2 ZB | - |
| 2015 | 15.5 ZB | ↑ 約 7.7배 (5년) |
| 2020 | 64.2 ZB | ↑ 約 4.1배 (5년) |
| 2023 | 120 ZB | ↑ 23.71% |
| 2024 | 147 ZB | ↑ 22.5% |
| 2025 | 181 ZB | ↑ 23.13% |
| 2026 (예측) | 221 ZB | ↑ 22.1% |

> 출처: [Statista / Exploding Topics - Amount of Data Created Daily (2026)](https://explodingtopics.com/blog/data-generated-per-day) (2026.02)
>
> **[출처 부연]** Exploding Topics는 Semrush가 운영하는 트렌드 분석 플랫폼으로, Statista 등 공신력 있는 데이터 소스를 바탕으로 시장 통계를 집계·발행한다. 위 수치는 2026년 2월 기준으로 업데이트된 자료다.

특히 주목할 점은, **전 세계 데이터의 약 90%가 최근 2년 이내에 생성**되었다는 사실이다. 소셜 미디어, IoT 기기, 스트리밍 서비스, 기업 내부 시스템 등 데이터 발생원이 폭발적으로 늘어나면서, 이를 의미 있게 분석하고 활용하는 능력의 중요성이 어느 때보다 커지고 있다.

#### 빅데이터 분석 시장 규모

데이터 분석 수요 급증에 따라 빅데이터 분석(Big Data Analytics) 시장 역시 가파른 성장세를 보이고 있다.

- **2023년** 글로벌 빅데이터 분석 시장 규모: **약 3,000억 달러(약 0.3조 달러)**
- **2032년** 예측 시장 규모: **약 1.1조 달러**
- **연평균 성장률(CAGR)**: **14.5%** (2024~2032년)

> 출처: [Allied Market Research - Big Data Analytics Market (2024-2032)](https://www.alliedmarketresearch.com/big-data-analytics-market) (2024.08)
>
> **[출처 부연]** Allied Market Research는 전 세계 주요 산업 분야의 시장 규모 및 전망을 분석하는 글로벌 시장 조사 기관이다. ISO/IEC 27001:2022 및 ISO 20252:2019 인증을 보유하고 있으며, 1,500개 이상의 기업 보고서와 공신력 있는 업계 자료를 기반으로 분석을 수행한다.

이 성장의 핵심 동력은 **데이터 기반 의사결정 문화의 확산**이다. 기업들은 경쟁 우위 확보, 운영 효율화, 고객 경험 개선을 위해 데이터 분석 투자를 빠르게 늘리고 있다.

#### 데이터 분석 역량 수요의 증가

데이터 처리·분석 시장의 성장은 관련 직군 수요 증가로도 이어지고 있다. 그 중심에는 **SQL(Structured Query Language)**이 있다. SQL은 데이터베이스에 저장된 데이터를 조회·집계·분석하기 위해 사용하는 표준 언어로, 엑셀의 필터·집계 기능을 수억 건의 대규모 데이터에 대해 처리할 수 있는 도구라고 이해하면 쉽다. _(SQL의 구체적인 개념과 구조는 [1.2절](#12-sql이란-무엇인가)에서 자세히 다룬다.)_

Stack Overflow 2025년 개발자 설문(전 세계 31,771명 참여)에 따르면, **SQL은 전체 개발자의 58.6%가 업무에서 사용하는 언어**로, JavaScript(66%), HTML/CSS(61.9%)에 이어 **3위**를 차지했다. Python(57.9%)을 넘어선 수치다.

#### 데이터 접근성 격차 — SQL을 모르면 데이터에 손을 댈 수 없다

그러나 문제는 **SQL 수요와 실제 활용 능력 사이의 격차**에 있다. 데이터를 가장 많이 필요로 하는 직군, 즉 기획자·마케터·운영 담당자 등 비개발 직군은 대부분 SQL을 다루지 못한다. 우아한형제들이 사내 구성원을 대상으로 실시한 설문에서도 **약 95%가 업무에 데이터를 활용하고 있다고 답했지만, 절반 이상이 SQL 활용에 어려움을 겪는다**고 응답했다.

이러한 격차는 개인 역량의 문제가 아니다. SQL은 학습에 상당한 시간이 필요하고, 실무 데이터에 맞는 비즈니스 로직까지 반영하려면 더 높은 진입 장벽이 생긴다. 결국 **데이터가 필요한 사람과 데이터에 접근할 수 있는 사람이 분리**되는 구조가 고착화된다.

##### 격차의 실태 — 수치로 보는 접근성 문제

이 격차를 구체적인 수치로 살펴보면 문제의 심각성이 더욱 선명해진다.

| 항목 | 수치 | 출처 |
|------|------|------|
| 업무에서 SQL을 쓰는 전체 개발자 비율 | **58.6%** | Stack Overflow Developer Survey 2025 |
| 데이터 직군 채용 공고 중 SQL을 필수로 요구하는 비율 | **약 65%** | DataCamp, 2023 |
| 스스로 데이터 리터러시를 갖췄다고 응답한 디지털 네이티브(16~21세) 비율 | **43%** | Exasol / TechRepublic, 2021 |
| 현재 측정 시스템이 속도·정확도·신뢰성에서 미달이라는 마케터 비율 | **75%** | IAB/BWG State of Data 2026 |

가장 디지털 친화적인 세대조차 절반도 안 되는 43%만이 스스로 데이터 리터러시가 있다고 응답했다는 사실은, 이 격차가 특정 세대나 직군의 문제가 아님을 방증한다. 데이터 교육과 현장 수요 사이의 구조적 불일치가 근본 원인이다.

##### 격차가 만드는 병목 — 조직 내 의존성 사슬

SQL을 모르는 구성원은 데이터가 필요할 때마다 반드시 누군가에게 요청해야 한다. 이 의존성 사슬은 조직 전체에 걸쳐 세 가지 형태의 비용을 발생시킨다.

```
[비개발 직군의 데이터 요청 흐름]

 비개발 직군        데이터/개발 팀
(기획자·마케터)          │
      │                  │
      │── 요청 전달 ────▶│
      │   (수 일 소요)   │
      │                  │── SQL 작성
      │                  │── 쿼리 실행
      │◀── 결과 반환 ────│
      │                  │
      ↓                  ↓
 늦은 의사결정        반복 단순 작업
(기회비용 발생)     (고부가가치 업무 집중도 하락)
```

1. **커뮤니케이션 비용**: 요구사항을 글로 설명하고, 오해를 바로잡고, 재요청하는 과정 반복
2. **대기 시간**: 데이터 팀의 백로그에 밀려 수 시간~수 일이 소요 → 실시간 의사결정 불가능
3. **기회비용**: IAB/BWG(2026) 보고서는 AI로 루틴 데이터 작업을 대체할 경우 **$6.2B(약 6조 2천억원)의 생산성 회복**이 가능하다고 추산 — 이는 현재 비개발 직군이 데이터 접근 과정에서 낭비하는 비용의 역산 추정치다.

##### 비즈니스 로직의 벽 — "SQL을 안다"고 끝이 아니다

SQL 문법을 학습했다 해도 실무에서는 또 다른 장벽이 기다린다. 아래와 같은 현업의 질문을 정확한 SQL로 변환하려면 단순한 언어 지식을 넘어선 능력이 필요하다.

> **예시 질문**: "지난달 대비 이번 달 ROAS가 10% 이상 떨어진 캠페인 중, 노출수는 증가했지만 클릭률이 감소한 캠페인만 추려서 일별 추이를 보여줘"

이를 SQL로 변환하려면:
- `impressions`, `clicks`, `conversions` 테이블 간 JOIN 구조 이해
- ROAS(광고 수익률) 메트릭의 비즈니스 정의 숙지
- 기간 비교를 위한 날짜 파티션 처리
- 복합 조건(증가/감소 동시 판별) 로직 구현

기술적 지식과 도메인 지식이 동시에 요구된다. 이 두 가지를 모두 갖춘 비개발 직군은 사실상 존재하지 않는다.

이것이 바로 현재 많은 조직이 직면한 **데이터 접근성 문제**이며, Text-to-SQL 기술이 주목받는 근본적인 이유다.

### 1.2 SQL이란 무엇인가?

**SQL(Structured Query Language, 구조화 질의 언어)**은 데이터베이스에 저장된 데이터를 조회·추가·수정·삭제하기 위해 사용하는 전용 언어다. 쉽게 말해, **데이터에게 말을 거는 방식**이다.

#### SQL을 사용하면 무엇을 할 수 있나?

엑셀 스프레드시트를 떠올려보자. 엑셀에서 특정 조건의 행만 필터링하거나, 합계·평균을 구하거나, 두 시트를 합치는 작업을 할 수 있다. SQL은 이 모든 것을 **수백만~수억 건의 대용량 데이터**에 대해 빠르게 처리할 수 있는 언어다.

| 목적 | SQL 예시 (개념) | 엑셀 비유 |
|------|---------------|----------|
| 특정 데이터 조회 | "지난 달 클릭수 상위 10개 광고 보여줘" | 필터 + 정렬 |
| 집계·통계 | "캠페인별 평균 전환율 계산해줘" | SUMIF, AVERAGEIF |
| 테이블 결합 | "노출 데이터와 클릭 데이터를 광고ID 기준으로 합쳐줘" | VLOOKUP |
| 기간 비교 | "이번 달과 지난 달 매출을 나란히 보여줘" | 피벗 테이블 |

#### SQL의 기본 구조 — 한 눈에 보기

SQL은 영어 문장처럼 읽힌다. 아래는 가장 기본적인 형태다.

```sql
SELECT  캠페인명, 클릭수, 전환수        -- 어떤 데이터를 볼지
FROM    광고_성과_테이블                -- 어느 테이블에서
WHERE   날짜 = '2026-03-27'           -- 어떤 조건으로
ORDER BY 클릭수 DESC                   -- 어떤 순서로
LIMIT   10;                           -- 몇 개까지
```

> 이 쿼리는 "2026-03-27의 광고 성과 테이블에서, 캠페인명·클릭수·전환수를 클릭수가 높은 순으로 10개만 보여줘"라는 의미다.

#### 왜 SQL을 배우기 어려운가?

SQL 자체의 문법은 비교적 직관적이지만, **실무에서 쓰기 어려운 이유**가 따로 있다.

1. **테이블 구조를 알아야 한다**: 데이터가 어느 테이블에, 어떤 컬럼으로 저장되어 있는지 사전에 파악해야 한다.
2. **비즈니스 로직을 코드로 표현해야 한다**: "ROAS가 10% 이상 하락한 캠페인"처럼, 업무 개념을 수식과 조건절로 변환하는 능력이 필요하다.
3. **조인(JOIN)·서브쿼리 등 고급 문법**: 여러 테이블을 연결하거나 복잡한 조건을 중첩하면 쿼리 난이도가 급격히 높아진다.
4. **데이터베이스마다 문법이 조금씩 다르다**: MySQL, PostgreSQL, Athena(AWS) 등 환경에 따라 세부 문법이 상이하다.

결국 **SQL은 배우는 데 시간이 걸리고, 쓰지 않으면 금방 잊혀진다**. 본업이 따로 있는 비개발 직군에게는 높은 진입 장벽이 될 수밖에 없다.

## 2. AS-IS vs TO-BE

### AS-IS (현재 상황)
- 비개발 직군(기획자, 마케터, 운영 담당자)은 데이터가 필요할 때마다 개발자에게 요청
- 데이터 추출까지 수 일이 소요 → 의사결정 지연
- 반복적인 요청으로 인한 커뮤니케이션 비용 증가
- SQL을 아는 구성원만 데이터에 직접 접근 가능 → 데이터 민주화 저해
- **측정 워크플로우가 여전히 대부분 수작업·저속**: 마케터의 75%가 현재 측정 시스템이 속도·정확도·신뢰성 면에서 미달이라고 응답 (IAB/BWG, 2026)
- **인사이트 도출보다 데이터 연결에 더 많은 시간 소비**: 팀들이 사일로화된 데이터를 이어붙이는 데 대부분의 시간을 소모 — 이 구조적 낭비를 AI로 해소하면 **$6.2B(약 6조 2천억원)의 생산성 회복**이 가능하다고 추산 (IAB/BWG, 2026)
  - 이는 역으로, 현재 비개발 직군이 **데이터 접근·정리 작업에 투입하는 비용(시간·인력)**이 그만큼 크다는 의미

### TO-BE (Text-to-SQL 도입 후)
- 자연어 질문(한국어) → 자동으로 SQL 생성 → 결과 반환
- 비개발 직군도 Slack, 웹 인터페이스를 통해 직접 데이터 조회
- 데이터 추출 시간 단축 (수일 → 수초)

### 3.3 어떤 직군에서 SQL을 쓰는가?

SQL은 개발자 전유물이 아니다. DataCamp 분석 기준, 아래 직군 모두에서 SQL이 핵심 도구로 활용된다.

| 직군 | SQL을 어디에 쓰나? |
|------|-------------------|
| **마케터** | 캠페인 성과 데이터 조회, 고객 세그먼트 분석 |
| **금융 전문가** | 매출·비용 KPI 추출, 재무 리포트 생성 |
| **의료 데이터 분석가** | 환자 데이터 조회, 임상 대시보드 생성 |
| **백엔드/앱 개발자** | 사용자 데이터 저장·조회, 실시간 처리 |
| **데이터 애널리스트** | 트렌드 분석, 비즈니스 인사이트 도출 |
| **비즈니스 애널리스트** | KPI 추적, 경영 의사결정 지원 |
| **데이터 사이언티스트** | 데이터 전처리, 피처 엔지니어링, 탐색적 데이터 분석(EDA) |

> **핵심 시사점**: SQL은 특정 직군의 전문 기술이 아니라, **데이터를 다루는 모든 직군의 공통 언어**다. 바꿔 말하면, SQL을 모르는 구성원은 데이터에 접근하기 위해 항상 다른 사람에게 의존해야 한다.

## 4. Text-to-SQL

### 4.1 기술 개요

#### 정의

**Text-to-SQL**은 사용자가 자연어(한국어, 영어 등)로 입력한 질문을 자동으로 SQL 쿼리로 변환해 데이터베이스에서 결과를 반환하는 기술이다. 비개발 직군도 별도의 SQL 학습 없이 데이터에 직접 접근할 수 있게 한다.

#### 공통 기술 패턴

```
[사용자 자연어 질문]
       ↓
[LLM + RAG (도메인 지식 검색)]
       ↓
[SQL 생성 및 검증]
       ↓
[쿼리 실행 (Athena / Trino / Redash)]
       ↓
[결과 반환 (Slack / 웹 인터페이스)]
```

#### 주요 구성 요소

| 컴포넌트 | 주요 기술 |
|----------|-----------|
| LLM 모델 | GPT-4o, Claude, Gemini-2.5 Pro |
| Vector DB | ChromaDB, OpenSearch |
| 오케스트레이션 | LangChain, LangGraph, Vanna AI |
| 검색 품질 개선 | Reranker (jina-reranker-v2 등) |
| 인터페이스 | Slack Bot, 웹 UI |
| 쿼리 엔진 | Athena, Trino, Redash |

#### 성공 요소

1. **도메인 지식 품질이 핵심**: 모델 성능보다 RAG에 들어가는 참조 데이터(테이블 스키마, 샘플 쿼리, 비즈니스 용어집)의 품질이 정확도를 결정
2. **Reranker 도입**: Bi-encoder 검색 후 Cross-encoder Reranker로 재정렬 → 정확도 향상
3. **검증 단계 포함**: SQL 생성 후 EXPLAIN, LIMIT 등을 통한 사전 검증
4. **피드백 루프**: 오답/수정 사례를 RAG 학습 데이터에 지속 반영

## 5. 글로벌 데이터 활용 현황

### IAB/BWG "State of Data 2026" 마케팅 측정 보고서

> 출처: [Martech.org - "75% of marketers say their measurement systems are falling short"](https://martech.org/75-of-marketers-say-their-measurement-systems-are-falling-short/) (2026.02.03)

#### 5.1 비개발 직군(마케터)의 데이터 분석 비용 — 수치 근거

마케터는 대표적인 비개발 데이터 활용 직군이다. 아래 수치는 이들이 현재 데이터 분석에 투입하는 **비용(input)**이 얼마나 큰지를 보여준다.

| 항목 | 수치 | 의미 |
|------|------|------|
| 현재 측정 시스템이 **속도·정확도·신뢰성**에서 미달이라고 응답한 마케터 비율 | **75%** | 10명 중 7~8명이 현재 체계에 불만 → 비효율적 반복 작업 발생 |
| **사일로 데이터 연결**에 소비되는 시간 비중 | 정성적 묘사 ("더 많은 시간") | 인사이트 도출보다 데이터 정리·연결에 더 많은 리소스 투입 |
| AI 루틴 작업 대체 시 회수 가능한 **생산성 가치** | **$6.2B (약 6조 2천억원)** | 현재 비개발 직군이 데이터 작업에 낭비하는 비용의 역산 추정치 |
| 더 빠른 측정 피드백 루프가 창출할 **미디어 투자 가치** | **$26.3B (약 26조 3천억원)** | 지연된 의사결정이 초래하는 기회비용 규모 |
| 게이밍 채널이 믹스 모델에서 **과소 표현**된다고 응답한 마케터 비율 | **77%** | 데이터 접근 불편으로 인해 올바른 채널 평가조차 불가능 |

> **[출처 부연] IAB/BWG "State of Data 2026"이란?**  
> IAB(Interactive Advertising Bureau)와 BWG(Brand-side Working Group)가 공동 발간한 마케팅 측정 현황 보고서. 바이사이드(광고주·마케터) 실무자 대상 설문을 포함하며, 측정 기술·AI 도입 현황을 심층 분석한다. 2026년 2월 발표.

#### 5.2 비개발 직군의 데이터 접근 문제 — 구조적 원인

기사는 비개발 직군(마케터·기획자)이 데이터 분석에 높은 비용을 지불하는 이유를 세 가지로 설명한다:

1. **단편화된 데이터(Fragmented Data)**: 여러 채널의 데이터가 분산되어 있어 한 곳에서 조회 불가능  
   → 팀이 직접 데이터를 이어붙이는 수작업 발생

2. **느린 피드백 루프(Slow Feedback Loops)**: 측정 모델 업데이트가 연간·분기 단위 → 실시간 의사결정 불가능  
   → 미디어 예산이 **불완전한 정보**를 기반으로 집행됨

3. **기술 장벽(Technical Barrier)**: 정교한 측정 기법(멀티터치 어트리뷰션, 크로스채널 리프트 분석)은 **고급 기술 인력만 사용 가능**  
   → 비개발 직군은 단순 집계 데이터에만 접근 가능, 심층 분석 의존도 높음

> **핵심 시사점**: 비개발 직군의 데이터 분석 비용은 단순 인건비가 아니다. 잘못된 의사결정으로 인한 **예산 오배분**, 지연된 인사이트로 인한 **기회비용**, 그리고 반복적인 수작업으로 인한 **생산성 손실**이 복합적으로 발생한다. IAB/BWG 보고서는 이 총합을 $26.3B(미디어 가치) + $6.2B(생산성)로 추산했다.

#### 5.3 AI 도입 현황 — 전환 속도

| 항목 | 수치 |
|------|------|
| 이미 AI 측정 프로그램을 확장 중인 바이사이드 마케터 비율 | **약 50%** |
| 2027년까지 AI 도입 예정인 미도입 팀 비율 | **70%+** |
| 파트너 계약에 AI 관련 조항을 추가한 바이사이드 팀 비율 | **37%** |

**핵심 인사이트**: 마케터 10명 중 7~8명은 현재 데이터 분석 체계에 불만을 가지고 있으며, AI 기반 데이터 분석 도구로의 전환이 빠르게 이루어지고 있다. 특히 분석 팀은 기획 팀보다 AI 기반 워크플로우를 도입할 가능성이 **2배 이상** 높다 — 기술 장벽이 여전히 존재한다는 반증이다.

### 6.1 비개발 직군의 데이터 리터러시 격차 (글로벌)

### 6.2 우아한형제들 사내 설문 — 국내 기업의 직접적 증거 (2024)

우아한형제들의 AI 데이터 분석가 '물어보새' 개발 TF는 서비스 기획 전, **전 구성원을 대상으로 사내 데이터 활용 현황을 설문 조사**했다. 이 결과는 비개발 직군이 데이터 접근에서 겪는 구체적 고충을 직접 수치로 드러낸다.

| 항목 | 수치 | 의미 |
|------|------|------|
| 업무에 데이터를 활용하는 구성원 비율 | **약 95%** | 사실상 전 구성원이 데이터를 필요로 함 |
| SQL 활용에 **어려움을 겪는** 구성원 비율 | **절반 이상 (50%+)** | 데이터를 필요로 하지만 SQL로 직접 조회하지 못함 |
| 어려움의 주요 원인 ① | **SQL 학습 시간 부족** | 본업이 따로 있어 SQL을 별도로 배울 여유가 없음 |
| 어려움의 주요 원인 ② | **비즈니스 로직을 SQL에 반영하기 어려움** | 도메인 지식을 쿼리로 표현하는 것이 기술적 장벽 |
| 어려움의 주요 원인 ③ | **추출 데이터의 신뢰도에 대한 불확실성** | 쿼리가 맞는지 확신이 없어 결과를 믿기 어려움 |

> **직접 인용**: "구성원의 **95%가 데이터를 활용해 업무를 수행**하고 있었지만, **절반 이상은 SQL을 업무에 활용하고 싶어도 학습 시간이 부족하거나, SQL 구문에 비즈니스 로직과 다양한 추출 조건을 반영하여 작성하는 데 어려움**을 느끼고 있었다."
>
> 출처: [우아한형제들 기술 블로그 - AI 데이터 분석가 '물어보새' 등장 – 1부 (2024.07.12)](https://techblog.woowahan.com/18144/)

### 6.3 비개발 직군 고충의 구조적 원인 분해

위 조사 결과를 바탕으로, 비개발 직군이 데이터 접근에 어려움을 겪는 원인은 크게 세 가지로 분류된다:

#### ① 기술 장벽 (Technical Barrier)
- SQL은 전문 기술이다. 배우는 데 시간이 필요하고, 일회성으로 배운 내용은 금방 잊혀진다.
- **비개발 직군은 본업이 따로 있다** — 기획자는 기획을, 마케터는 마케팅을 해야 한다. SQL 학습에 투자할 여력 자체가 없다.
- 데이터 직군 채용 공고의 **65%가 SQL을 필수 스킬**로 요구하지만 (DataCamp, 2023), 비개발 직군 채용에서는 이것이 요건이 아니다.

#### ② 도메인 로직 장벽 (Domain Logic Barrier)
- SQL 문법을 안다고 해도, 복잡한 비즈니스 로직을 정확한 쿼리로 표현하는 것은 별개의 능력이다.
- **"광고 ROI가 높은 캠페인 상위 10개를 지난달 대비 비교해줘"** — 이 요구사항을 SQL로 변환하려면 테이블 구조 이해 + 비즈니스 메트릭 정의 + 쿼리 최적화가 동시에 필요하다.

#### ③ 의존성 장벽 (Dependency Barrier)
- 결국 비개발 직군은 데이터 팀 또는 개발자에게 요청을 해야 한다.
- 이 과정에서 **커뮤니케이션 비용, 대기 시간, 요구사항 오해**가 발생한다.
- 데이터 팀의 반복적인 단순 쿼리 작업이 늘어나고, **고부가가치 업무 집중도가 낮아진다**.

### 6.5 비개발 직군 데이터 의존 현황 — 역할별 실태

DataCamp 직군별 분석에 따르면, SQL이 필요한 업무를 담당하지만 SQL을 직접 쓰지 못하는 비개발 직군은 다음과 같다:

| 직군 | 데이터 필요 업무 | SQL 없이 하는 방식 | 문제점 |
|------|---------------|------------------|--------|
| **마케터** | 캠페인 성과 조회, 고객 세그먼트 분석 | 데이터 팀에 요청 | 수 일 소요, 실시간 대응 불가 |
| **기획자** | KPI 추출, 사용자 행동 분석 | 대시보드 의존 | 커스텀 분석 불가, 새로운 질문에 답 못함 |
| **운영 담당자** | 이상 지표 탐지, 현황 파악 | 정기 리포트 대기 | 즉각 대응 불가, 늦은 인사이트 |
| **경영진** | 비즈니스 KPI 현황 파악 | BI 툴 정해진 메뉴만 활용 | 즉흥적 질문에 답 못함 |

### 6.6 핵심 결론: Text-to-SQL이 해결하는 것

비개발 직군이 데이터에 접근하지 못하는 문제는 **의지의 문제가 아니라 구조의 문제**다.

- 95%의 구성원이 데이터를 필요로 하지만 (우아한형제들 내부 조사)
- 절반 이상이 SQL이라는 기술 장벽 앞에 막혀 있고
- 결국 데이터 팀에 의존하거나, 불완전한 정보로 의사결정을 내린다

Text-to-SQL은 이 구조적 단절을 해소하는 도구다:

```
[현재 구조]
비개발 직군 → SQL 장벽 → 데이터 팀 의존 → 수 일 대기 → 늦은 인사이트

[Text-to-SQL 도입 후]
비개발 직군 → 자연어 질문 → 즉각적인 SQL 생성 + 결과 반환 → 빠른 의사결정
```
