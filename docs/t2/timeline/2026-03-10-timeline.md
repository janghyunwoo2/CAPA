## 📋 상세 작업 내역

### 1. Hourly ETL 실행 오류 (오전 초반)

#### 문제 인식
- **증상**: 
  - `hourly_etl.py` 실행 시 INSERT INTO 쿼리 실패
  - "Queries of this type are not supported" 오류 발생
  - ETL 프로세스 전체 중단

#### 원인 분석
1. **Athena의 제약사항**
   - AWS Athena는 Presto 기반 읽기 전용 엔진
   - INSERT/UPDATE/DELETE 미지원
   - CTAS만 지원하지만 경로 지정 불가

2. **기존 코드의 문제**
   - Athena에서 지원하지 않는 INSERT INTO 구문 사용
   - 테이블 파티션에 직접 데이터 삽입 시도

#### 해결 방법
```python
# Python + Boto3로 직접 저장 방식으로 변경
1. PyAthena로 SELECT 결과 조회
2. Pandas DataFrame으로 변환
3. 로컬에서 Parquet 파일 생성
4. Boto3로 S3에 업로드
5. MSCK REPAIR TABLE로 파티션 등록
```

#### 해결 여부
- ✅ **완료**: Python 기반 워크플로우로 전환하여 해결

### 2. 테이블 삭제 오류 (오전 중반)

#### 문제 인식
- **증상**: 
  - `EntityNotFoundException` 발생
  - DESCRIBE 쿼리 실행 시 테이블 찾을 수 없음
  - 의존 테이블(impressions, clicks) 누락

#### 원인 분석
1. **테이블 존재 확인 로직의 문제**
   - `_table_exists()` 메서드의 예외 처리 미흡
   - athena_utils의 재시도 중 예외 전파

2. **의존성 검증 부재**
   - impressions, clicks 테이블 존재 여부 미확인
   - 필수 테이블 없이 ETL 실행 시도

#### 해결 방법
```python
# 의존 테이블 검증 추가
def _validate_dependencies(self) -> bool:
    required_tables = ['impressions', 'clicks']
    for table in required_tables:
        if not self._table_exists(table):
            logger.error(f"Required table {table} not found")
            return False
    return True
```

#### 해결 여부
- ✅ **완료**: 의존성 검증 로직 추가 및 테이블 초기화 스크립트 실행

### 3. 테이블 파티셔닝 구조 개선 계획 (오전 후반)

#### 문제 인식
- **증상**: 
  - 단일 dt 문자열 파티션 사용으로 성능 저하
  - Athena 쿼리 시 전체 S3 스캔 발생
  - 파티션 프루닝 최적화 불가

#### 원인 분석
1. **현재 구조의 한계**
   - dt STRING 단일 파티션 (예: "2026-03-10-14")
   - WHERE 절에 year/month/day 조건 사용 불가
   - S3 파일 구조가 파티션 구조와 불일치

2. **원본 테이블과의 불일치**
   - impressions/clicks는 year/month/day/hour 파티션
   - ad_combined_log는 dt 단일 파티션

#### 해결 방법
```sql
-- 개선된 테이블 구조
CREATE EXTERNAL TABLE ad_combined_log (
    -- 컬럼 정의
)
PARTITIONED BY (
    year STRING,
    month STRING,
    day STRING,
    hour STRING
)
STORED AS PARQUET
LOCATION 's3://bucket/summary/ad_combined_log/'
```

#### 해결 여부
- ⏳ **계획 수립**: 구조 개선 계획 완성 (실행은 향후 예정)

### 4. CSV & 메타데이터 파일 축적 (오후 초반)

#### 문제 인식
- **증상**: 
  - S3 `athena-results/temp/` 폴더에 파일 계속 축적
  - 매일 ~50개 파일 생성 (CSV + metadata)
  - 월간 1,500+ 파일 예상

#### 원인 분석
1. **Athena 쿼리 메커니즘**
   - 모든 쿼리 결과가 S3에 자동 저장
   - 각 쿼리마다 고유 ID로 폴더 생성
   - 자동 삭제 정책 부재

2. **반복 실행으로 인한 축적**
   - hourly_etl: 24회/일
   - daily_etl: 1회/일
   - 각 실행마다 2개 파일 생성

#### 해결 방법
```json
// S3 Life Cycle Policy 적용
{
  "Rules": [{
    "ID": "DeleteAthenaTemp",
    "Filter": { "Prefix": "athena-results/temp/" },
    "Expiration": { "Days": 1 }
  }]
}
```

#### 해결 여부
- ✅ **완료**: S3 라이프사이클 정책 적용으로 자동 정리

### 5. Athena 해시 테이블명 자동 생성 (저녁)

  - 예상하지 않은 테이블 축적

   - Glue가 모든 CSV를 테이블로 인식

### 6. S3 경로 불일치 및 타입 불일치 (종합)

#### 문제 인식
- **증상**: 
  - NoSuchKey 오류 발생
  - HIVE_BAD_DATA: Boolean 타입 불일치
  - Parquet 스키마 미정의

#### 원인 분석
1. **S3 경로 하드코딩**
   - 실제 Athena 결과 경로와 코드의 경로 불일치
   
2. **타입 변환 누락**
   - Athena는 모든 결과를 문자열로 반환
   - Boolean과 숫자 타입 변환 필요

3. **Parquet 스키마 부재**
   - 명시적 스키마 정의 없이 저장

#### 해결 방법
```python
# 1. Athena 메타데이터 직접 조회
results = self.executor.get_query_results(query_id)

# 2. 명시적 타입 변환
df['is_click'] = df['is_click'].str.lower() == 'true'
df[numeric_columns] = df[numeric_columns].apply(pd.to_numeric)

# 3. PyArrow 스키마 정의
schema = pa.schema([
    ('impression_id', pa.string()),
    ('is_click', pa.bool_()),
    ('timestamp', pa.int64()),
    # ... 31개 컬럼 모두 정의
])
```

#### 해결 여부
- ✅ **완료**: 모든 타입 문제 해결
