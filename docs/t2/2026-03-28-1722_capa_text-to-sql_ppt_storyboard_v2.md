## 스토리라인 요약 (v2)

```
[배경] 데이터 폭발적 증가
    ↓
[문제] 비개발자는 데이터에 손을 댈 수 없다 (격차)
    ↓ ← 브리지: "이 격차를 해결하는 기술이 있다면?"
[해결] Text-to-SQL — 자연어로 데이터에 접근
    ↓ ← 브리지: "이미 누군가 먼저 해봤다 — 그런데 우리 도메인은 뭔가?"
[증거] 국내외 도입 사례 3건 검증
    ↓
[도메인] 우리가 다루는 광고 데이터 (퍼널 구조 + 핵심 KPI)
    ↓
[CAPA] 우리가 만든 것 — 전체 파이프라인 (단계별 화살표 흐름)
    ↓ ← 인간적 브리지: "가장 인상 깊었던 순간"
[증거] 실제 데모 — 질문 → SQL → 결과
    ↓
[비교] AS-IS vs TO-BE
    ↓
[규모] 숫자로 보는 CAPA
    ↓
[교훈] 시행착오 + 핵심 교훈 3가지
    ↓
[CTA] 지금 써보세요 / 함께 발전시켜요
```

### Slide 01 — 표지

**제목**: 자연어로 데이터를 묻다  
**부제**: CAPA Text-to-SQL — 비개발자를 위한 AI 데이터 파이프라인

|------|----------------|
| 2010 | 2 ZB |
| 2020 | 64 ZB |
| 2024 | 147 ZB |
| 2026 (예측) | 221 ZB |

> **[출처]** Exploding Topics (Semrush 운영) — Statista 집계 기반

**시각화 제안**: 2 ZB → 221 ZB 상승 애니메이션 바 차트. 마지막 "221 ZB"에서 멈추며 한 줄 메시지 등장

### Slide 04 — 문제: 데이터가 필요한 사람 ≠ 데이터에 접근할 수 있는 사람

**헤드라인**: SQL을 모르면 데이터에 손을 댈 수 없다

**격차 시각화 (2-컬럼 대비)**:

```
왼쪽: 데이터가 필요한 사람        오른쪽: 데이터에 접근할 수 있는 사람
──────────────────────           ───────────────────────────────────
마케터 / 기획자 / 운영            개발자 / 데이터 엔지니어
        ↓                                   ↓
   "이거 뽑아줘" 요청  ──→──→──→  수 시간 ~ 수 일 후 결과 전달
```

**핵심 수치 (슬라이드 하단 강조)**:
- 마케터의 **75%**: 측정 시스템이 속도·정확도·신뢰성에서 미달  
  > [출처] IAB/BWG "State of Data 2026" — https://martech.org/75-of-marketers-say-their-measurement-systems-are-falling-short/ (2026.02)
- 비개발 직군의 **절반 이상**: SQL 활용에 어려움  
  > [출처] 우아한형제들 기술 블로그 — https://techblog.woowahan.com/18144/

**한 줄 메시지**: "데이터가 필요한 사람과 데이터에 접근할 수 있는 사람이 다르다"

**시각화 제안**: 두 그룹을 대비하는 2-컬럼 레이아웃. 화살표와 시간 표시로 병목 강조

**헤드라인**: 데이터 접근 병목이 만드는 3가지 비용

### Slide 06 — 해결책: Text-to-SQL

**헤드라인**: 자연어로 물으면, AI가 SQL을 만든다

**파이프라인 다이어그램 (5단계, 화살표 애니메이션)**:

```
①  사용자 자연어 질문 (한국어 OK)
        ↓
②  LLM + RAG (도메인 지식 자동 검색)
        ↓
③  SQL 자동 생성 + 검증
        ↓
④  쿼리 실행 (Athena)
        ↓
⑤  결과 반환 (Slack / 웹 인터페이스)
```

**예시 대화 (시각적 말풍선 형태)**:

```
👤 사용자: "지난달 대비 이번달 클릭률이 10% 이상 떨어진 캠페인 알려줘"
🤖 CAPA:   SELECT campaign_id, ... → 실행 → 결과 테이블 반환
```

**한 줄 메시지 (굵게 크게)**: "SQL을 몰라도 된다 — 질문만 하면 된다"

**시각화 제안**: 5단계 세로 플로우 차트 (클릭마다 1단계씩 등장하는 애니메이션)

**도메인 선택 이유 — 3가지 이유 카드 (상단, 가로 배치)**:

```
┌─────────────────────────┐  ┌─────────────────────────┐  ┌─────────────────────────┐
│  📊 데이터가 가장 많다    │  │  💡 질문이 즉각적이다     │  │  💸 돈이 걸려 있다       │

│ 클릭 한 번 = 데이터 1건  │  │ "이 광고 잘 되고 있어?"   │  │ 광고비 낭비 1%만 줄여도  │
```

**광고 이벤트 퍼널 다이어그램 (넓은 바 → 좁아지는 깔때기 형태)**:

```
┌──────────────────────────────────────────────────┐
│  👁  광고 노출 (Impression)           100%        │  ← 사용자가 광고를 봤다
└─────────────────────┬────────────────────────────┘
                      │ CTR 1~5%
          ┌───────────▼──────────────────┐
          │  👆 광고 클릭 (Click)   1~5% │  ← 광고를 눌렀다
          └───────────┬──────────────────┘
                      │ CVR 1~10%
                ┌─────▼──────────┐
                │ 💰 전환    ~0.1%│  ← 구매·설치·가입
                └────────────────┘
```

**KPI 5종 카드 (퍼널 옆 배치)**:

| 지표 | 계산식 | 의미 |
|------|--------|------|
| **CTR** (클릭률) | 클릭수 ÷ 노출수 × 100 | 광고 소재 매력도 |
| **CVR** (전환율) | 전환수 ÷ 클릭수 × 100 | 랜딩페이지·상품 경쟁력 |
| **CPC** (클릭당 비용) | 총 광고비 ÷ 클릭수 | 트래픽 효율성 |
| **ROAS** (광고 수익률) | 전환 매출 ÷ 광고비 × 100 | 캠페인 수익성 |
| **CPM** (1천 노출당 비용) | 총 광고비 ÷ 노출수 × 1,000 | 브랜딩 비용 효율 |

**한 줄 메시지**: "광고가 데이터가 많고, 질문이 많고, 돈이 걸려 있기 때문에 선택했다 — 그리고 이 퍼널이 그 데이터의 전부다"

**시각화 제안**:
- **상단**: 도메인 선택 이유 3-카드 (가로 배치, 아이콘 포함) — 슬라이드 진입 시 첫 등장
- **하단**: 왼쪽 퍼널 다이어그램 + 오른쪽 KPI 카드 5개 배치 — 애니메이션으로 단계별 등장
- 흐름: "왜 광고?" → (카드 3개) → "광고 데이터는 이렇게 생겼다" → (퍼널·KPI)

### Slide 09 — CAPA가 만든 것: 전체 파이프라인 (비개발자용 5단계)

**헤드라인**: 광고 로그가 질문의 답이 되기까지 — 5단계

> *(비개발자 우선. 전체 기술 스택 상세는 발표자 노트로 이동)*

**비개발자용 단순 5단계 화살표 흐름 (크게, 아이콘 포함)**:

```
① 로그 생성         ②  실시간 수집         ③  저장
   광고가 노출·         Kinesis가              S3에
   클릭·전환될 때       이벤트를 받아           Parquet 형태로
   자동으로 기록        → 파이어호스 전달        시간별 파티션 저장
        ↓
④ ETL 변환 (Airflow)                       ⑤  AI 질문 응답
   매시간/매일                                Slack에서 자연어로
   집계·정제하여                              질문하면 Vanna API가
   분석 가능한 형태로                         SQL 생성 → Athena 실행
   변환
```

**핵심 메시지 (슬라이드 하단)**:
> "수집 → 저장 → 정제 → 질문 — 이 4단계가 30만 건의 광고 로그를 자동으로 처리합니다"

**시각화 제안**: 아이콘 5개 + 화살표로 이어지는 단순 플로우. 클릭 애니메이션으로 단계별 등장

**발표자 노트 (기술 세부 사항)**:
- 수집: Python + Faker (EKS Pod), Kinesis Data Streams × 3 (imp/clk/cvs)
- 저장: Kinesis Firehose × 3, S3 Parquet/ZSTD, year/month/day/hour 파티셔닝
- 변환: Airflow 2.7+ KubernetesExecutor, Hourly(매시간 10분) / Daily(매일 02시)
- 쿼리: AWS Glue Catalog + Athena Serverless (capa_ad_logs DB)
- AI: Vanna AI + ChromaDB + jina-reranker-v2 + Slack Bolt
- IaC: Terraform (전체 AWS 리소스 코드 관리)

### Slide 10 — CAPA가 만든 것: 데이터 수집 & 적재 구조

**헤드라인**: 광고 로그 3종이 실시간으로 S3에 쌓이는 방법

**로그 3종 + 트래픽 패턴 (핵심만, 2-컬럼 배치)**:

**왼쪽 — 이벤트 3종**:

| 이벤트 | 의미 | 발생 비율 |
|--------|------|---------|
| **Impression** | 광고 노출 | 기준 (100%) |
| **Click** | 광고 클릭 | CTR 1~5% |
| **Conversion** | 전환 (구매 등) | CVR 1~10% |

**오른쪽 — 시간대별 트래픽 패턴**:

| 시간대 | 트래픽 | 시간당 생성량 |
|--------|--------|------------|
| 00~07시 (새벽) | 낮음 | ~800~2,000개 |
| 11~14시 (점심) | 보통 | ~15,000~30,000개 |
| 17~21시 (저녁 피크) | **높음** | **~60,000개** |
| **일일 합계** | — | **약 28만~30만개** |

**S3 경로 핵심만 (트리 축약)**:

```
s3://capa-data-lake-.../
  raw/impressions | clicks | conversions  ← year/month/day/hour/
  summary/ad_combined_log                 ← Hourly 집계 (hour 파티션)
  summary/ad_combined_log_summary         ← Daily 집계 (day 파티션)
```

**시각화 제안**: 2-컬럼 레이아웃(이벤트표 + 트래픽 패턴 바차트). S3 경로는 아이콘 트리로 단순화

**발표자 노트**: S3 전체 경로 구조, Firehose 버퍼 설정(60초), Parquet/ZSTD 압축 세부 사항

### Slide 11 — CAPA가 만든 것: ETL 파이프라인 (Airflow)

**헤드라인**: 원시 로그 → 분석 가능한 형태로, 자동으로

**ETL 2단계 핵심 구조 (시각 강조)**:

```
매시간 10분 (Stage 1)                  매일 02시 (Stage 2)
────────────────────                   ─────────────────────
Impression + Click                     ad_combined_log(24h)
     ↓ LEFT JOIN                             + Conversion
 ad_combined_log                             ↓ LEFT JOIN
 (시간별 · 27필드)                    ad_combined_log_summary
                                            (일별 · 35필드)
```

**테이블 비교 (핵심 3행)**:

| 항목 | ad_combined_log | ad_combined_log_summary |
|------|----------------|------------------------|
| 갱신 | 매 시간 | 매일 02시 |
| 포함 데이터 | impression + click | + **conversion** |
| 주요 용도 | **시간대별 CTR·실시간** | **ROAS·전환율·KPI** |

**한 줄 포인트**: "어떤 테이블에 데이터가 쌓이느냐가 AI가 답할 수 있는 범위를 결정한다"

**시각화 제안**: 2단계를 좌(Stage 1) / 우(Stage 2) 배치. 테이블 비교는 색상 강조로 차이 부각

**발표자 노트**: Athena INSERT 미지원 → Python DataFrame → Parquet → S3 업로드 → MSCK REPAIR TABLE 우회 처리 / DAG 2개 분리 근거 / pendulum UTC→KST 변환

### Slide 14 — AS-IS vs TO-BE: 도입 전후 비교

**헤드라인**: 무엇이 바뀌는가

| 항목 | ❌ AS-IS (현재) | ✅ TO-BE (CAPA 도입 후) |
|------|----------------|------------------------|
| 데이터 조회 방식 | 개발자에게 요청 | Slack에서 직접 자연어로 질문 |
| 소요 시간 | 수 시간 ~ 수 일 | **수 초** |
| 접근 가능 직군 | 개발자·데이터 엔지니어 | **전 구성원** |
| SQL 지식 필요 | 필수 | **불필요** |
| 커뮤니케이션 비용 | 높음 (반복 요청) | 최소화 |
| 의사결정 속도 | 느림 | **실시간** |

**시각화 제안**: 왼쪽 열 회색 배경(AS-IS) + 오른쪽 열 초록 배경(TO-BE). ❌/✅ 아이콘으로 직관적 대비

### Slide 16 — 시행착오 & 아쉬운 점

**헤드라인**: 만들면서 배웠고, 못 만든 것이 다음을 만든다

**시행착오 (핵심 3개, 간결하게)**:

| 문제 | 해결 | 교훈 |
|------|------|------|
| Prophet 신뢰구간이 음수 → 잘못된 이상치 판단 | 실제 데이터 최솟값으로 하한 보정 | 모델보다 데이터 특성 보정이 더 중요 |
| CloudWatch 집계 지연 → 실시간 데이터 누락 | 2분 버퍼 추가 (2, 7, 12... 분마다 실행) | 지연 가능성을 설계 단계부터 반영해야 함 |
| Athena INSERT 미지원 → 파티션 반영 안 됨 | Python DataFrame → Parquet → S3 우회 | Athena 제약은 Python이 중간 레이어로 흡수 |

**아쉬운 점 (2개, 간결하게)**:

- **상대 날짜 미구현**: "지난 주", "이번 달" 같은 표현을 SQL 날짜로 변환하는 로직 미완성
- **Conversion 이상치 탐지 중단**: 타임스탬프 불일치 문제 (컬럼값 vs 실제 timestamp)

**향후 방향 (간결하게)**:

> 상대 날짜 파싱 → 이상치 탐지 고도화 → 입체적 분석(추세·비교) 지원

**시각화 제안**: 시행착오는 3-행 표 (좌: 문제 / 중: 해결 / 우: 교훈). 아쉬운 점은 2-행 텍스트. 상세 내용은 발표자 노트로

**발표자 노트**: 리포트 Redash 연동 미완성, Conversion 이상치 탐지 1시간 단위 전환 검토 중 상세 설명
