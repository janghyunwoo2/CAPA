### Slide 09 — 왜 광고 도메인인가: 구조적 고통과 데이터 이해

**[대본]**

저희는 이런 문제를 해결하기 위해 도메인으로 광고로그를 선정했습니다.

광고 도메인을 선택한 이유는 세 가지입니다.

| 이유 | 설명 |
|------|------|
| 📊 **데이터가 가장 많다** | 클릭 한 번 = 데이터 1건. 하루 30만 건 자동 생성. AI 학습·검증에 최적 |
| 💡 **질문이 즉각적이다** | "이 광고 잘 되고 있어?" — 마케터가 매일 묻는 질문. SQL 요청 병목이 극심 |
| 💸 **돈이 걸려 있다** | 광고비 낭비 1%만 줄여도 수억 원 절감 가능. 실시간 의사결정이 필수 |

여러분도 이런 경험이 있으실 겁니다. 데이터가 필요해서 담당자에게 요청했는데 **5시간 10분을 기다려야 하는 상황**. 데이터는 분명 어딘가에 있는데, 정작 필요한 순간에 꺼낼 수 없는 현실입니다.

광고 도메인에서는 이 문제가 특히 심각합니다. 캠페인 성과를 빠르게 확인하고 즉시 예산을 조정해야 수익을 극대화할 수 있는데, 데이터 요청 → 대기 → 전달 과정에서 **의사결정이 지연**됩니다.

마케터들이 매일 겪는 네 가지 고통이 있습니다:
1. SQL을 몰라 데이터팀에 요청하고 몇 시간을 기다린다
2. 매주 같은 리포트를 수기로 만들며 시간을 낭비한다
3. 로그 이상을 뒤늦게 발견해서 예산을 낭비한다
4. 늘 봐야 하는 지표를 매번 요청하며 리소스를 소모한다

저희 CAPA는 이 네 가지 문제를 각각 해결했습니다.

---

또한 CAPA를 설명하기 전에, 저희가 다루는 광고 데이터가 어떻게 생겼는지 먼저 이해하셔야 합니다.

온라인 광고는 세 가지 이벤트로 구성됩니다.

```
👁  광고 노출 (Impression)     ← 100%
        ↓ CTR 1~5%
🖱  광고 클릭 (Click)           ← 1~5%
        ↓ CVR 1~10%
💰  전환 (Conversion)          ← 0.01~0.5%
    (구매 / 앱 설치 / 회원가입)
```

| 이벤트 | 의미 | 핵심 필드 예시 |
|--------|------|----------------|
| **Impression** | 광고가 화면에 노출됨 | user_id, ad_id, campaign_id, cost_per_impression |
| **Click** | 사용자가 광고를 클릭함 | impression_id, landing_page_url, cost_per_click |
| **Conversion** | 클릭 후 목표 행동 완료 | click_id, conversion_type, conversion_value |

그리고 이 데이터로 계산하는 핵심 KPI가 있습니다.

| 지표 | 계산식 | 비즈니스 의미 |
|------|--------|--------------|
| **CTR** (클릭률) | 클릭수 ÷ 노출수 × 100 | 광고 소재의 매력도 |
| **CVR** (전환율) | 전환수 ÷ 클릭수 × 100 | 랜딩페이지·상품 경쟁력 |
| **CPC** (클릭당 비용) | 총 광고비 ÷ 클릭수 | 트래픽 효율성 |
| **ROAS** (광고 수익률) | 전환 매출 ÷ 광고비 × 100 | 캠페인 수익성 |
| **CPM** (1천 노출당 비용) | 총 광고비 ÷ 노출수 × 1,000 | 브랜딩 비용 효율 |

노출 → 클릭 → 전환, 이 퍼널을 데이터로 추적하는 것이 CAPA의 출발점입니다.

> 📌 **[슬라이드 수치 출처 요약]**
> | 수치 | 출처 | 설명 |
> |------|------|------|
> | CTR 1~5% / CVR 1~10% / 최종 전환 0.01~0.5% | CAPA 프로젝트 자체 시뮬레이션 설계값 | 업계 평균 기반 로그 제너레이터 설정값 (log-generator 서비스 참고) |
> | 하루 30만 건 로그 생성 | CAPA 자체 측정 | EKS 상 log-generator Pod 실측값 |

### Slide 10 — CAPA가 제시하는 해결책: 4가지 핵심 기능

**[대본]**

저희 CAPA는 마케터의 네 가지 고통을 해결하기 위해 네 가지 핵심 기능을 제공합니다.

| 기능 | 해결하는 고통 | 내용 |
|------|-------------|------|
| **Report** | 수기 리포트 반복 제거 | Airflow 기반 일/주/월 리포트 자동 생성·전송 |
| **Dashboard** | 반복 요청 없는 셀프 확인 | SQL 없이도 Redash 대시보드를 원클릭 접근 |
| **Ask** | SQL 지식 없이 즉시 조회 | Slack에서 자연어로 데이터 조회. 의사결정 지연 제거 |
| **Alert** | 이상 발생 즉시 감지 | Impression·Click·Conversion 이상 징후 5분 주기 탐지 + Slack 알림 |

이제 말보다 직접 보여드리는 것이 빠르겠죠. 시연을 통해 설명드리겠습니다.

### Slide 13 — 시연 Scene 1: 리포트 자동화 (Slack 전송)

**[대본]**

### Slide 14 — 시연 Scene 2: Redash 대시보드 딥링크

**[대본]**

두 번째 시연 두 번째 파트는 Redash 대시보드 딥링크 기능입니다.

리포트 하단의 링크를 클릭하면 **SQL 지식 없이도 원클릭으로 Redash 대시보드에 연결**됩니다. 마케터는 복잡한 쿼리 작성 없이도 필요한 시각화 대시보드를 바로 확인할 수 있습니다. CTR, CVR, ROAS 등 핵심 지표를 한눈에 파악하고 즉시 의사결정에 활용할 수 있습니다.

### Slide 12 — 시연 Scene 3: 자연어 → Text-to-SQL

**[대본]**

첫 번째 시연은 가장 핵심 기능인 자연어 질문이 SQL로 변환되는 과정입니다.

슬랙에서 자연어로 질문하면:
1. Vanna AI가 SQL을 생성하고
2. AWS Athena를 통해 실행된 후
3. 결과가 Slack으로 반환됩니다

이 과정이 **과거 반나절이 걸리던 작업을 단 수십 초 만에** 처리합니다. 비개발 직군도 데이터에 즉시 접근할 수 있게 됩니다.

### Slide 15 — 시연 Scene 4: 이상치 알림

**[대본]**

세 번째 시연은 이상치 알림 기능입니다.

Prophet과 Isolation Forest 앙상블 모델을 활용하여 광고 로그의 이상 징후를 탐지합니다. Airflow가 **5분 주기**로 이를 감지하여 Slack으로 즉시 알림을 보냅니다.

마케터는 엔지니어의 개입 없이도 이상 상황에 즉시 대응할 수 있습니다. 예산 낭비를 원천 차단하는 효과입니다.

### Slide 17 — CAPA 전체 시스템 아키텍처

**[대본]**

이제 CAPA 시스템의 전체 구조를 설명드리겠습니다.

**전체 데이터 흐름**:

```
[Log Generator]
광고 Impression / Click / Conversion 로그 생성 (Python/EKS)
        ↓
[Kinesis Data Streams × 3]
imp / clk / cvs 이벤트별 분리 수집
        ↓
[Kinesis Firehose × 3]
JSON → Parquet 자동 변환
        ↓
[S3 Data Lake]
Parquet/ZSTD, 시간 파티셔닝 (year/month/day/hour)
        ↓
[Airflow ETL (Hourly/Daily)]
시간별 요약(ad_combined_log) / 일별 요약(ad_combined_log_summary) 생성
        ↓
[AWS Athena + Glue Catalog]
capa_ad_logs DB 쿼리 엔진
        ↓
[Vanna API] [Slack Bot] [Redash] [Anomaly Detector]
```

**EKS 클러스터 내 AI 레이어**:
- **Vanna API (FastAPI)**: ChromaDB + LLM + RAG 기반 Text-to-SQL
- **Slack Bot**: 자연어 질문 인터페이스
- **Anomaly Detector**: Prophet + Isolation Forest 이상 탐지
- **Airflow**: 5분 주기 이상 탐지 + 일/주/월 리포트 자동화
- **Redash**: Athena 연동 대시보드 + 딥링크

| 레이어 | 기술 | 역할 |
|--------|------|------|
| 수집 | Python + Faker (EKS Pod) | 광고 로그 시뮬레이션 |
| 스트리밍 | Kinesis Data Streams × 3 | 이벤트별 분리 수집 |
| 적재 | Kinesis Firehose × 3 | JSON → Parquet + S3 |
| 저장 | Amazon S3 (Parquet/ZSTD) | 시간 파티셔닝 |
| 처리 | Airflow + Athena + Glue | ETL 및 쿼리 |
| AI | Vanna API + ChromaDB | Text-to-SQL |

### Slide 18 — 데이터 수집 및 저장: Kinesis + S3/Athena 최적화

**[대본]**

이제 데이터가 어떻게 수집되고 저장되는지 상세히 설명드리겠습니다.

저희가 적용한 핵심 최적화 전략은 세 가지입니다.

**① 이벤트별 스트림 분리**

Impression, Click, Conversion은 발생 타이밍과 트래픽 특성이 다릅니다. 이를 **각각 독립적인 Kinesis 스트림으로 분리**하여 각 데이터의 특성에 맞게 처리할 수 있게 했습니다.

**② S3 파티셔닝 + Parquet 형식**

S3에 데이터를 저장할 때 `year/month/day/hour` 구조로 파티셔닝하고 Parquet 형식을 사용합니다. Athena 쿼리 시 **불필요한 데이터 스캔을 줄여 비용을 획기적으로 절감**하는 핵심 전략입니다.

**③ Airflow를 통한 사전 요약 테이블 생성**

Airflow가 매 시간/매일 집계 요약 테이블을 미리 생성해 두어, 대시보드나 LLM이 **가벼운 연산만 수행**하도록 최적화했습니다.

Airflow 구성은 추후 말씀드리겠습니다.

### Slide 19 — Text-to-SQL의 핵심: 11단계 파이프라인

**[대본]**

이제 가장 핵심적인 부분입니다. 자연어가 어떻게 SQL이 되는지 설명드리겠습니다.

일반적인 LLM은 존재하지 않는 테이블명이나 컬럼명을 지어내는 **할루시네이션 문제**가 심각합니다. 저희는 사내 스키마를 학습시키고 RAG를 통해 맥락을 주입하여 이 문제를 해결했습니다.

**파이프라인 5단계 구조**:

| 단계 | 처리 내용 |
|------|-----------|
| ① 입력 처리 | 자연어 질문 수신 + 멀티턴 이전 대화 이력(DynamoDB) 로드 |
| ② RAG 검색 | ChromaDB에서 관련 스키마·샘플 쿼리·비즈니스 용어 검색 |
| ③ SQL 생성 | LLM이 도메인 맥락을 바탕으로 SQL 초안 생성 |
| ④ 검증 | sqlglot 문법 검사 + Athena EXPLAIN 실행 계획 확인 + 셀프 코렉션 |
| ⑤ 실행 및 반환 | Athena 실행 → 결과 반환 → 대화 이력 저장 |

**멀티턴 대화 지원**: DynamoDB에 이전 대화 이력을 저장하여 "이전 질문 기준으로 이번 달은?" 같은 연속 질문이 가능합니다.

### Slide 22 — 대시보드 구축: Why Redash?

**[대본]**

마케터가 SQL을 알지 못하더라도 직접 활용할 수 있도록 대시보드를 구축했습니다.
저희가 Redash를 선택한 이유는 네 가지입니다.

1. Athena에서의 쿼리를 Redash 에서 그대로 사용합니다. 데이터소스를 네이티브로 지원**
2. Redash 날짜선택은 쿼리 파라미터 형태로 jinja2 문법으로조회 가능
3. 모든 쿼리를 Redash에서 관리하여 쿼리 결과를 한곳에서 관리
4. 사용자 계정을 제공함으로써, 원하는 형태의 대시보드를 생성하고,
SQL 언어를 아는 사용자라면 쿼리를 수정해 자연어로 조회가 되지 않은 데이터를 직접 조회할 수 있습니다.

덕분에 마케터는 편리하게 달력만 클릭하고, 시스템은 파티션 프루닝을 통해 **성능과 비용을 동시에** 잡을 수 있었습니다.

### Slide 27 — AS-IS vs TO-BE: CAPA 도입 전후 비교

**[헤드라인]** 내부사용자 누구나 데이터에 접근할 수 있는 환경 — 데이터 조회 시간 **99% 단축**

| 항목 | AS-IS (도입 전) | TO-BE (CAPA 도입 후) |
|------|---------------|---------------------|
| **데이터 조회** | 요청 → 최소 1시간~3일 대기 | 자연어 질문 → 수십 초 내 즉시 응답 |
| **담당자 개입** | 데이터팀 개입 필수 | 마케터 스스로 직접 조회 |

| **이상 탐지** | 사람이 직접 확인 | AI 모델 실시간 감지 + 즉시 알림 |
| **총 소요 시간** | 수 시간~수 일 | 수십 초 |

### Slide 28 — 시행착오 & 교훈 (Lessons Learned)

**[대본]**

저희가 개발하며 겪은 세 가지 핵심 시행착오를 공유합니다.
#### AI 엔진

**① EKS 노드 그룹 분리 실패 → Karpenter로 해결**

초기에 Airflow 워커와 API 서버를 같은 노드에 두었습니다. 결과적으로 Airflow가 메모리를 과점유하면 API 서버가 다운되는 문제가 발생했습니다.

**해결**: Karpenter를 도입하여 작업 특성별로 노드를 분리했습니다.

**② LLM의 "그럴싸한 거짓말" — 비즈니스 로직 오류**

LLM이 문법적으로는 완벽하지만 비즈니스 로직이 틀린 쿼리를 생성하는 문제가 있었습니다. 예를 들어, ROAS 계산식을 잘못 적용하는 경우입니다.

**해결**: 사내 공식 지표 계산식을 프롬프트에 명시적으로 정의하고, 4중 검증 로직을 추가했습니다.

**③ Reranker 도입 후 롤백**

jina-reranker-v2를 도입하여 정확도를 2% 높였지만, **응답 지연이 3초 이상** 발생해 사용자 경험을 해쳤습니다.

**해결**: 과감히 Reranker를 제거하고 ChromaDB의 메타데이터 필터링을 고도화하는 방향으로 선회했습니다.

> ⚠️ 참고: 이후 2026-03-24 작업에서 ChromaDB distances를 reranker에 정확히 전달하는 버그 수정으로 재도입을 검토한 사례가 있습니다. 발표 시점 기준으로 최신 상태를 확인하여 언급할 것.

### Slide 29 — 발전 방향

**[대본]**

저희는 데이터 조회와 이상 탐지 알림을 넘어, **데이터 기반의 즉각적인 행동과 최적화**를 목표로 합니다.

**① 역질문 봇**

모호한 질문에 대해 AI가 스스로 되물어 SQL 정확도를 높입니다.  
예: "지난주"가 월~일인지, 7일 기준인지 명확히 확인하는 방식

**② Slack 액션**

예: 보고서 하단에 "효율 저하 캠페인 즉시 Off" 버튼 → 관리자 콘솔 없이 광고 송출 중단

**③ 정량 평가 시스템 고도화**

Spider 벤치마크 기반으로 정확도를 지속적으로 측정하고, RAG 개선 루프를 완성합니다.

결국, **데이터를 조회하는 것을 넘어 데이터로 즉시 행동하는 시스템**을 만드는 것이 저희의 비전입니다.
