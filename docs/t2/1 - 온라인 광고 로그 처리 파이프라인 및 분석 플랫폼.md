# 1 - 온라인 광고 로그 처리 파이프라인 및 분석 플랫폼

안녕하세요. 오늘 저희 CAPA 팀은 온라인 광고 로그 처리 파이프라인과 분석 플랫폼에 대해 말씀드리겠습니다. 저희는 비개발자도 실시간으로 데이터를 분석하고 이상 징후를 탐지할 수 있는 자동화 시스템을 만들었습니다. 이 시스템은 광고 로그 데이터를 클라우드 기반 AI 파이프라인으로 처리해서, 누구나 쉽게 인사이트를 얻도록 돕습니다. 저희 팀은 장현우, 김시현, 김병훈 세 명으로 구성되어 있습니다. 다음 슬라이드에서 저희 팀의 역할 분담에 대해 더 자세히 설명드리겠습니다.

# 2 - 팀 구성 및 역할 분담 (R&R)

저희 CAPA 팀은 각자의 전문성을 결합해서 데이터 분석의 물리적, 기술적 한계를 극복하고자 했습니다. 장현우 님은 인프라와 AI 엔진을 담당하며 AWS 클라우드 아키텍처 설계와 Text-to-SQL 엔진 구축을 맡았습니다. 김시현 님은 데이터 수집과 시각화 분야에서 실시간 데이터 파이프라인 구현과 Redash 대시보드 설계를 담당했습니다. 마지막으로 김병훈 님은 자동화와 이상 탐지를 맡아 Airflow 기반 리포트 자동화와 Prophet 모델을 활용한 이상치 탐지 시스템을 개발했습니다. 이렇게 각자의 강점을 살려 시너지를 냈습니다. 다음으로는 저희가 해결하고자 했던 문제의 핵심, 바로 '기다림'에 대해 이야기해보겠습니다.

# 3 - 누구나 공감하는 '기다림'

여러분도 이런 경험 있으실 겁니다. 데이터가 필요해서 담당자에게 요청했는데, 5시간 10분이나 기다려야 하는 상황이죠. 이 대화는 저희가 흔히 겪는 답답함을 보여줍니다. 데이터는 분명 어딘가에 존재하지만, 정작 필요한 순간에 바로 꺼내 쓸 수 없는 현실입니다. 이런 비효율은 비즈니스 의사결정을 지연시키고, 결국 기회비용으로 이어집니다. 왜 이런 '기다림'이 구조적으로 발생할까요? 다음 슬라이드에서 그 원인을 살펴보겠습니다.

# 4 - 왜 이 문제가 구조적으로 생기나?

이러한 기다림은 두 가지 구조적인 병목 현상 때문에 발생합니다. 첫째는 기술 장벽입니다. 데이터를 직접 보려면 SQL을 알아야 하는데, 이 장벽 때문에 기업 데이터의 68%가 비개발 직군에게 닿지 못하고 방치됩니다. 둘째는 분석팀 병목입니다. 모든 데이터 요청이 소수의 분석팀에 집중되면서, 끝없는 요청과 한정된 분석팀 리소스 사이에서 병목이 생기는 거죠. 이 두 가지 병목은 의사결정 비용으로 직결되는데, 특히 광고 도메인에서는 그 영향이 훨씬 큽니다. 왜 광고 도메인에서 이 문제가 더 심각한지 다음 슬라이드에서 설명드리겠습니다.

# 5 - 왜 하필 광고 도메인인가?

광고 도메인에서는 데이터 지연이 곧 예산 낭비로 이어집니다. 캠페인 성과를 빠르게 확인하고 즉시 예산을 조정해야 수익을 극대화할 수 있기 때문입니다. 하지만 데이터는 있는데 꺼낼 수 없어서 요청하고, 대기하고, 전달받는 과정에서 의사결정이 지연됩니다. 마케터들이 매일 겪는 네 가지 고통이 있습니다. SQL을 몰라 데이터팀에 요청하고 몇 시간을 기다리거나, 매주 같은 리포트를 수기로 만들며 시간을 낭비합니다. 또, 로그 이상을 뒤늦게 발견해서 예산을 낭비하기도 하고, 늘 봐야 하는 지표를 매번 요청하며 리소스를 소모합니다. 저희 CAPA는 이 네 가지 문제를 각각 해결했습니다.

# 6 - CAPA가 제시하는 해결책 (4가지 핵심 기능)

이전 슬라이드에서 마케터가 겪는 네 가지 어려움을 말씀드렸습니다. CAPA는 이 문제들을 해결하기 위해 네 가지 핵심 기능을 제공합니다. 첫째, Ask 기능으로 SQL 지식 없이도 슬랙에서 자연어로 데이터를 즉시 조회할 수 있습니다. 이는 의사결정 지연을 없애줍니다. 둘째, Report 기능은 Airflow를 통해 리포트를 자동으로 생성하고 전송하여 수기 엑셀 작업의 반복을 제거합니다. 셋째, Alert 기능은 Impression, Click, Conversion 건수의 실시간 이상 감지를 통해 예산 낭비를 막아줍니다. 마지막으로 Dashboard 기능은 SQL 없이도 Redash 대시보드를 원클릭으로 접근하게 하여 반복적인 요청 없이 언제든 셀프 확인을 가능하게 합니다. 이제 말보다 직접 보시는 것이 빠르겠죠? 다음 슬라이드에서 첫 번째 시연을 보여드리겠습니다.

# 7 - 시연 Scene 1 — 자연어 → text-to-sql

CAPA의 첫 번째 시연은 자연어 질문이 SQL로 변환되는 과정입니다. 보시는 것처럼 슬랙에서 자연어로 질문하면 Vanna AI가 SQL을 생성하고, Athena를 통해 실행된 후 결과가 반환됩니다. 이 과정은 과거 반나절이 걸리던 작업을 단 수십 초 만에 처리합니다. 이는 비개발 직군도 데이터에 즉시 접근할 수 있게 하여 업무 효율을 극대화합니다. 다음 시연에서는 이렇게 얻은 데이터를 어떻게 시각화하는지 보여드리겠습니다.

# 8 - 시연 Scene 2 — 리포트 & Redash 대시보드 딥링크

두 번째 시연은 리포트와 Redash 대시보드 딥링크 기능입니다. Airflow가 자동으로 리포트를 생성하고 슬랙으로 전송합니다. 여기서 중요한 점은 SQL 지식 없이도 원클릭으로 Redash 대시보드에 연결된다는 것입니다. 마케터는 복잡한 쿼리 작성 없이도 필요한 시각화 대시보드를 바로 확인할 수 있습니다. 이는 데이터 접근성을 혁신적으로 개선합니다. 다음으로는 이상치 알림 기능을 살펴보겠습니다.

# 9 - 시연 Scene 3 — 이상치 알림

세 번째 시연은 이상치 알림 기능입니다. Prophet과 Isolation Forest 모델을 활용하여 광고 로그의 이상 징후를 탐지하고, Airflow가 5분 주기로 이를 감지하여 슬랙으로 즉시 알림을 보냅니다. 이 시스템은 마케터가 실시간으로 광고 성과를 모니터링하고, 엔지니어의 개입 없이도 이상 상황에 즉시 대응할 수 있게 합니다. 이는 예산 낭비를 효과적으로 방지합니다. 이제 CAPA 시스템의 전체적인 아키텍처를 설명드리겠습니다.

# 10 - CAPA 시스템 아키텍처 및 광고 데이터 흐름

지금 보시는 것은 CAPA 시스템의 전체 아키텍처와 광고 데이터 흐름입니다. Log Generator에서 생성된 Impression, Click, Conversion 로그는 Kinesis Data Streams를 거쳐 S3 데이터 레이크에 Parquet 형식으로 저장됩니다. 이후 AWS Glue Catalog와 Athena를 통해 데이터가 처리되고, EKS 클러스터 내 AI 레이어에서 Vanna AI, Airflow, Anomaly Detector가 작동합니다. 특히 Slack Bot과 Vanna AI, Redash API, Athena가 연동되어 쿼리 조회 흐름을 만들고, Airflow는 Anomaly Detector와 Slack을 통해 5분 단위로 이상 탐지 알림을 보냅니다. 또한 Airflow는 Athena와 Report Generator를 활용해 주기적인 리포트 자동화를 담당합니다. 다음 슬라이드에서는 데이터 수집 및 저장 단계의 최적화 방안을 자세히 설명드리겠습니다.

# 11 - 데이터 수집 및 저장 (Kinesis + S3/Athena)

이전 슬라이드에서 전체 아키텍처를 보셨으니, 이제 데이터가 어떻게 수집되고 저장되는지 자세히 살펴보겠습니다. 저희는 Kinesis와 S3, Athena를 활용해 비용 효율적이면서도 성능 좋은 데이터 파이프라인을 구축했습니다. 트래픽과 타이밍이 다른 임프레션, 클릭, 전환 로그를 각각 독립적인 Kinesis 스트림으로 분리했습니다. 이렇게 하면 각 데이터의 특성에 맞춰 유연하게 처리할 수 있죠. 또한, S3에 데이터를 저장할 때는 파티셔닝과 Parquet 형식을 적극 활용했습니다. 이는 Athena 쿼리 시 불필요한 데이터 스캔을 줄여 비용을 획기적으로 절감하는 핵심 전략입니다. 마지막으로, Airflow를 통해 미리 요약 테이블을 만들어 대시보드나 LLM이 가벼운 연산만 수행하도록 했습니다. 이 모든 과정은 다음 슬라이드에서 설명할 Text-to-SQL 파이프라인의 기반이 됩니다.

# 12 - 어떻게 자연어가 SQL이 되는가 — 11단계 Text-to-SQL 파이프라인

이제 가장 핵심적인 부분, 즉 자연어가 어떻게 SQL로 변환되는지 11단계 파이프라인을 통해 설명드리겠습니다. 일반적인 LLM은 없는 테이블이나 컬럼명을 지어내는 할루시네이션 문제가 심각합니다. 하지만 저희 CAPA는 사내 스키마를 학습시키고 RAG를 통해 맥락을 주입하여 이 문제를 해결했습니다. 이 파이프라인은 크게 입력 처리, RAG 검색, SQL 생성 및 검증, 실행 및 분석, 그리고 기록의 5단계로 나뉩니다. 특히 멀티턴 대화를 지원하기 위해 DynamoDB에 이전 대화 이력을 저장하고, 비동기 방식으로 쿼리를 처리하여 사용자 경험을 최적화했습니다. 이 복잡한 과정을 통해 우리는 할루시네이션 없이 정확한 SQL을 생성할 수 있었습니다. 다음 슬라이드에서는 이 정확도를 높이기 위해 어떤 고민들을 했는지 더 자세히 말씀드리겠습니다.

# 13 - 정확도를 높이기 위해 우리가 고민한 것들

앞서 설명드린 11단계 파이프라인에서 SQL 정확도를 끌어올리기 위해 저희는 세 가지 핵심 전략을 사용했습니다. 첫째, 4중 검증과 가드레일입니다. sqlglot을 이용해 SELECT 외의 명령을 차단하고, Athena EXPLAIN으로 실행 전 쿼리 플랜을 분석해 문법 오류나 풀스캔을 막았습니다. 또한, 결과 행수를 1,000개로 제한하고, 검증 실패 시 LLM이 스스로 수정하도록 하는 셀프 코렉션 루프를 도입했습니다. 둘째, LLM의 사고 절차를 설계하는 프롬프트 엔진입니다. CoT 방식을 적용해 DDL 확인부터 집계 설계까지 6단계 사고 절차를 강제하고, 프롬프트 인젝션을 방지했습니다. 셋째, Athena 도메인 지식을 주입하는 지능형 시딩입니다. Athena 문법 최적화, 비즈니스 로직 규격화, Few-shot 기반 패턴 학습, 그리고 오답 패턴 사전 보정 등을 통해 LLM이 실제 환경에 맞는 SQL을 생성하도록 유도했습니다. 이러한 노력들이 모여 다음 슬라이드에서 보실 놀라운 정확도 향상을 이끌어냈습니다.

# 14 - 0%에서 91%로 — 4라운드 개선 스토리

이러한 노력의 결과로, 저희는 SQL 생성 정확도를 0%에서 91%까지 끌어올릴 수 있었습니다. 단순한 튜닝이 아니라, 매 라운드마다 '왜 틀렸는가'를 깊이 파고들며 개선을 거듭했습니다. 라운드 1에서는 DDL 전달 버그로 인해 정확도가 0%였지만, 라운드 2에서는 RAG 도입과 리랭커 교체로 35%까지 올랐습니다. 라운드 3에서는 CoT와 가드레일, 그리고 시딩 재설계를 통해 68%를 달성하며 목표에 근접했습니다. 그리고 라운드 4에서는 코사인 지표와 DDL 역추적 안정화를 통해 최종적으로 91%의 실행 정확도를 달성했습니다. 이는 실무에 바로 투입할 수 있는 수준의 정확도입니다. 다음 슬라이드에서는 이렇게 생성된 SQL을 활용해 대시보드를 구축하는 과정에 대해 말씀드리겠습니다.

# 15 - 대시보드 구축 — Why Redash?

이제 이렇게 생성된 SQL을 마케터가 직접 활용할 수 있도록 대시보드를 구축하는 방법에 대해 설명드리겠습니다. 저희는 Redash를 선택했습니다. Redash는 Athena 데이터소스를 네이티브로 지원하고, CTR, CVR, ROAS 같은 핵심 지표를 한눈에 볼 수 있는 친화적인 UI를 제공하기 때문입니다. 하지만 여기서 한 가지 딜레마가 있었습니다. 마케터는 달력 UI로 날짜를 선택하고 싶어 하지만, Athena는 파티셔닝을 위해 year/month/day 구조를 요구했죠. 이 충돌을 해결하기 위해 저희는 CTE와 date_parse() 함수를 활용했습니다. Redash 쿼리 안에서 마케터가 선택한 날짜 파라미터를 받아, 백엔드에서 자동으로 파티션 컬럼으로 변환하도록 했습니다. 덕분에 마케터는 편리하게 달력만 클릭하고, 시스템은 파티션 프루닝을 통해 성능과 비용을 동시에 잡을 수 있었습니다. 다음 슬라이드에서는 리포트 자동화에 대해 말씀드리겠습니다.

# 16 - 일일/주간/월간 리포트 자동화 (Airflow K8s 기반)

우리는 앞서 마케터들이 매일 겪는 4가지 고통을 살펴봤습니다. 그중 하나가 바로 매주 같은 리포트를 수기로 만들며 시간을 낭비하는 것이었죠. CAPA는 Airflow K8s 기반으로 일일, 주간, 월간 리포트 생성을 완전히 자동화했습니다. 이제 마케터는 매일 아침 전날 데이터를 엑셀로 수기 결산하는 대신, Airflow가 자동으로 생성해 Slack으로 보내주는 리포트를 받게 됩니다. 특히, KubernetesPodOperator를 활용해 각 보고서 생성 작업을 독립된 파드로 실행함으로써 라이브러리 충돌을 막고 리소스를 효율적으로 사용합니다. 이 자동화된 리포트에는 핵심 지표 요약과 캠페인별 성과가 한눈에 들어오도록 구성되어 있습니다. 다음으로는 광고 이상 탐지 시스템에 대해 설명드리겠습니다.

# 17 - 실시간 광고 이상 탐지 및 Slack 알림

리포트 자동화와 더불어, 광고 예산 낭비를 막기 위한 실시간 이상 탐지 시스템도 중요합니다. CAPA는 Prophet과 Isolation Forest 앙상블 모델을 활용해 광고 로그의 이상 징후를 놓치지 않고 포착합니다. Prophet은 시계열 데이터의 트렌드와 계절성을 학습해 정상 범위를 예측하고, Isolation Forest는 단기적인 급변 값을 빠르게 찾아냅니다. 이 두 모델의 장점을 결합해 복합적인 이상 징후를 누락 없이 감지하는 거죠. Airflow가 5분마다 최신 로그를 분석하고, 이상 징후가 발견되면 즉시 Slack으로 알림을 보내 마케터가 빠르게 대응할 수 있도록 돕습니다. 이는 설정 오류나 비정상 트래픽으로 인한 광고비 소진을 원천적으로 차단하는 효과를 가져옵니다. 이제 이 모든 시스템을 뒷받침하는 인프라에 대해 말씀드리겠습니다.

# 18 - 인프라 코드화 + 트래픽 대응 (Terraform + EKS + Karpenter)

이러한 자동화된 시스템을 안정적으로 운영하려면 강력한 인프라가 필수적입니다. 우리는 Terraform, EKS, 그리고 Karpenter를 활용해 인프라를 코드화하고 트래픽 변화에 유연하게 대응합니다. Terraform을 통해 인프라 전체를 코드로 관리하며, 단 25분 만에 전체 인프라와 애플리케이션을 배포할 수 있습니다. 이는 환경 재현성을 100% 보장하죠. Amazon EKS는 안정적인 컨테이너 환경을 제공하며, 마이크로서비스 아키텍처 기반의 유연한 확장성을 가능하게 합니다. 특히 Karpenter는 트래픽 스파이크 발생 시 기존 스케일러보다 훨씬 빠르게 노드를 프로비저닝하여 초고속 스케일링을 지원합니다. 또한, Karpenter를 통해 저렴한 Spot 인스턴스를 적극 활용하여 인프라 비용을 최대 70%까지 절감할 수 있었습니다. 그럼, CAPA 도입 전후의 변화를 비교해볼까요?

# 19 - AS-IS vs TO-BE — 도입 전후 비교

지금까지 CAPA의 주요 기능과 기술 스택을 설명드렸습니다. 이제 CAPA 도입 전과 후의 변화를 비교하며, 이 시스템이 가져온 혁신적인 효과를 보여드리겠습니다. 과거에는 데이터 요청부터 응답까지 최소 1시간에서 3일까지 걸렸지만, CAPA 도입 후에는 수십 초 내에 즉시 답변을 받을 수 있습니다. 데이터팀의 개입 없이 마케터 스스로 데이터를 조회하고 분석할 수 있게 된 거죠. 정기 리포트 작성도 수기 엑셀 작업에서 Airflow 기반의 완전 자동화로 바뀌어 반복적인 리소스 낭비를 없앴습니다. 이상 탐지 역시 사람이 직접 확인하던 방식에서 AI 모델이 실시간으로 감지하고 즉시 알림을 보내는 방식으로 전환되어 예산 낭비를 막을 수 있게 되었습니다. 결과적으로 데이터 추출 업무 소요 시간을 약 99% 단축했습니다. 마케터는 이제 오직 마케팅 본질에만 집중할 수 있게 된 것입니다. 다음으로는 이 모든 과정에서 겪었던 시행착오와 교훈에 대해 이야기하겠습니다.

# 20 - 시행착오 & 아쉬운 점 (Lessons Learned)

저희는 CAPA를 개발하며 여러 시행착오를 겪었습니다. 특히 EKS 노드 그룹 분리 실패, LLM의 비즈니스 로직 오류, 그리고 Reranker 도입 후 롤백이 주요 경험입니다. 초기에는 Airflow 워커와 API 서버를 같은 노드에 두어 API 서버가 메모리 부족으로 다운되는 문제가 있었습니다. Karpenter를 도입해 작업 특성별로 노드를 분리하여 해결했습니다. 또한, LLM이 문법적으로는 완벽하지만 비즈니스 로직이 틀린 쿼리를 생성하는 '그럴싸한 거짓말' 문제도 있었습니다. 사내 공식 지표 계산식을 프롬프트에 하드코딩하고 4중 검증 로직을 추가하여 해결했습니다. 마지막으로, Reranker를 도입하여 정확도를 2% 높였지만, 응답 지연이 3초 이상 발생해 사용자 경험을 해쳤습니다. 과감히 Reranker를 제거하고 ChromaDB의 메타데이터 필터링을 고도화하는 방향으로 선회했습니다. 이러한 경험을 통해 얻은 교훈을 바탕으로, 다음 슬라이드에서는 CAPA의 발전 방향을 말씀드리겠습니다.

# 21 - 발전방향

이제 CAPA의 미래 발전 방향에 대해 말씀드리겠습니다. 저희는 데이터 조회와 이상 탐지 알림을 넘어, 데이터 기반의 즉각적인 행동과 최적화를 목표로 합니다. 첫째, '역질문 봇'을 개발하여 모호한 질문에 대해 AI가 스스로 되물어 SQL 정확도를 높일 것입니다. 예를 들어, '지난주'가 월요일부터 일요일인지, 아니면 7일 기준인지 명확히 확인하는 식입니다. 둘째, '슬랙 액션' 기능을 통해 데이터 조회 후 즉시 행동할 수 있도록 할 것입니다. 보고서 하단에 '효율 저하 캠페인 즉시 Off' 버튼을 추가하여 관리자 콘솔 없이도 광고 송출을 중단할 수 있게 됩니다. 마지막으로, '정량 평가' 시스템을 고도화하여 LLM 응답의 정확도와 유용성을 지속적으로 측정하고 개선하는 평가 파이프라인을 구축할 것입니다. 이는 Spider 벤치마크를 기반으로 정확도를 정량적으로 측정하여 RAG 개선 루프를 완성하는 것입니다. 결국, 데이터를 조회하는 것을 넘어 데이터로 즉시 행동하는 시스템을 만드는 것이 저희의 비전입니다. 다음은 Q&A 시간입니다.
