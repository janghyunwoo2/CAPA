## 🔍 문제 해결 중심 정리

### 📅 3/9 (리포트 구조 개선)

#### 🔹 문제 상황
- Redash, T2S 등 기존 시스템과 기능이 중복됨
- 유지보수 비용 증가가 예상됨

#### 🔹 해결
- 불필요한 기능을 과감히 제거
  - AI 분석 기능 제거
  - 온디맨드 요청 방식(필요할때마다 요청) 제거 (온디맨드 -> Scheduled)
- 정기 리포트 중심 + 핵심 지표만 제공하도록 단순화

#### 🔹 결과
- 전체 구조가 단순해짐
- 시스템 역할이 명확해짐

> 기능 추가보다, 시스템 간 역할을 명확히 분리하는 것이 유지보수 측면에서 더 중요하다는 것을 경험

### 📅 3/16 (이상치 탐지 시스템)

#### 🔹 문제 상황
- Prophet 모델의 특성상 예측 신뢰구간이 음수로 내려가는 문제가 발생
  → 실제 로그 데이터는 음수가 될 수 없어, 잘못된 이상치 판단 가능성 존재
- 실시간 처리 시 CloudWatch의 데이터 집계가 지연되어,
  수집 시점에 따라 데이터 누락이 발생할 수 있음

#### 🔹 해결
- 신뢰구간 하한이 음수가 되는 경우,
  실제 데이터의 최소값(2월 기준)을 기준으로 보정하여 비현실적인 예측 제거
- CloudWatch의 집계 지연을 고려하여,
  즉시 조회 대신 2분의 버퍼를 두고 데이터를 수집하도록 설계

#### 🔹 결과
- 데이터 특성을 반영한 안정적인 이상치 탐지 가능
- 실시간 처리 환경에서도 데이터 누락 없이 안정적으로 동작
- 이상치 발생 시 Slack 알림 자동화

> 모델 성능보다, 실제 데이터 특성에 맞게 보정하는 과정이 더 중요하다는 것을 경험
