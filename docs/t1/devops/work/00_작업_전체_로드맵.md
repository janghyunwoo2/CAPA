# CAPA E2E 파이프라인 구축 로드맵

> **목적**: E2E 파이프라인 연결 확인 (기능 구현 X, 길 뚫기 O)  
> **참조**: [devops_implementation_guide.md](../devops_implementation_guide.md)  
> **최종 목표**: Kinesis → S3 → Athena → EKS (Sl ack Bot) 전체 데이터 흐름 확인

---

## 작업 방식

```
작업.md 생성 (규모/역할별)
     ├─► 실행 → 결과.md 저장
     ├─► 불만족 시 → 작업.md 수정 후 재실행
     └─► ✅ 만족 → 다음 작업.md로 이동
```

## 전체 작업 단계

### Phase 0: 사전 준비 (필수)
- [ ] **00-1**: 로컬 개발 환경 설정
  - AWS CLI, Terraform, kubectl 설치 확인
  - AWS 자격 증명 설정
  - 파일: `01_로컬환경_설정.md`

- [ ] **00-2**: Git 저장소 및 폴더 구조 설정
  - `infrastructure/` 구조 확인
  - `.gitignore` 설정 (*.tfstate, .env 등)
  - 파일: `02_저장소_구조_설정.md`

---

### Phase 1: Terraform Base Layer (AWS 리소스)
**목표**: AWS 기본 인프라 구축 (Kinesis, S3, IAM, EKS, CloudWatch, SNS)

- [ ] **01-1**: Terraform Backend 설정
  - S3 bucket (terraform state 저장)
  - DynamoDB table (state locking)
  - 파일: `03_terraform_backend.md`

- [ ] **01-2**: IAM Role 생성 (IRSA 준비)
  - EKS 클러스터 Role
  - Firehose Role
  - Airflow/Bot/Redash/Vanna ServiceAccount Role
  - CloudWatch Alarms Role
  - 파일: `04_iam_roles.md`

- [ ] **01-3**: Kinesis + S3 + Glue
  - Kinesis Data Stream
  - Kinesis Firehose → S3 (Parquet)
  - Glue Catalog (메타데이터)
  - 파일: `05_data_pipeline_기본.md`

- [ ] **01-4**: EKS 클러스터 구축
  - EKS Cluster (1.30)
  - Node Group (t3.medium × 2, **AL2023**)
  - OIDC Provider (IRSA 전제조건)
  - EBS CSI Driver Addon
  - 파일: `06_eks_cluster.md`

- [ ] **01-5**: CloudWatch + SNS (Alert 시스템)
  - CloudWatch Alarms (Kinesis, EKS Metrics)
  - SNS Topic (Alert 알림)
  - 파일: `07_alert_system.md`

---

### Phase 2: E2E 연결 테스트 (MVP)
**목표**: 데이터가 Kinesis부터 Athena까지 흐르는지 확인

- [ ] **02-1**: Log Generator 배포 (로컬 or EC2)
  - JSON 로그 생성 → Kinesis Stream 전송
  - 샘플 광고 로그 포맷
  - 파일: `08_log_generator.md`

- [ ] **02-2**: S3 → Glue → Athena 검증
  - S3에 Parquet 파일 생성 확인
  - Athena SELECT 쿼리 실행
  - 데이터 흐름 검증
  - 파일: `09_athena_데이터_검증.md`

---

### Phase 3: EKS Apps Layer (Helm 배포)
**목표**: Airflow, Redash, Vanna AI, Slack Bot 배포

- [ ] **03-1**: Helm Values 준비
  - airflow.yaml
  - redash.yaml
  - vanna.yaml
  - chromadb.yaml
  - slack-bot.yaml (echo 버전)
  - 파일: `10_helm_values_준비.md`

- [ ] **03-2**: Airflow 배포
  - Helm Chart 배포
  - IRSA 연동 확인
  - Webserver 접속 확인
  - 파일: `11_airflow_배포.md`

- [ ] **03-3**: Redash 배포
  - Helm Chart 배포
  - Athena 데이터 소스 연결
  - KPI 대시보드 생성
  - 파일: `12_redash_배포.md`

- [ ] **03-4**: Report Generator 배포
  - 기본 컨테이너 이미지 빌드 (Hello World API)
  - ECR 푸시
  - Helm으로 EKS 배포
  - Health Check API 확인 (/health)
  - 파일: `13_report_generator_배포.md`

- [ ] **03-5**: Vanna AI + ChromaDB 배포
  - Vanna API 컨테이너 빌드
  - ChromaDB Helm 배포
  - Vanna AI Helm 배포
  - Health Check 확인
  - 파일: `14_vanna_ai_배포.md`

- [ ] **03-6**: Slack Bot Echo 배포
  - 컨테이너 이미지 빌드 (Dockerfile)
  - ECR 푸시
  - Helm으로 EKS 배포
  - Slack에서 "@capa-bot echo test" 확인
  - 파일: `15_slack_bot_echo.md`

- [ ] **04-2**: 모니터링 설정 (기본)

| Phase | 작업 수 | 예상 소요 | 핵심 검증 |

| **합계** | **17개** | **~3.5시간** | **파이프라인 길 뚫기 + Dashboard + Alert 완료** |

## 검증 기준 (각 작업마다)

### 필수 포함 항목
1. **목표**: 이 작업이 달성해야 하는 것
2. **실행 명령어**: Terraform, kubectl 등 구체적 명령어
3. **검증 방법**: 성공 여부를 확인하는 방법
4. **실패 시 대응**: 일반적인 오류 및 해결 방법

### 결과.md 포함 항목
1. **실행 로그**: 명령어 출력 결과 (스크린샷 or 텍스트)
2. **검증 결과**: OK/FAIL
3. **문제점 및 해결**: 발생한 이슈와 조치 사항
4. **다음 단계**: 다음 작업으로 넘어가기 전 체크사항
