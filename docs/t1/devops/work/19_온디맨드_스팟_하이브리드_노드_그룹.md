### 목표
- **온디맨드 노드에 핵심 서비스 집중 배치**: "망가지면 짜증나는 것들"
- **스팟 노드에 일꾼 파드 분리 배치**: "회수되어도 재시작하면 그만인 것들"
- **현재 월 $136 대비 비용 절감** (온디맨드 단독 대비 예상 40~50% 절감)

### 최종 구성 목표

```
┌──────────────────────────────────────────────────────┐
│    [온디맨드 노드 그룹] t3a.large x 1~2대              │
│    - airflow-webserver, airflow-scheduler            │
│    - airflow-postgresql, redash-postgresql           │
│    - redash-server (메인), redash-redis              │
│    - kube-system 핵심 파드들                          │
└──────────────────────────────────────────────────────┘
┌──────────────────────────────────────────────────────┐
│    [스팟 노드 그룹] t3a.large, t3.large, m5a.large     │
│    (다양한 인스턴스 타입으로 회수 리스크 분산)            │
│    - airflow-triggerer (배치 작업용)                  │
│    - redash-adhocworker                              │
│    - redash-genericworker                            │
│    - redash-scheduledworker                          │
│    - vanna-api, slack-bot, report-generator          │
└──────────────────────────────────────────────────────┘
```

### 노드 배치 메커니즘 (Taints & Tolerations)

| 개념 | 역할 |
| :--- | :--- |
| **Taint** | 노드에 붙이는 '출입 제한' 꼬리표. "스팟 허가증 없는 파드는 들어오지 마라" |
| **Toleration** | 파드가 갖는 '출입 허가증'. "나는 스팟 노드에서도 죽을 각오가 됐다" |
| **NodeAffinity** | 파드가 "나는 꼭 온디맨드 노드에 있어야 한다" 선호도를 표현하는 방식 |

## 📝 작업 단계

### Step 1. 테라폼 노드 그룹 분리 (`06-eks.tf`)
기존의 단일 `aws_eks_node_group.main`을 유지하면서, 스팟 전용 노드 그룹을 새로 추가합니다.

**수정 파일**: `infrastructure/terraform/06-eks.tf`
```hcl
# [추가] 스팟 노드 그룹
resource "aws_eks_node_group" "spot" {
  cluster_name    = aws_eks_cluster.main.name
  node_group_name = "${var.project_name}-spot-group"
  node_role_arn   = aws_iam_role.eks_node.arn
  subnet_ids      = [sort(data.aws_subnets.default.ids)[0]]

  # 여러 인스턴스 타입 = 회수 리스크 분산
  instance_types = ["t3a.large", "t3.large", "m5a.large"]
  capacity_type  = "SPOT"

  scaling_config {
    desired_size = 1
    min_size     = 0   # 부하 없을 때 0대까지 축소 허용
    max_size     = 3
  }

  taint { # 스팟 허가증 없는 파드는 여기 못 들어옴
    key    = "node-type"
    value  = "spot"
    effect = "NO_SCHEDULE"
  }

  tags = {
    Name = "${var.project_name}-spot-group"
    "k8s.io/cluster-autoscaler/enabled"                     = "true"
    "k8s.io/cluster-autoscaler/${var.project_name}-cluster" = "owned"
  }
}
```

### Step 2. 온디맨드 노드 수 조정 (`06-eks.tf`)
핵심 파드만 온디맨드에 배치하므로 노드 수를 2대 → **1~2대**로 줄여 비용을 절감합니다.

```hcl
# 기존 main 노드 그룹 수정
scaling_config {
  desired_size = 2  # 실제 적용 후 상황 보고 1로 줄일지 검토
  min_size     = 1
  max_size     = 2
}
```

### Step 3. 파드별 스팟 Toleration 및 Affinity 적용

스팟 노드에 배치할 파드들에 아래 설정을 추가합니다.

**수정 파일**: `infrastructure/terraform/11-k8s-apps.tf` (Slack Bot, Report Generator, Vanna API)

**수정 파일**: `infrastructure/helm-values/airflow.yaml` (airflow-triggerer)

**수정 파일**: `infrastructure/terraform/10-applications.tf` (Redash workers)

```yaml
# 공통 패턴 (스팟 파드에 적용)
tolerations:
  - key: "node-type"
    operator: "Equal"
    value: "spot"
    effect: "NoSchedule"

nodeAffinity:
  preferredDuringSchedulingIgnoredDuringExecution:
    - weight: 100
      preference:
        matchExpressions:
          - key: eks.amazonaws.com/capacityType
            operator: In
            values: ["SPOT"]
```

### Step 4. 적용 및 검증

```bash
# 1. 테라폼 적용 (노드 그룹 추가)
terraform apply -target="aws_eks_node_group.spot"

# 2. 스팟 노드 확인
kubectl get nodes -L eks.amazonaws.com/capacityType

# 3. 파드 배치 상태 확인 (어떤 파드가 어느 노드에)
kubectl get pods -A -o wide --sort-by='.spec.nodeName'

# 4. 전체 적용 후 비용 비교
kubectl top nodes
```
