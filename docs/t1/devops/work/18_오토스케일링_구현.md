# 작업 18: 오토스케일링 구현 (HPA + Cluster Autoscaler)

> **Phase**: 5 (Auto-Scaling)
> **담당**: Infra Architect / SRE Engineer
> **예상 소요**: 30~40분
> **이전 단계**: [17_모니터링_기본.md](./17_모니터링_기본.md)
> **참고**: [devops_implementation_guide.md](../devops_implementation_guide.md) §10

---

## 1. 목표

EKS 클러스터에 **HPA(Pod 수평 확장)** + **Cluster Autoscaler(노드 확장)** 두 계층의 오토스케일링을 구성합니다.

> **Why?**
> - 이전 작업(19번)에서 노드 2대 기준 리소스 경합(217% 초과)으로 Airflow/Redash 장애 경험
> - 리소스 Limits를 수동 다이어트하는 방식은 **부하 급증 시 무력함**
> - CA가 노드를 자동 증설하면 인력 개입 없이 장애를 예방 가능

---

## 2. 현재 상태 (Before)

| 항목 | 현재 설정 | 문제점 |
|------|----------|--------|
| **Node Group** | t3.medium × 2, min=2, max=3 | max=3으로 확장 여유 1대뿐 |
| **HPA** | 미설치 | Pod 수평 확장 불가 |
| **Cluster Autoscaler** | 미설치 | 노드 자동 증설 불가 |
| **metrics-server** | 상태 불명확 | HPA 동작의 전제조건 |

---

## 3. 목표 상태 (After)

| 항목 | 목표 설정 | 효과 |
|------|----------|------|
| **Node Group** | t3.medium × 2, min=2, **max=3** | 현재 노드 2대의 한계를 극복하기 위해 1대 추가 여유 확보 |
| **metrics-server** | Helm 배포 (k8s-sigs) | CPU/Memory 메트릭 제공 |
| **Cluster Autoscaler** | Helm 배포 (IRSA 연동) | Pending Pod 감지 → 노드 자동 추가 |
| **HPA** | 핵심 Pod에 적용 (Vanna API, Slack Bot) | CPU 70% 초과 시 Pod 수평 확장 |

---

## 4. 아키텍처

```
부하 급증 시 스케일링 흐름:

사용자 요청 증가
  ↓
[HPA] CPU 70% 감지 → Pod 수 증가 (예: 1 → 3)
  ↓
노드 리소스 부족 (Pending Pod 발생)
  ↓
[Cluster Autoscaler] Pending 감지 → AWS Node Group에 노드 추가 요청
  ↓
새 노드 Ready (~2분) → Pending Pod 배치 완료
  ↓
부하 감소 후 → HPA Pod 축소(5분 쿨다운) → CA 노드 축소(10분 쿨다운)
```

## 5. 구현 단계

### Step 1: Node Group max_size 확장 (06-eks.tf)

**파일**: `infrastructure/terraform/06-eks.tf`

```hcl
resource "aws_eks_node_group" "main" {
  # ... 기존 설정 유지 ...

  scaling_config {
    desired_size = 2
    min_size     = 2
    max_size     = 3  # 비용 최적화를 위해 최대 3대 유지
  }

  # CA가 노드를 인식할 수 있도록 태그 추가
  tags = {
    Name                                                = "${var.project_name}-node-group"
    "k8s.io/cluster-autoscaler/enabled"                 = "true"
    "k8s.io/cluster-autoscaler/${var.project_name}-cluster" = "owned"
  }
}
```

> **⚠️ 주의**: 태그가 없으면 CA가 이 Node Group을 관리 대상으로 인식하지 못합니다.

---

### Step 2: Cluster Autoscaler IAM Role 추가 (02-iam.tf)

**파일**: `infrastructure/terraform/02-iam.tf` (또는 별도 파일 `12-autoscaling.tf`)

```hcl
# ============================================================
# Cluster Autoscaler IRSA Role
# ============================================================
data "aws_iam_policy_document" "cluster_autoscaler_assume_role" {
  statement {
    actions = ["sts:AssumeRoleWithWebIdentity"]
    effect  = "Allow"

    principals {
      type        = "Federated"
      identifiers = [aws_iam_openid_connect_provider.eks.arn]
    }

    condition {
      test     = "StringEquals"
      variable = "${replace(aws_iam_openid_connect_provider.eks.url, "https://", "")}:sub"
      values   = ["system:serviceaccount:kube-system:cluster-autoscaler"]
    }
  }
}

resource "aws_iam_role" "cluster_autoscaler" {
  name               = "${var.project_name}-cluster-autoscaler-role"
  assume_role_policy = data.aws_iam_policy_document.cluster_autoscaler_assume_role.json
}

resource "aws_iam_policy" "cluster_autoscaler" {
  name = "${var.project_name}-cluster-autoscaler-policy"
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect = "Allow"
        Action = [
          "autoscaling:DescribeAutoScalingGroups",
          "autoscaling:DescribeAutoScalingInstances",
          "autoscaling:DescribeLaunchConfigurations",
          "autoscaling:DescribeScalingActivities",
          "autoscaling:DescribeTags",
          "autoscaling:SetDesiredCapacity",
          "autoscaling:TerminateInstanceInAutoScalingGroup",
          "ec2:DescribeInstanceTypes",
          "ec2:DescribeLaunchTemplateVersions",
          "eks:DescribeNodegroup"
        ]
        Resource = "*"
      }
    ]
  })
}

resource "aws_iam_role_policy_attachment" "cluster_autoscaler" {
  role       = aws_iam_role.cluster_autoscaler.name
  policy_arn = aws_iam_policy.cluster_autoscaler.arn
}
```

### Step 3: metrics-server + Cluster Autoscaler Helm 배포 (11-k8s-apps.tf)

**파일**: `infrastructure/terraform/11-k8s-apps.tf` 하단에 추가

```hcl
# ============================================================
# metrics-server (HPA 동작 전제조건)
# ============================================================
resource "helm_release" "metrics_server" {
  name       = "metrics-server"
  repository = "https://kubernetes-sigs.github.io/metrics-server/"
  chart      = "metrics-server"
  version    = "3.12.1"
  namespace  = "kube-system"

  set {
    name  = "args[0]"
    value = "--kubelet-insecure-tls"  # 기본 VPC 환경에서 TLS 인증 우회
  }

  depends_on = [module.eks]
}

# ============================================================
# Cluster Autoscaler
# ============================================================
resource "helm_release" "cluster_autoscaler" {
  name       = "cluster-autoscaler"
  repository = "https://kubernetes.github.io/autoscaler"
  chart      = "cluster-autoscaler"
  version    = "9.37.0"
  namespace  = "kube-system"

  set {
    name  = "autoDiscovery.clusterName"
    value = aws_eks_cluster.main.name
  }

  set {
    name  = "awsRegion"
    value = var.aws_region
  }

  set {
    name  = "rbac.serviceAccount.annotations.eks\\.amazonaws\\.com/role-arn"
    value = aws_iam_role.cluster_autoscaler.arn
  }

  set {
    name  = "extraArgs.balance-similar-node-groups"
    value = "true"
  }

  set {
    name  = "extraArgs.skip-nodes-with-system-pods"
    value = "false"
  }

  # 리소스 절약 (단일 AZ 환경)
  set {
    name  = "resources.requests.cpu"
    value = "100m"
  }
  set {
    name  = "resources.requests.memory"
    value = "128Mi"
  }
  set {
    name  = "resources.limits.cpu"
    value = "200m"
  }
  set {
    name  = "resources.limits.memory"
    value = "256Mi"
  }

  depends_on = [
    helm_release.metrics_server,
    aws_iam_role.cluster_autoscaler
  ]
}
```

### Step 4: HPA 적용 대상 선정

**적용 기준**: 부하 변동이 큰 서비스 & 상태 비저장(Stateless)

| 서비스 | HPA 적용 | 이유 |
|--------|----------|------|
| **Vanna API** | ✅ 적용 | 사용자 쿼리 기반, 부하 변동 큼 |
| **Slack Bot** | ✅ 적용 | 이벤트 기반, Stateless |
| **Airflow Scheduler** | ❌ 미적용 | StatefulSet, HPA 부적합 |
| **Redash** | ❌ 미적용 | DB 연결 수 제한 위험, 신중하게 접근 |
| **ChromaDB** | ❌ 미적용 | Stateful, 벡터 DB |

**HPA 설정** (`infrastructure/helm-values/vanna.yaml` 또는 별도 HPA 매니페스트):

```yaml
# vanna-hpa.yaml (kubectl apply -f 또는 Terraform kubernetes_manifest)
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: vanna-api-hpa
  namespace: ai-apps
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: vanna-api
  minReplicas: 1
  maxReplicas: 3
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # 5분 쿨다운
```

**Terraform HPA 리소스** (`11-k8s-apps.tf`에 추가):

```hcl
resource "kubernetes_horizontal_pod_autoscaler_v2" "vanna_api" {
  metadata {
    name      = "vanna-api-hpa"
    namespace = kubernetes_namespace.ai_apps.metadata[0].name
  }

  spec {
    scale_target_ref {
      api_version = "apps/v1"
      kind        = "Deployment"
      name        = "vanna-api"
    }

    min_replicas = 1
    max_replicas = 3

    metric {
      type = "Resource"
      resource {
        name = "cpu"
        target {
          type                = "Utilization"
          average_utilization = 70
        }
      }
    }

    behavior {
      scale_down {
        stabilization_window_seconds = 300
      }
    }
  }

  depends_on = [helm_release.vanna_api]
}

resource "kubernetes_horizontal_pod_autoscaler_v2" "slack_bot" {
  metadata {
    name      = "slack-bot-hpa"
    namespace = kubernetes_namespace.ai_apps.metadata[0].name
  }

  spec {
    scale_target_ref {
      api_version = "apps/v1"
      kind        = "Deployment"
      name        = "slack-bot"
    }

    min_replicas = 1
    max_replicas = 3

    metric {
      type = "Resource"
      resource {
        name = "cpu"
        target {
          type                = "Utilization"
          average_utilization = 70
        }
      }
    }
  }

  depends_on = [helm_release.slack_bot]
}
```

## 7. 검증 방법

### 7.1 metrics-server 동작 확인

```powershell
# metrics-server Pod 상태
kubectl get pods -n kube-system | Select-String "metrics-server"
# 예상: metrics-server-xxxxx   1/1   Running

# 노드 리소스 사용량 확인 (metrics-server 있어야 동작)
kubectl top nodes
# 예상:
# NAME                             CPU(cores)   CPU%   MEMORY(bytes)   MEMORY%
# ip-172-31-xx-xx.internal         250m         13%    1500Mi          48%
```

### 7.2 Cluster Autoscaler 동작 확인

```powershell
# CA Pod 상태
kubectl get pods -n kube-system | Select-String "cluster-autoscaler"
# 예상: cluster-autoscaler-xxxxx   1/1   Running

# CA 로그 확인 (정상 동작 여부)
kubectl logs -n kube-system deployment/cluster-autoscaler --tail=20
# 예상: "Successfully registered cluster" 등 메시지
```

### 7.3 HPA 동작 확인

```powershell
# HPA 상태 확인
kubectl get hpa -A
# 예상:
# NAMESPACE   NAME            REFERENCE              TARGETS   MINPODS   MAXPODS   REPLICAS
# ai-apps     vanna-api-hpa   Deployment/vanna-api   10%/70%   1         3         1
# ai-apps     slack-bot-hpa   Deployment/slack-bot   5%/70%    1         3         1
```

### 7.4 부하 테스트 (선택적 검증)

```powershell
# 부하 발생 (별도 터미널에서)
kubectl run load-test --image=busybox --restart=Never -it --rm -- \
  sh -c "while true; do wget -q -O- http://vanna-api.ai-apps.svc.cluster.local:8080/health; done"

# HPA 실시간 모니터링 (다른 터미널)
kubectl get hpa vanna-api-hpa -n ai-apps -w

# 예상: CPU% 상승 → REPLICAS 1 → 2 → 3으로 자동 증가
```

### 7.5 성공 기준

- [ ] `kubectl top nodes` 정상 출력 (metrics-server 동작)
- [ ] `kubectl get pods -n kube-system | grep cluster-autoscaler` → Running
- [ ] `kubectl get hpa -A` → TARGETS에 실제 %값 표시 (Unknown 아님)
- [ ] Node Group 태그 `k8s.io/cluster-autoscaler/enabled: true` 확인
- [ ] CA 로그에 에러 없음

## 8. 주요 트러블슈팅

| 증상 | 원인 | 해결 방법 |
|------|------|----------|
| `HPA: unknown` (TARGETS에 Unknown) | metrics-server 미동작 또는 Pod에 requests 미설정 | `kubectl top pods` 확인, Helm Values에 `resources.requests` 추가 |
| CA가 노드를 추가하지 않음 | Node Group 태그 누락 또는 IAM 권한 부족 | 태그 및 `aws_iam_policy.cluster_autoscaler` 권한 확인 |
| CA 로그 `i0` (스케일업 조건 미충족) | Pending Pod 없음 (정상 상태) | 의도적 부하 유발 후 재확인 |
| `metrics-server` CrashLoopBackOff | TLS 인증 실패 | `--kubelet-insecure-tls` args 추가 확인 |

## 9. 리소스 영향도 분석 (추가 비용)

| 컴포넌트 | 추가 리소스 | 비용 영향 |
|----------|------------|----------|
| metrics-server | CPU 10m / Mem 32Mi | 미미함 |
| Cluster Autoscaler | CPU 100m / Mem 128Mi | 미미함 |
| 신규 노드 (CA 트리거 시) | t3.medium = $0.052/시간 | **부하 시에만 과금** |

> **핵심**: CA는 부하 없을 때 기존 노드 2대를 유지합니다. 추가 비용은 실제 부하 때만 발생합니다.
