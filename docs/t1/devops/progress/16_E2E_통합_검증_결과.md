## 🔍 [1단계] 데이터 파이프라인 정합성 검증

### 1) 테스트 환경 및 조건
- **데이터 소스**: 로컬 `Log Generator` (Python)
- **수집 인프라**: AWS Kinesis Data Stream + Firehose
- **저장소**: S3 Data Lake (Parquet Format)
- **메타데이터**: AWS Glue Catalog (Crawler 방식)
- **쿼리 엔진**: AWS Athena

### 2) 검증 상세 결과
- **Log Generation**: 약 30분간 실행하여 실시간 로그 생성 및 전송.
- **S3 적재 확인**: `s3://capa-data-lake-xxx/raw/` 경로에 26개의 Parquet 파일 생성 확인.
- **자동 스키마 인식**: `capa-log-crawler`를 통해 `ad_events_raw` 테이블 및 `year/month/day` 파티션 자동 생성 성공.
- **데이터 정합성 (Athena 쿼리)**:
    - **총 레코드 수**: **1,504건** (충분한 테스트 표본 확보)
    - **스키마 매핑**: `event_id`, `event_type`, `timestamp`, `bid_price` 등 모든 컬럼이 정상 타입으로 매핑됨.
    - **조회 샘플**:
        ```json
        {
          "event_id": "3c3a7257-568c-4794-b3dc-f729e5ec4b5c",
          "event_type": "impression",
          "device_type": "iOS",
          "bid_price": 318.15
        }
        ```

### 3) 판정: ✅ **Pass**
> 로그 생성부터 Athena 쿼리까지 끊김 없는 데이터 흐름을 확인하였으며, 특히 Crawler 도입을 통해 스키마 관리 자동화까지 검증 완료함.

## 🔍 [2단계] Airflow 오케스트레이션 및 서비스 연동 검증

### 1) 테스트 환경 및 조건
- **워크플로우 엔진**: Airflow (EKS 클러스터 내 Helm 설치)
- **실행 방식**: `KubernetesPodOperator` (KPO)를 통한 태스크 격리
- **연동 대상**: 
    - **분석**: AWS Athena (Boto3 via Pod)
    - **트리거**: Report Generator Service (REST API via Curl Pod)

### 2) 검증 상세 결과
- **의존성 격리 확인**: 
    - 기본 이미지(`python-slim`)에서 `boto3` 부재로 인한 실패를 확인하고, 전용 이미지(`apache/airflow`)로 교체하여 성공함. 
    - **Airflow 본체의 수정 없이 태스크별 의존성 해결 능력을 증명함.**
- **Athena 집계 연동**:
    - `athena_aggregation_pod` 태스크가 오늘 날짜의 로그 데이터를 요약 쿼리함.
    - **성공 로그**: `[base] Aggregation Success` 확인. (Query ID: `218ec294...`)
- **서비스 간 트리거 연동**:
    - `trigger_report_pod` 태스크에서 `curlimages/curl` 이미지를 사용하여 내부 DNS 호출.
    - **성공 로그**: `{"status":"pending","message":"Report generation in progress"}` 응답 수신.
- **보안 및 권한**: 
    - `airflow-scheduler` ServiceAccount를 통해 IRSA 권한이 Pod에 정상 상속됨을 확인함.

### 3) 판정: ✅ **Pass**
> Airflow가 데이터 분석과 서비스 트리거라는 두 가지 핵심 오케스트레이션 역할을 모두 수행할 수 있음을 확인하였으며, 특히 KPO를 이용한 클라우드 네이티브한 아키텍처를 검증 완료함.

## 🔍 [3단계] Athena 통합 데이터 소스 검증

### 3.1) Vanna AI API 커스텀 연동 (✅ Pass)

#### ① 테스트 환경 및 조건
- **AI 서비스**: Vanna API (FastAPI 기반)
- **핵심 엔진**: `VannaAthena` (Anthropic_Chat, ChromaDB_VectorStore 상속)
- **사용 모델**: `claude-3-5-haiku-20241022`
- **데이터 소스**: Amazon Athena (DB: `capa_db`, Table: `ad_events_raw`)

#### ② 기술적 해결 및 검증 상세
- **라이브러리 제약 극복 (Custom Engine)**:
    - Vanna 0.7.9 버전에서 Athena를 직접 지원하지 않는 문제를 확인하고, `boto3` Athena 클라이언트를 직접 사용하는 분석 엔진을 자체 구현하여 `main.py`에 이식함.
    - `run_sql()` 메서드를 통해 쿼리 실행(`start_query_execution`), 상태 폴링, 결과 데이터프레임 변환 로직을 통합함.
- **Anthropic API 트러블슈팅**:
    - 초기 `claude-3-5-sonnet` 모델 호출 시 발생한 404 에러를 디버깅하여 모델 가용성 문제를 파악하고, 최신 가성비 모델인 `claude-3-5-haiku`로 교체하여 호출 성공함.
- **S3 Staging 및 IAM 권한 수선**:
    - Athena의 쿼리 결과 저장을 위해 `S3_STAGING_DIR` 환경 변수를 주입하고, `s3:GetBucketLocation` 및 `s3:PutObject` 권한을 IRSA 정책에 추가하여 `InvalidRequestException` 오류를 해결함.
- **자연어 질의 실전 검증**:
    - **질문**: "전체 광고 매출액(bid_price 합계)이 얼마인지 계산해줘"
    - **SQL 생성**: `SELECT SUM(bid_price) AS total_revenue FROM ad_events_raw`
    - **추출 결과**: `1,597,688.40` (Athena 데이터 직접 조회 및 수치 반환 성공)

#### ③ 판정: ✅ **Pass**
> 라이브러리 미지원 기능을 직접 구현으로 극복하였으며, LLM 모델 최적화와 인프라 권한 설정을 통해 AI가 데이터베이스 실측치를 정확히 읽어오는 데 성공함.

### 3.2) Report Generator 서비스 고도화 및 연동 (✅ Pass)

#### ① 테스트 환경 및 조건
- **서비스명**: Report Generator (FastAPI)
- **주요 기능**: 데이터 집계(Athena) + AI 인사이트 추출(Vanna API) + 리포트 생성
- **데이터 소스**: Amazon Athena (`ad_events_raw`)
- **분석 파트너**: Vanna API (`/query` 엔드포인트 호출)

#### ② 검증 상세 결과
- **Athena 통계 직접 추출 성공**:
    - `athena_utils.py` 구현을 통해 리포트 서버가 독립적으로 Athena 쿼리를 수행할 수 있음을 검증.
    - **오늘의 실측치 확보**: 노출 1,403회, 클릭 135회, 매출액 **1,597,688.41** 데이터 추출 성공.
- **서비스 간 체인 연동(Chained Integration)**:
    - `Airflow` -> `Report Generator` -> `Vanna AI`로 이어지는 3단 호출 체인이 정상 작동함을 확인.(airflow에서 리포트 제네레이터는 2단계에서 함)
    - 리포트 서버가 수집한 데이터를 Vanna AI에게 전달하여 분석을 요청하는 흐름 검증 완료.
- **인프라 환경 수선**:
    - **IAM 권한**: `glue:GetTable`, `s3:GetBucketLocation` 등 Athena 쿼리 필수 권한을 리포트 파드에 추가 주입.
    - **환경 변수**: `VANNA_API_URL` 주입을 통해 클러스터 내부 DNS 기반 통신 안정화.

#### ③ 판정: ✅ **Pass (Core Logic)**
> 단순 API 응답(Mock) 단계를 벗어나 실제 데이터베이스를 관통하고 AI에게 분석을 의뢰하는 비즈니스 로직의 E2E 연동을 완료함. (단, AI 엔진의 SQL 문법 최적화는 별도 진행 예정)

### 3.3) Redash 데이터 시각화 연동 (✅ Pass)

#### ① 테스트 환경 및 조건
- **서비스명**: Redash (Open Source Dashboard)
- **인프라**: EKS LoadBalancer (Port 5000)
- **데이터 소스**: Amazon Athena (DB: `capa_db`)
- **인증 방식**: IRSA (capa-redash-role)

#### ② 검증 상세 결과
- **인프라 표준화 (Workgroup)**: 
    - 전용 아테나 워크그룹(`capa-workgroup`)을 신설하여 쿼리 실행 정책 및 S3 결과 경로를 중앙 관리화함.
- **권한 연동 (IRSA)**: 
    - Redash 파드가 별도의 AWS Access Key 없이 EKS IAM Role을 통해 Athena에 접근 가능함을 확인.
- **데이터 소스 연결 성공**: 
    - Redash 설정 페이지에서 Athena 커넥터 테스트(Test Connection) 시 **"Success"** 확인.
- **시각화 및 쿼리 테스트 성공**: 
    - `ad_events_raw` 테이블에 대한 SQL 쿼리 실행 및 데이터 추출 성공.
    - 바 차트(Bar Chart) 생성 및 실시간 지표 대시보드 위젯 구성 완료.

#### ③ [Troubleshooting] 연동 중 시행착오 및 해결
- **이슈 1: Redis 인증 오류 (500 Internal Server Error)**
    - **증상**: 주소 접속 시 `Internal Server Error` 발생.
    - **원인**: Redash 헬름 차트 업데이트 과정에서 이전 버전의 Redis 인증 정보가 남아 `WRONGPASS` 오류 발생.
    - **해결**: `redis.auth.enabled=false` 설정을 테라폼에 명시하고, 관련 시크릿 삭제 후 모든 `redash-*` 파드를 `rollout restart` 하여 설정을 새로고침함.
- **이슈 2: Athena 작업 공간 설정 부재**
    - **증상**: 쿼리 실행 시 결과 저장 경로 설정 누락으로 인한 실패 가능성 식별.
    - **해결**: 테라폼으로 `aws_athena_workgroup`을 생성하여 S3 결과 경로를 인프라 차원에서 강제(Enforce)함.
