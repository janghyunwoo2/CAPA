# 🚀 EKS 오토스케일링 구현 및 노드 인스턴스 사양 업그레이드 보고서

본 문서는 EKS 클러스터의 고질적인 리소스 부족 및 노드 불안정(`NotReady`) 문제를 근본적으로 해결하기 위해 **노드 인스턴스 사양 업그레이드**와 **오토스케일링(HPA/CA) 시스템**을 구축한 과정을 기록합니다.

## 🔍 [1단계] 이슈 분석 및 솔루션 선정

### 1) 현상 (Task #19 리소스 튜닝의 한계)
- **메모리 임계치 도달**: 리소스 Limits를 깎는 '다이어트' 방식으로는 Airflow, Redash 등 헤비한 앱들을 2대의 `t3.medium`(4GB RAM) 노드에서 안정적으로 돌리기에 한계가 있음을 확인.
- **Max Pods 제한**: `t3.medium`은 노드당 최대 17개의 파드만 수용 가능하여, 클러스터 전체 파드(약 33개)를 감당하기에 여유가 전혀 없음.
- **불안정성**: 노드 CPU 점유율이 90%를 상회하며 Kubelet이 응답을 멈추는 현상 반복.

### 2) 해결책: 인프라 사양의 '그릇' 키우기
- **Instance Type**: `t3a.large` (2 vCPU, 8GB RAM) 채택.
  - 가격 대비 메모리가 2배로 늘어나 시스템 안정성 극대화.
  - 노드당 최대 파드 수용량 증설 (17개 -> 35개).
- **Capacity Type**: `ON_DEMAND`로 고정. 
  - 비용 절감보다는 안정적인 서비스 운영을 위해 Spot 인스턴스의 '중단 리스크' 원천 차단.

## 🔍 [2단계] 오토스케일링 시스템 구축

### 1) Cluster Autoscaler (CA)
- **설정**: `min_size=2`, `desired_size=2`, `max_size=3`
- **목적**: 평상시에는 2대로 운영하여 비용을 절감하되, 배치 작업(Airflow 등)이나 대량의 데이터 처리 시 자동으로 노드를 3대까지 늘려 장애 전파 차단.

### 2) Horizontal Pod Autoscaler (HPA)
- **대상**: Slack Bot, Vanna API, Report Generator
- **동작**: CPU 사용률 70~80% 도달 시 파드 개수를 자동으로 조절하여 서비스 응답성 유지.

## 📊 현재 진행 상황 및 기대 효과

### 1) 진행 상황
- 현재 테라폼 `-target` 명령을 통해 기존 노드를 폐기하고 새로운 `t3a.large` 노드 그룹을 생성 중입니다.
- 리대시(Redash) 등 애플리케이션 설정 변경과 충돌하지 않도록 노드 인프라를 먼저 선행 업그레이드합니다.

### 2) 기대 효과
- **OOM 장애 근절**: 가용 메모리가 2배로 늘어나 파드들이 더 이상 메모리 부족으로 죽지 않음.
- **운영 편의성**: 트래픽에 따라 노드와 파드가 알아서 늘어났다가 줄어드는 지능형 인프라 완성.
- **애플리케이션 가용성**: `t3a.large` 성능 대역에서 리대시 워커들이 더 빠른 쿼리 응답 속도를 보일 것으로 예상.
