## 1. DableTalk: SQL Agent 구축기 (Dable AI팀)

**출처**: https://dabletech.oopy.io/2ce5bbc0-e5c2-8089-9dcd-c449b51eba46  
**작성**: 이서림 (AI팀 데이터 엔지니어), 2025.12.19

### 개요
- Dable AI팀이 사내 비개발자를 위해 구축한 자연어 SQL 조회 서비스 "DableTalk"
- **타겟 사용자**: 사내 비개발 직군 (SQL을 모르지만 데이터 조회가 필요한 구성원)
- Slack Bot + Streamlit Web UI **2가지 인터페이스**로 제공
- 기술 스택: **Vanna AI Framework** + **Google Gemini-2.5 Pro** + **ChromaDB** + **Trino**
- 인프라: **Kubernetes** + **ECR** + **AWS Secrets Manager**

### 문제 정의: SQL 쿼리 요청이 쌓일수록 생기는 일들

- 비개발직군(PO, 마케터, 사업팀)이 데이터 접근을 위해 데이터팀에 SQL 요청
- 요청 → 맥락 파악 → SQL 작성 → 검증 → 전달의 반복
- 데이터팀의 리소스가 반복적 SQL 작성에 묶이는 병목 발생

### 핵심 아키텍처

```
[User Interfaces]        [Core Application]           [Data Layer]
Slack Bot            →   VannaDtalkApp            →   ChromaDB (Vector Storage)
(Slack Bolt, Python)      ├── Vanna AI Framework       Trino (Analytics DB)
Streamlit Web UI     →    ├── Google Gemini API        PostgreSQL (App Data)
                          └── ChromaDB Client

[Infrastructure]
Kubernetes → ECR (Container Registry) → AWS Secrets Manager
```

### 질문 처리 플로우 (핵심)

```
사용자 → Slack 스레드에서 @dtalk mention으로 질문
    ↓
1. [질문 유효성 판단]
   - 데이터 조회와 관계 없는 질문 필터링
    ↓
2. [SQL 생성 Prompt 구성]
   - ChromaDB에서 관련 스키마/예시 SQL 검색 (RAG)
   - 질문 + 검색 결과로 Prompt 구성
    ↓
3. [Gemini-2.5 Pro로 SQL 생성 요청]
    ↓
4. [SQL 문법 검증 - Trino EXPLAIN]
   - 생성된 SQL을 실행 전에 EXPLAIN으로 문법 검증
   - 비용 없이 SQL 유효성 사전 확인
    ↓
5-A. [성공 시]
   - Redash Query 자동 생성 → 시각화 링크 확보
   - Slack 스레드로 SQL + Redash Query 링크 응답
   - History DB에 저장 (질문-SQL-결과 쌍)

5-B. [실패 시]
   - 오류 정보 + 사용한 Prompt를 함께 응답 (투명성)
   - History DB에 저장 (실패 기록도 축적)
```

### Step 2: 질문 → SQL까지의 핵심 기능 4가지

| 단계 | 기능 | 설명 |
|------|------|------|
| **4.1** | 질문 유효성 판단 | 데이터와 무관한 질문(인사, 잡담)을 먼저 필터링 |
| **4.2** | RAG + Prompting | ChromaDB에서 관련 스키마/예시 SQL 검색 → LLM 프롬프트에 포함 |
| **4.3** | SQL 문법 검증 (EXPLAIN) | Trino `EXPLAIN`으로 SQL 유효성 사전 검증. 비용 없이 문법 오류 탐지 |
| **4.4** | Redash 자동 생성 + Slack 답변 | 검증 통과 시 Redash Query 자동 생성, Slack으로 SQL + 시각화 링크 전달 |

### Step 3: 운영과 개선 – LLM에 맥락 전달하기

LLM에 2종류의 지식을 보강하여 SQL 품질 향상:

#### 1) 회사 데이터 구조 관련 지식
- 테이블/컬럼별 상세 설명 (컬럼이 무엇을 의미하는지)
- 코드값 의미 매핑 (status='01'이 활성 상태인지 등)
- 테이블 간 조인 관계 정의
- 자주 쓰이는 SQL 패턴 문서화

#### 2) SQL 실행 환경 특화 지식 (Trino)
- Trino 방언/함수 사용법 (date 함수, 집계 함수 차이 등)
- 파티셔닝 규칙 (조회 시 반드시 파티션 조건 포함)
- Trino 특화 문법 주의사항

### 운영 및 모니터링: History와 피드백 루프

- 모든 질문-SQL 쌍을 **History DB**에 저장 (성공/실패 모두)
- 축적된 History를 분석해 자주 실패하는 패턴 발견 → 학습 데이터 보강
- 사용자 피드백을 통해 정답 쿼리를 계속 축적

### 남은 과제
- 복잡한 비즈니스 로직 반영 고도화
- 대화 맥락 유지 (멀티턴)
- 더 다양한 SQL 패턴 학습 필요

### CAPA 적용 시사점
- ✅ **Vanna AI 사용**: CAPA도 동일 프레임워크 사용 중 → 방향성 검증됨
- ✅ **SQL EXPLAIN 검증**: Athena에서도 `EXPLAIN`으로 비용 없이 SQL 유효성 사전 확인 가능
- ✅ **실패 투명성**: 오류 시 오류 정보 + Prompt를 함께 노출하는 UX → 사용자 신뢰 확보
- ✅ **Redash 연동**: SQL 결과를 Redash Query로 자동 생성 → 시각화 링크 제공 방식 참고
- ✅ **History 축적**: 질문-SQL 쌍을 DB에 저장해 피드백 루프로 품질 향상
- ✅ **SQL 환경 특화 지식**: Athena Presto SQL 방언/함수/파티션 규칙 문서 필수

## 2. 배달의민족 '물어보새': RAG + Text-to-SQL (우아한형제들)

**출처**: https://techblog.woowahan.com/18144/  
**작성**: BADA팀 (AI Data PM), 2024.07.12

### 개요
- 우아톤 2023(사내 해커톤) 1위 수상 후 TF를 구성해 발전시킨 서비스
- **타겟 사용자**: 전사 구성원 (95%가 데이터 활용). 특히 SQL 학습 시간이 부족하거나 복잡한 비즈니스 로직 반영이 어려운 비개발 직군
- 기술 스택: **GPT-4o** + **RAG** + **LangChain** + **Slack 인터페이스**

### 물어보새 아키텍처

```
1. [비정형 데이터 파이프라인]
   - 비즈니스 용어, 테이블 메타, SQL 코드 등을 벡터화 → VectorDB 저장
   - Airflow로 매일 자동 업데이트

2. [RAG 기반 멀티 체인]
   - Router Supervisor: 질문 의도 파악 → 적합한 체인으로 분기
     (쿼리 생성 / 해설 / 문법 검증 등)
   - 질문 유형에 따라 필요한 Retriever가 선별적으로 데이터 추출

3. [LLMOps]
   - 실험 환경 구축, 리더보드 성능 관리
   - API 로드 밸런싱, GPT 캐시 적용

4. [Slack 인터페이스]
   - 긍정/부정 피드백 반영
   - 캐시에 저장 → 표준 지식 확산
```

### Text-to-SQL 구현의 4가지 핵심

#### 1) 데이터 보강 (Garbage In, Garbage Out)

| 데이터 유형 | 설명 |
|------------|------|
| **풍부한 메타데이터** | 테이블 목적, 특성, 상세 설명, 주요 값/키워드를 추가한 고도화 DDL |
| **비즈니스 용어집** | 데이터 거버넌스 조직의 표준 용어 사전을 Text-to-SQL 전용으로 구축 |
| **Few-shot SQL** | 데이터 분석가가 작성한 고품질 쿼리-질문 쌍 데이터셋 |
| **자동 업데이트** | Airflow 비정형 데이터 파이프라인으로 매일 최신 정보 반영 |

#### 2) 검색 알고리즘 개발
- **질문 구체화**: 모호한 질문 → 비즈니스 용어 기반으로 정교화
- **동적 추출**: 질문 문맥에 맞춰 DDL / 메타 정보 / Few-shot SQL 중 최적 정보를 선택적 입력

#### 3) 프롬프트 엔지니어링
- **페르소나 부여**: 데이터 분석가 역할 설정
- **ReAct 활용**: Step-by-step 추론(COT) + 데이터 검색(Acting) 결합

#### 4) 실험 및 평가 시스템 구축
- **자체 리더보드**: Yale Spider, BIRD 벤치마크를 참고해 사내 도메인 특화 평가 데이터 개발
- **A/B 테스트**: 500번 이상 테스트 → 프롬프트, Retriever, 체인 조합 최적화
- **Langserve Playground**: 성능 변화 실시간 확인

### 핵심 교훈

1. **데이터 품질이 전부**: 모델 성능보다 메타데이터/비즈니스 사전의 품질과 양이 SQL 정확도를 결정
2. **내부 평가 체계 필수**: 공개 벤치마크로는 사내 비즈니스 문제 해결 불가 → 자체 리더보드 구축
3. **지속적 업데이트**: Few-shot SQL 등 학습 데이터를 도메인 전문가와 협업해 매일 최신화
4. **확장성**: 단순 쿼리 생성을 넘어 '데이터 디스커버리' 기능에 대한 요구가 큼

### CAPA 적용 시사점
- ✅ **비즈니스 용어 사전** 구축: CTR, ROAS, CVR 등 광고 도메인 표준 용어 정의 필수
- ✅ **Few-shot SQL 관리**: 도메인 전문가(광고 운영자)가 검증한 고품질 쿼리-질문 쌍 확보
- ✅ **자동 업데이트 파이프라인**: Airflow DAG로 학습 데이터 최신화 자동화 고려
- ✅ **평가 체계**: 광고 도메인 특화 Q&A 테스트셋으로 SQL 정확도 측정
- ✅ **Router(의도 분류)**: 질문 의도 파악 후 적합한 체인으로 분기하는 구조 참고

## 3. InsightLens: Text2SQL 기반 데이터 분석 서비스 (SK플래닛)

**출처**: https://tacademy.skplanet.com/front/centernews/viewCenterNews.do?seq=538  
**작성**: 김동국, 이준 (AT기술개발팀), 2026.02.24

### 개요
- LLM + RAG 기술을 결합한 사내 Text-to-SQL AI 에이전트 "InsightLens"
- **타겟 사용자**: 기획자, 마케터, 운영 담당자 등 데이터 분석팀에 Jira로 추출 요청하던 비개발 직군
- 기술 스택: **Python** + **LangChain/LangGraph** + **Vector Store** + **Reranker**
- 에이전트 그래프 구조: **의도 파악 후 분기 처리**, 각 노드는 단일 책임 원칙(SRP) 준수
- 인터페이스: **대화형 ChatGPT 스타일 UI** + **Jira 연동** (기존 워크플로우 유지)

### 개발 배경: 기존 데이터 추출 프로세스의 문제점

```
기존 프로세스:
1. 현업 부서 → 분석지원 Jira 티켓으로 데이터 추출 요청 (요청 내용 불명확한 경우 빈번)
2. 요청자-담당자 간 여러 차례 코멘트 주고받으며 요구사항 조율
3. 담당자가 테이블 스키마, 과거 히스토리 참고해 복잡한 SQL 작성
4. 결과 검토 → 예상과 다르면 피드백 → 쿼리 수정 반복

→ 단순 요청: 하루 이내 / 복잡한 요청: 며칠 소요
→ 담당자 휴가/바쁜 경우 대기 시간 더욱 증가
```

**핵심 가설**: "프론티어 모델에 적합한 데이터만 제공하면 복잡한 SQL도 충분히 생성 가능하지 않을까?"

### Text2SQL에 RAG가 필요한 이유

LLM 단독으로는 실무 환경에서 다음 문제가 즉시 발생:

| 문제 | 설명 |
|------|------|
| **스키마 환각** | 존재하지 않는 테이블/컬럼을 그럴듯하게 생성 |
| **조인 관계 추정 오류** | FK 관계나 조인 키를 잘못 추측 → 잘못된 결과 반환 |
| **도메인 규칙 누락** | "취소 제외", "특정 코드값 매핑", "파티션 처리" 등 업무 규칙 빠뜨림 |
| **조직 표준 미준수** | 집계 기준, 기간 정의, 지표 산식 등 내부 합의 미반영 |

> RAG는 문제를 "해결"한다기보다, LLM이 올바른 근거를 보고 생성하도록 유도해 **실패 확률을 크게 낮추는 방식**.

### Vector Store에 저장하는 3가지 데이터

| 데이터 유형 | 용도 | 데이터 소스 |
|------------|------|------------|
| **Q&A 데이터셋** | Few-shot Learning 예시로 활용 | 분석지원 Jira 티켓 + 정답 쿼리, 분석지원 포털, 쿼리 저장소, **사용자 직접 등록 정답** |
| **테이블 스키마** | 환각 방지, 실제 존재하는 테이블/컬럼만 사용 유도 | 사내 메타데이터 관리 시스템 연동 → **주기적 자동 업데이트** |
| **정책 데이터** | 비즈니스 규칙 반영 (파티션 UNION, 코드값 매핑 등) | 도메인 전문가가 작성한 규칙/지침 |

> **선순환 구조**: 사용자가 검증한 정답 쿼리 → Vector Store 축적 → 유사 질문 시 정답 참조 → 품질 향상

### 에이전트 그래프 노드 구조 (9단계)

핵심 설계 원칙: **"의도 파악 후 분기"** — 모든 질문을 동일 파이프라인으로 처리하지 않음

```
[시작]
  ↓
1. 이전 대화 내역 조회
   - 세션 ID 기반 MySQL에서 대화 기록 조회
   - "연령대별로 나눠줘" 같은 후속 질문의 맥락 유지
  ↓
2. 의도 분류 (LLM 기반)
   - SQL 조회 / 쿼리 수정 / 일반 질문 / 범위 외 등으로 분기
   - Few-shot examples 포함해 분류 정확도 향상
  ↓
3. 사용자 질문 정제
   - 인사말, 감사 인사, 부연 설명 제거
   - "안녕하세요, 매월 요청 드리는 건인데요, 1월 신규 가입자 수 부탁드립니다"
     → "1월 신규 가입자 수" (핵심만 추출)
  ↓
4. 키워드 추출
   - 정제된 질문에서 핵심 명사/용어를 단어 단위로 추출
   - 질문 전체보다 키워드로 검색이 더 정확 (특히 스키마 검색)
  ↓
5. Q&A 데이터셋 검색 파이프라인 (3단계)
   - 1단계: 벡터 유사도 검색 (후보 N개)
   - 2단계: Reranker로 관련성 재평가
   - 3단계: LLM 기반 최종 선별 (0개도 가능 → 노이즈 제거)

6. 테이블 스키마 조회
   - Q&A에서 사용된 테이블 정보 추출 + 키워드 기반 검색
   - 참조 횟수 기반 스코어링 → 우선순위 부여

7. 정책 조회
   - 키워드 기반으로 비즈니스 규칙 검색
   - 파티셔닝, 코드값 매핑, 집계 규칙 등
  ↓
8. 참조 데이터 수집 (멀티턴용)
   - 이전 턴에서 사용한 스키마/정책 정보를 함께 수집
   - 일관된 맥락에서 쿼리 수정 가능
  ↓
9. SQL 쿼리 생성/수정
   - 모든 참조 데이터를 종합한 최종 프롬프트로 SQL 생성
   - 응답: SQL + 쿼리 설명 + 예상 출력 컬럼 + 참조 데이터 목록
```

### Q&A 데이터셋 3단계 검색 파이프라인 (핵심)

| 단계 | 방식 | 목적 |
|------|------|------|
| **1단계** | 벡터 유사도 검색 | 후보 문서 N개 추출 |
| **2단계** | Reranker 모델 | 관련성 재평가, 순위 재조정 |
| **3단계** | LLM 기반 최종 선별 | 정말 도움 되는 문서만 선택 (0개도 가능) |

3단계 설계 이유:
- **정밀도 향상**: 단계마다 관련 없는 문서 제거 → 최종 품질 높음
- **노이즈 제거**: 관련 없는 참조 데이터가 프롬프트에 들어가면 오히려 혼란
- **토큰 최적화**: 컨텍스트 윈도우 제한 + 비용 절감
- **유연한 대응**: 정확히 일치하는 과거 사례 없으면 0개 선택 (억지로 포함시키지 않음)

### 프롬프트 엔지니어링 전략

#### 기본 원칙 5가지

| 원칙 | 설명 |
|------|------|
| **영어 기반 프롬프트** | LLM은 영어 지시가 더 정확 + 한국어보다 토큰 효율적 (응답은 한국어) |
| **XML 포맷 구조화** | `<instructions>`, `<table_schemas>`, `<examples>` 등 태그로 영역 구분 |
| **대화 맥락 유지** | 최근 N개 턴만 포함하거나 핵심 정보만 요약해서 포함 |
| **응답 포맷 명시** | 모범 예시와 함께 일관된 출력 포맷 강제 (파싱 용이) |
| **Few-shot Learning** | 의도 분류, 질문 정제, 키워드 추출 등 노드별 3~5개 예시 제공 |

#### SQL 생성 프롬프트 구조

```xml
<instructions>SQL 문법 규칙, 성능 지침, 출력 형식, 제약 조건</instructions>
<conversation_history>이전 대화 내역 + 이전 쿼리 + 참조 데이터</conversation_history>
<table_schemas>검색된 테이블 스키마 (참조 점수 포함)</table_schemas>
<qa_datasets>유사 Q&A 데이터셋 (과거 정답 쿼리 예시)</qa_datasets>
<policies>비즈니스 규칙 및 특수 처리 지침</policies>
<output_format>출력 형식 (SQL + 설명 + 예상 컬럼 + 참조 데이터 목록)</output_format>
<user_request>사용자의 현재 요청 (정제된 질문)</user_request>
```

### PoC 주요 발견

**성공 요인**:
- 프론티어 모델(Claude/OpenAI)의 SQL 생성 능력은 이미 충분
- 적절한 스키마 + 예시 쿼리를 컨텍스트로 주면 복잡한 JOIN/GROUP BY도 정확히 생성

**실패 원인** (참조 데이터 품질 문제):
- Q&A 데이터셋 **포맷 불일치** (질문-쿼리 구분이 불명확한 데이터 존재)
- **맥락 부재** (SQL은 있지만 어떤 비즈니스 요청인지 기록 없음)
- 스키마 **설명 부재** (컬럼이 있어도 의미를 모름)
- **코드값 정의 미흡** (`status = '01'`이 활성인지 비활성인지 문서화 안됨)

**개선 방향**:
1. Q&A 데이터셋 **표준 포맷 정의** (질문/SQL/사용 테이블 목록 필드 포함)
2. LLaMA-3.3-70B-Instruct로 **데이터 증강** (SQL → 자연어 질문 역생성)

### 개발 회고: 핵심 교훈

> **"모델 성능"보다 "참조 데이터 품질"이 실무 성능을 결정한다.**

| 교훈 | 상세 |
|------|------|
| **데이터 품질 > 모델 성능** | 스키마 설명 부실 → 컬럼 오해, 코드값 없음 → 잘못된 필터, 유사 정답 없음 → 패턴 재발명 시도 → 실패 |
| **검색 정확도 관리** | 정답과 거리가 먼 검색 결과 제외 필요 |
| **최신성 관리** | 스키마/정책 변경이 Vector Store에 늦게 반영되면 품질 저하 |
| **피드백 루프** | 사용자 검증 정답 쿼리 축적 → 검색 + 생성 품질 동반 개선 |
| **프롬프트 관리 체계** | 버전 관리, 테스트 자동화, A/B 테스트 등 LLMOps 필수 |

### CAPA 적용 시사점
- ✅ **스키마 품질이 핵심**: Athena 테이블 컬럼에 설명 + 코드값 의미 문서화 필수
- ✅ **Q&A 데이터셋 표준화**: 질문 / SQL / 사용 테이블 목록을 포함한 일관된 포맷
- ✅ **정책 데이터 별도 관리**: CTR/ROAS 계산식, 파티션 처리 규칙 문서화
- ✅ **3단계 검색 파이프라인**: 벡터검색 → Reranker → LLM 선별 (노이즈 제거)
- ✅ **의도 분류 → 분기 처리**: Slack Bot에서 범위 외 질문 명확히 처리
- ✅ **피드백 루프**: 사용자가 검증한 정답 쿼리 축적 → 품질 선순환
- ✅ **프롬프트 영어 기반**: SQL 생성 프롬프트는 영어로, 응답만 한국어로
- ✅ **Jira/Slack 연동**: 기존 워크플로우를 유지하면서 AI 기능 제공
