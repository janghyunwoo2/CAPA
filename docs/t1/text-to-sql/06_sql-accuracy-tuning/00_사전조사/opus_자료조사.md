## 조사 결과 요약

현재 시스템의 코드, 테스트 결과, 시딩 데이터를 전부 분석하고 최신 Text-to-SQL 연구를 조사한 결과를 정리했습니다.

### 핵심 진단: **왜 시딩과 프롬프트만으로는 안 되는가?**

현재 시스템은 **"1-shot 생성 → 그대로 실행"** 구조입니다. LLM이 한 번에 맞는 SQL을 뽑아야만 성공하고, 틀리면 그대로 실패합니다. QA 예제를 아무리 늘려도:

1. 프롬프트 규칙은 **참고사항**일 뿐, LLM이 100% 따르지 않음
2. 실패 케이스별 QA 추가는 **전형적 과적합** — 일반화 불가

### 과적합 없이 정확도를 높이는 5대 전략

| 순위 | 전략                                                                   | 효과     | 구현 난이도 |
| ---- | ---------------------------------------------------------------------- | -------- | ----------- |
| 🥇   | **Self-Correction Loop** (SQL 실패 시 에러 피드백 → 재생성 3회) | +5~15%p  | 낮음        |
| 🥈   | **질문 패러프레이징** (같은 SQL에 다양한 표현 시딩)              | +10~20%p | 낮음        |
| 🥉   | **Self-Consistency** (SQL 후보 여러 개 생성 → 다수결)           | +3~8%p   | 중간        |
| 4    | **Schema Linking** (질문 키워드로 관련 컬럼만 선별)              | +5~10%p  | 중간        |
| 5    | **Execution Feedback** (실행 결과 검증 → 재생성)                | +5~10%p  | 높음        |

### 즉시 조치 (Quick Win)

| 항목                                | 내용                                          |
| ----------------------------------- | --------------------------------------------- |
| **temperature=0**             | SQL 생성 시 결정론적 출력으로 안정성 확보     |
| **system/user 프롬프트 분리** | 현재 규칙이 user 메시지에 섞여 있어 효율 낮음 |
| **네거티브 규칙 강화**        | "하지 마라" 규칙 추가 (DATE() 금지 등)        |
| **RAG n_results 조정**        | 너무 많은 검색 결과가 노이즈로 작용           |

### 가장 중요한 결론

> **"시딩을 더 하거나, 실패 케이스마다 QA를 추가하는 건 하지 마세요."**
>
> 대신 **Self-Correction Loop** (3~5일 구현)만 추가해도 즉각적인 효과가 있습니다. SOTA 논문들(DIN-SQL, MAGIC, CSC-SQL)의 공통점은 **"한 번에 완벽한 SQL을 만들려 하지 않고, 생성→검증→수정 사이클을 돌린다"**는 것입니다.
