# 기술 선택 이유 (도구 채택 근거)

## 핵심 기술 선택 비교표

| 카테고리 | 채택 기술 | 비교 대안 | 채택 이유 |
|----------|-----------|----------|-----------|
| **Text-to-SQL 엔진** | **Vanna AI** | LangGraph 직접 구현 (InsightLens 방식) | `generate_sql()` 단일 호출로 RAG → LLM 파이프라인을 내부 추상화. 직접 구현 대비 개발 공수 70% 절감. 커스텀 ChromaDB 연동, 증분 학습(`vanna.train()`) API 제공으로 피드백 루프 구현이 용이. |
| **LLM (대형 언어 모델)** | **Anthropic Claude Haiku** (`claude-haiku-4-5-20251001`) | OpenAI GPT-4o, Google Gemini 2.5 Pro | Sonnet 대비 응답 속도가 빠르고 운영 비용이 낮아 Text-to-SQL 파이프라인의 다중 LLM 호출(의도 분류·질문 정제·SQL 생성·AI 분석)에 적합. Messages API의 멀티 content block 구조(`content[0]=instructions / content[1]=data`)를 통해 프롬프트 인젝션 방어(SEC-09) 가능. |
| **벡터 데이터베이스** | **ChromaDB** | Pinecone, pgvector, Weaviate | 완전 오픈소스(무료), Vanna AI SDK와 네이티브 통합. 로컬 → EKS 컨테이너로 동일 코드 그대로 배포. `hnsw:space: cosine` 옵션으로 ko-sroberta 임베딩 모델과 정합성 확보. 운영 복잡도 최소화. |
| **쿼리 엔진** | **Amazon Athena** | Trino(자체 클러스터), Amazon Redshift, BigQuery | Serverless → 클러스터 유지·운영 비용 없음. S3 + Parquet 네이티브 통합 (Firehose 파이프라인 그대로 연결). Presto 기반 SQL 문법은 Trino와 사실상 동일, 마이그레이션 부담 없음. 파티션 pruning(year/month/day/hour) 지원으로 스캔 비용 제어. |
| **BI 도구 / 쿼리 영속화** | **Redash** | Apache Superset, Metabase, Grafana | REST API 기반 쿼리 생성·실행·시각화 자동화 지원 (SQL 영속화 FR-11). 생성된 SQL을 Redash에 저장하면 사용자가 추후 직접 수정·재실행 가능. 오픈소스로 EKS 내 자체 호스팅 가능. |
| **메시지 인터페이스** | **Slack (Socket Mode)** | MS Teams, Discord, 자체 웹 UI | 사내 업무 커뮤니케이션 도구와 동일한 플랫폼으로 학습 비용 제로. ML 이상 탐지 결과를 차트 이미지와 함께 직접 전송(Block Kit) 하거나, Text-to-SQL 인터페이스(스레드 기반 멀티턴 세션)로 활용. |
| **스트리밍 데이터 파이프라인** | **Kinesis Data Stream + Firehose** | Kafka(MSK), Pub/Sub | AWS 네이티브 서비스로 별도 클러스터 운영 불필요. Firehose가 S3 Parquet 변환·파티셔닝을 자동 처리. Glue Catalog 스키마와 직접 연동되어 Athena 즉시 쿼리 가능. 소규모 트래픽에서는 Kafka 대비 비용 효율적. |
| **메타데이터 카탈로그** | **AWS Glue Catalog + Crawler** | AWS Glue ETL, Apache Hive Metastore | Crawler가 S3 Parquet 데이터를 자동 크롤링하여 스키마·파티션 정보를 Glue Catalog에 등록. Athena가 이 카탈로그를 기반으로 쿼리하므로 스키마 정의를 코드로 관리할 필요 없음. Firehose의 Parquet 변환 시 스키마 참조 용도로도 직접 활용. |
| **데이터 레이크 스토리지** | **Amazon S3** | Amazon EFS, HDFS, Azure Blob Storage | Parquet 포맷으로 Firehose가 직접 적재(별도 ETL 불필요). Athena의 쿼리 결과 저장소로도 사용. Versioning으로 데이터 보호, Lifecycle Policy(90일 자동 삭제)로 비용 절감. Glue Catalog·Firehose·Athena 모두 S3를 중심으로 네이티브 통합되어 파이프라인 구성이 단순. |
| **모니터링 및 데이터 소스** | **Amazon CloudWatch** | Datadog, Prometheus + Grafana | 지능형 이상 탐지 모델(`CloudWatchSource`)의 실시간 입력 데이터(Kinesis 지표) 저장소로 활용. EKS 컨테이너 로그 및 인프라 메트릭을 별도 구축 없이 중앙 집중식으로 관리 가능. |
| **지능형 이상 탐지 (ML)** | **Facebook Prophet + Isolation Forest** | Amazon Lookout for Metrics, Datadog Anomaly Detection | 광고 지표의 계절성(Prophet)과 통계적 이상치(Isolation Forest)를 동시 감지하여 오탐률 최소화. Python 생태계의 오픈소스로 EKS 컨테이너 내에서 독립적 실행 및 커스텀 로직(상/하단 임계치 가중치 등) 구현 가능. |
| **이상 탐지 스케줄러** | **Apache Airflow (KPO)** | CloudWatch Events + Lambda | `KubernetesPodOperator`(KPO)를 통해 5분 단위로 이상 탐지 전용 컨테이너를 실행. 실패 시 재시도(`retries`) 및 실행 이력 관리가 용이하며, 향후 모델 자동 재학습(Retrain) 파이프라인으로 확장이 가능함. |
| **컨테이너 오케스트레이션** | **Amazon EKS (Kubernetes)** | Amazon ECS, AWS Lambda, GKE | Helm 기반으로 Airflow·Vanna API·ChromaDB·이상 탐지 봇 등을 단일 클러스터에 배포·관리. IRSA(IAM Roles for Service Accounts)로 세밀한 AWS 권한 제어. Karpenter 연동으로 비용 효율적인 노드 자동 확장 가능. |
| **노드 자동 확장** | **Karpenter** | Cluster Autoscaler (CA) | Pending 파드 감지 후 20~60초 내 최적 노드 직접 프로비저닝(CA는 수 분 소요). 스팟 인스턴스 가격 비교 후 최저가 선택. 빈 노드 자동 반납(Consolidation) 지원. AWS EKS와 네이티브 통합으로 설정 단순화. |
| **IaC (Infrastructure as Code)** | **Terraform** | AWS CDK, CloudFormation, Pulumi | 멀티 클라우드·멀티 프로바이더 지원 (AWS + Kubernetes + Helm 단일 코드베이스). 선언형 HCL 문법으로 인프라 상태 관리 및 변경 이력 추적이 용이. 방대한 Provider 생태계(EKS, Karpenter, Helm 등)와 커뮤니티 모듈 활용. |
| **배치 스케줄러** | **Apache Airflow (EKS)** | AWS Step Functions, Prefect, Dagster | DAG 코드로 복잡한 ETL 의존성 표현 가능. KubernetesPodOperator로 독립적인 컨테이너 환경에서 각 Task 실행. 기존 EKS 클러스터 내 Helm 배포로 별도 인프라 불필요. 광범위한 커뮤니티와 플러그인 생태계 활용 가능. |
| **대화 이력 저장소** | **Amazon DynamoDB** | Amazon RDS(PostgreSQL), Redis, JSON Lines | 키-값 + GSI 구조로 `session_id` 기반 멀티턴 이력 조회(FR-20)를 O(1)로 수행. PROVISIONED 모드로 무료 티어(25 WCU/RCU) 내 운영 가능. TTL 기능으로 오래된 이력 자동 삭제. Serverless 특성으로 별도 DB 서버 관리 불필요. |
| **프롬프트 관리** | **YAML 파일** (intent_classifier.yaml 등) | 코드 내 하드코딩, Langchain PromptTemplate | 코드 배포 없이 프롬프트만 독립 수정·버전 관리 가능. 비개발자도 프롬프트 검토·수정 용이. Git 이력으로 프롬프트 변경 추적. 여러 파이프라인 Step이 동일 구조로 일관성 유지. |
| **컨테이너 레지스트리** | **Amazon ECR** | Docker Hub, GitHub Container Registry | EKS와 동일 AWS 계정 내 서비스로 인증 자동화 (IRSA). Private 레지스트리 보안. 이미지 취약점 스캔 내장. IAM 정책으로 접근 제어 용이. |

## 주요 아키텍처 의사결정 요약

### 1. Trino 대신 Athena를 선택한 이유

```
DableTalk(참고 사례): Trino  → 자체 클러스터 운영 필요, 비용·관리 부담 높음
CAPA:                Athena → Serverless, S3 + Parquet 네이티브, 운영 부담 없음
```
- Presto 기반 SQL 문법 동일 → Text-to-SQL 생성 SQL 마이그레이션 불필요
- 파티션 pruning(year/month/day/hour) 동일 지원

### 2. LangGraph 직접 구현 대신 Vanna AI를 선택한 이유

```
InsightLens(참고 사례): 9단계 에이전트 직접 구현 (질문 정제 → RAG → SQL 검증)
CAPA:                   Vanna AI가 내부 추상화 → generate_sql() 한 번 호출로 처리
```
- Vanna가 제공하지 않는 의도 분류·SQL EXPLAIN 검증은 직접 구현하여 보완

### 3. Cluster Autoscaler 대신 Karpenter를 선택한 이유

| 항목 | Cluster Autoscaler | Karpenter |
|------|-------------------|-----------|
| 노드 프로비저닝 속도 | 수 분 | 20~60초 |
| 인스턴스 선택 방식 | 기존 Node Group 중 선택 | AWS EC2 직접 주문 (최적화) |
| 스팟 회수 대응 | 수동 설정 | SQS 이벤트 자동 처리 |
| 비용 최적화 | 제한적 | Consolidation으로 빈 노드 자동 반납 |
