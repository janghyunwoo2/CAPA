항상 통합 테스트(E2E)를 통해 실제 응답 시간을 확인해야 함.

그리고 배포와 개발 환경을 구분(Feature Flag)하여 개발하는 것이 필수적임.

### Reranker: 0% 탈출의 주역에서 성능의 병목으로
*   **도입 배경 (성능 비약적 향상)**: 프로젝트 초기, 기본 검색만으로는 DDL(테이블 구조)을 제대로 찾지 못해 **정답률(Exec Accuracy)이 0%**가 나오는 치명적인 문제가 있었음. 이를 해결하기 위해 `jina-reranker-v2-base-multilingual`을 도입했고, DDL 검색 정확도가 90% 이상으로 올라가며 **베이스라인 확보 및 초기 정답률 상승의 일등공신** 역할을 했음.
*   **최종 롤백/비활성화 사유 (Trade-off)**:
    *   **성능 저하(Latency)**: 정확도는 확보했으나 CPU 환경에서 리랭킹에만 **57초** 소요. 슬랙 봇 응답 속도로는 부적합하여 타임아웃 위험이 큼.
    *   **검색 모델 최적화**: 그 사이 임베딩 모델(ko-sroberta)과 시딩 데이터셋을 고도화한 결과, 리랭커 없이도 기본 검색 유사도가 평균 0.88까지 올라오는 '상향 평준화'가 이루어짐.
    *   **미미한 추가 이득**: 전수 조사 결과, 리랭커를 유지했을 때 얻는 추가 정확도 이득이 2~3% 수준에 불과했음. 1분의 대기 시간을 감수할 만큼의 임팩트가 아니라고 판단함.
*   **결론**: 리랭커는 **"문제 해결의 시작(0% 탈출)"**에는 필수적이었으나, 시스템 안정화 단계에서는 **"가성비와 실시간성"**을 위해 제거하는 것이 최적의 아키텍처적 결정이었음.
