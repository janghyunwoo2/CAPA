# CAPA: 마케터를 위한 Text-to-SQL 광고 분석 플랫폼 발표 스토리라인 (20분 분량)

> **발표자 노트**: 20분 발표 분량에 맞춰 **총 24장** 슬라이드로 구성했습니다. 각 장표당 약 50초 내외를 권장합니다.
> **프로젝트 공식 명칭**: CAPA (Cloud-native AI Pipeline for Ad-logs)
> - **팀명/프로젝트 의미(Capacity)**: 어떤 시스템이나 사람이 처리할 수 있는 최대 능력치. 마케터 데이터 분석의 한계를 끌어올린다는 포부를 담음.
> - **서비스 정의**: 온라인 광고 로그 처리 파이프라인 및 분석 플랫폼

- **부제**: Cloud-native AI Pipeline for Ad-logs 기반 Text-to-SQL 분석 플랫폼
- **팀명 의미 (Capacity) 소개**: "CAPA는 Capacity(수용력)의 줄임말로, 시스템이나 사람이 처리할 수 있는 최대 능력치를 뜻합니다."

### **Slide 4: 왜 이 문제가 구조적으로 생기나?**
- **발표자 멘트**: "이건 담당자가 불성실한 게 아닙니다. 구조의 문제입니다."
- **원인 1 - 기술 장벽**: 데이터를 꺼내려면 SQL을 알아야 합니다. 이 장벽 탓에 기업이 수집한 데이터의 **68%는 비개발 직군에게 닿지 못하고 방치**됩니다. ([Seagate & IDC Report](https://www.seagate.com/content/dam/seagate/migrated-assets/www-content/our-story/rethink-data/files/Rethink_Data_Report_2020.pdf))
- **원인 2 - 분석팀 병목**: 결국 모든 요청이 소수의 분석팀에게 몰립니다. 진짜 병목은 기술이 아니라 **"끝없는 요청 vs 소수의 분석팀"** 구조입니다. ([Instawork: Analytics in the Age of AI](https://engineering.instawork.com/analytics-in-the-age-of-ai-864a4fb980e0))
- **발표자 멘트**: "그런데 이 문제가 가장 극단적으로 터지는 곳이 있습니다. 바로 광고 도메인입니다."

### **Slide 5: 왜 하필 광고 도메인인가?**
- **이유**: 광고는 캠페인을 하루에도 수십 번 켜고 끕니다. CTR(클릭률), CVR(전환율)을 빠르게 확인하고 즉각 액션을 취해야 수익이 나는 업종입니다. 타이밍이 곧 돈입니다.
- **현재 프로세스 (AS-IS)**:
  ```text
  마케터: "어제 A캠페인 CTR이 왜 떨어졌지?"
     ↓
  데이터팀 백로그 대기     → 수 시간 지연
  요청 해석 및 SQL 작성    → 1~2시간
  결과 전달 및 추가 요청    → 전체 반복
  ======================================
  총 소요 시간: 최소 반나절 ~ 3일
  ```
- **발표자 멘트**: "이 대기 시간 동안 비효율 캠페인은 예산을 계속 태우고 있습니다. 광고 마케터가 데이터팀에 의존할 수밖에 없는 순간은 크게 4가지가 있습니다. 데이블, SK플래닛, 물어보새 같은 기업들이 이 문제를 해결하려고 Text-to-SQL 기술을 도입하기 시작한 이유이기도 합니다."

### **Slide 6: CAPA가 제시하는 해결책 (4가지 핵심 기능)**
- **발표자 멘트**: "저희 CAPA는 마케터가 데이터팀에 손 내밀 수밖에 없는 4가지 상황을 각각 해결합니다."
- **4가지 상황과 해결책**:
  | 상황 | 기능 | 해결 |
  | --- | --- | --- |
  | 🤔 갑자기 궁금한 게 생겼다 | 🗣️ **Ask** | SQL 몰라도 슬랙에서 자연어로 즉시 조회 |
  | 📅 매주 같은 리포트를 사람이 만든다 | 📝 **Report** | Airflow + Claude가 자동 생성·전송 |
  | 😴 내가 안 보는 사이 로그 유입이 이상해졌다 | 🔔 **Alert** | Impression/Click/Conversion 건수 실시간 이상 감지 |
  | 👀 늘 봐야 하는 지표가 있다 | 📊 **Dashboard** | SQL 없이 Redash 대시보드 원클릭 |
- **발표자 멘트**: "말로 설명하기 전에, 실제로 어떻게 바뀌는지 먼저 보여드리겠습니다."

## 2. 시연 영상 (약 3분)

### **Slide 8: 데모 Scene 1 - 자연어 → SQL 응답 (속도)**
- 슬랙에 일상어로 질문 후 수초 만에 SQL 및 결과 표 렌더링 화면 응답 확인 (직접 녹화 화면).
- **포인트**: "반나절 걸리던 작업이 단 수십 초 만에."

### **Slide 9: 데모 Scene 2 - Redash 대시보드 딥링크**
- 응답에 포함된 딥링크를 눌러 Redash 대시보드가 뜨는 모습.
- **포인트**: SQL 지식 없이 시각화 대시보드까지 원클릭.

### **Slide 10: 데모 Scene 3 - 이상치 알림**
- 광고 로그 유입량(Impression/Click)이 비정상적으로 급변할 때 Slack 알림 수신.
- **포인트**: 내가 안 보는 사이에도 이상 감지 → 엔지니어 개입 없이 즉시 마케터에게 알림.

## 3. 전체 프로세스 (약 8분)

### **Slide 11: CAPA 시스템 조감도 및 광고 데이터 흐름**
- **시각화**: 이벤트 발생(앱) → 수집 파이프라인 → 데이터 변환/저장 → 데이터 마트 → AI 엔진 → 인터페이스(Slack) 통합 조감도.
- **도메인 지식 (배달앱 광고 로그 Funnel)**:
  - `Impression(노출)`: 배너 렌더링 창출 (가장 트래픽 볼륨이 거대함).
  - ➔ `Click(클릭)`: 유저 관심 맵핑.
  - ➔ `Conversion(전환/구매)`: 최종 수익 달성.
- **발표자 멘트**: "데모가 가능하려면 보이지 않는 백엔드에서 방대한 로그를 실시간으로 견뎌내는 인프라가 필수입니다. 구조를 하나씩 짚겠습니다."

### **Slide 12: 실시간 로그 스트리밍 (Kinesis)**
- **사용 기술**: Kinesis Data Stream + Kinesis Firehose
- **Why?**:
  - Fan-out 아키텍처: 실시간 이상치 탐지와 장기 보관(S3)을 동시에 지원.
  - 트래픽 스파이크 시에도 버퍼링으로 로그 유실 방지.

### **Slide 13: Data Lake 및 서버리스 쿼리 엔진**
- **사용 기술**: S3 (Parquet 저장) + Glue Catalog + Athena
- **Why?**: 데이터는 저렴한 S3에 담고, Glue로 스키마 관리, Athena로 조회할 때만 비용 발생 — 가장 경제적인 구조.

### **Slide 14: "왜 챗GPT를 바로 쓰지 않았을까?" (LLM 한계)**
- **Hallucination 문제**: 일반 LLM에 "어제 캠페인 실적 줘"라고 하면 존재하지 않는 테이블을 찾거나 없는 컬럼명을 지어내는 환각 현상이 발생합니다.

### **Slide 15: RAG + Vector DB 기반 Text-to-SQL (Vanna AI)**
- **차별점**:
  - ChromaDB에 사내 **데이터 스키마(Glue DDL)**와 **마케터 예시 쿼리(SQL)**를 학습.
  - 질문을 RAG(유사도 검색)에 먼저 던져 관련 문맥을 추출, LLM 프롬프트에 주입 → 환각 억제.
  - 데이블, SK플래닛 등도 유사한 방식으로 RAG 기반 Text-to-SQL을 도입.

### **Slide 16: 봇의 응답 파이프라인 상세**
- 질문 입력 ➜ 의도 파악 및 Vector 검색 ➜ 정교한 프롬프트 생성 ➜ SQL 생성 및 Athena 실행 ➜ 결과 오류 검증(Self-correction) ➜ Slack 포맷팅 출력.

### **Slide 17: 대시보드 구축 (Why Redash?)**
- **Why?**:
  - Athena 데이터소스 네이티브 지원.
  - 생성된 SQL 쿼리를 Vanna AI 학습 데이터(Training Data)로 역활용 가능.
- **핵심 지표**: CTR, CVR, CPC 3대 지표 위주 마케터 친화적 UI.

### **Slide 18: 주간 성과 자동화 리포트 (Airflow + Claude)**
- **도입 배경**: 마케터가 주간 회의 때마다 수기로 결산하는 시간 낭비.
- **해결**: Airflow 스케줄러가 매주 월요일 아침 9시 지난주 데이터를 집계 → Claude API로 인사이트·권고 액션 도출 → PDF/Markdown 자동 발송.

### **Slide 19: 실시간 이상치 탐지 (Prophet + Isolation Forest)**
- **탐지 대상**: Impression / Click / Conversion — 3개 로그 유형별 독립 탐지 서비스.
- **파이프라인**: CloudWatch(Kinesis) → **5분 Tumbling Window** 집계 → 이중 모델 판별 → Slack 알림.
- **이중 앙상블 모델**:
  - **Prophet**: 배달앱 트래픽 계절성(점심 ×3, 저녁 ×4, 주말 ×1.4) 학습 → 95% 신뢰구간 이탈 시 Sudden Drop / Sudden Spike 분류.
  - **Isolation Forest**: Prophet이 놓치는 극단 이상값 보완 탐지.
- **운영**: Airflow DAG이 매 5분마다 Docker 컨테이너 자동 실행. 이상 감지 시 PNG 차트와 함께 Slack 즉시 알림.

### **Slide 20: 인프라 코드화 + 트래픽 대응 (Terraform + EKS)**
- **Terraform (Why?)**:
  - AWS 리소스(`base`)와 K8s 앱(`apps`) 상태(State) 분리.
  - `terraform apply` 한 번에 전체 파이프라인 재현 자동화. (마우스 클릭 0번)
- **Auto-scaling**:
  - HPA: 파드 CPU 70% 초과 시 즉시 파드 수 확장.
  - Cluster Autoscaler: EKS 노드 2개 → 4개 자동 확장.
- **보안 (Least Privilege)**: IRSA로 파드별 최소 권한 정책 적용. 파드가 털려도 S3 전체 권한 탈취 방지.

## 4. 결론 (약 4분)

### **Slide 21: AS-IS vs TO-BE 도입 전후 비교**
- **정량 성과 비교**:
  | 항목 | AS-IS (도입 전) | TO-BE (CAPA 도입 후) |
  | --- | --- | --- |
  | 데이터 요청~응답 시간 | 최소 1시간 ~ 3일 | **약 수십 초 내** |
  | 데이터팀 개입 여부 | 매 요청마다 필수 | **불필요** |
  | 리포트 작성 | 수기 엑셀 (수시간) | **자동 생성** |
  | 이상치 감지 | 사람이 직접 확인 | **실시간 자동 알림** |
  | 마케터 데이터 접근성 | SQL 지식 필요 | **자연어로 즉시 조회** |
- **결론**: 데이터 추출 업무 소요 시간 **약 99% 단축** (AS-IS 최소 1시간 = 3,600초 vs TO-BE 수십 초 기준). 마케터는 즉각적인 지표 확인을 통해 최적의 광고 집행 타이밍을 확보합니다.
- **한 줄 슬로건**: **"마케터는 오직 마케팅 본질에만 집중하십시오. 데이터 조회는 CAPA가 맡겠습니다."**

### **Slide 22: 시행착오 & 아쉬운 점 (Lessons Learned)**
- **시행착오 1 - EKS StorageClass 이슈**:
  - `PersistentVolumeClaim`이 `Pending` 상태에서 멈추는 문제 발생.
  - 원인: EBS CSI Driver가 기본 설치되지 않는 EKS 버전 차이.
  - 해결: Terraform add-on으로 `aws-ebs-csi-driver` 코드화.
- **시행착오 2 - Nested 쿼리 SQL 오류율**:
  - 복잡한 하위 질의 생성 시 문법 오류율 증가.
  - 해결: Prompt Engineering 강화 + 집계 뷰(View) 테이블 사전 생성 후 RAG 투입.
- **아쉬운 점**:
  - 평가 체계(Spider EM/Exec) 구축이 일정상 완전히 마무리되지 못함.
  - 시뮬레이션 로그 기반 검증이라 실 서비스 적용 시 추가 튜닝 필요.
