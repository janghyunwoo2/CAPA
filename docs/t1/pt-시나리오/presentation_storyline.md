### **Slide 4: 왜 이 문제가 구조적으로 생기나?**
- **발표자 멘트**: "이건 담당자가 불성실한 게 아닙니다. 구조의 문제입니다."
- **원인 1 - 기술 장벽**: 데이터를 꺼내려면 SQL을 알아야 합니다. 이 장벽 탓에 기업이 수집한 데이터의 **68%는 비개발 직군에게 닿지 못하고 방치**됩니다. ([Seagate & IDC Report](https://www.seagate.com/content/dam/seagate/migrated-assets/www-content/our-story/rethink-data/files/Rethink_Data_Report_2020.pdf))
- **원인 2 - 분석팀 병목**: 결국 모든 요청이 소수의 분석팀에게 몰립니다. 진짜 병목은 기술이 아니라 **"끝없는 요청 vs 소수의 분석팀"** 구조입니다. ([Instawork: Analytics in the Age of AI](https://engineering.instawork.com/analytics-in-the-age-of-ai-864a4fb980e0))
- **발표자 멘트**: "그런데 이 인간 병목이 의사결정 비용으로 직결되는 도메인이 있습니다. 바로 광고입니다."

### **Slide 5: 왜 하필 광고 도메인인가?**
- **핵심**: 광고는 캠페인 성과를 빠르게 보고 즉각 예산을 조정해야 수익이 납니다. 데이터는 주기적으로 집계됩니다. 문제는 그 데이터가 준비됐는데도 **데이터팀을 통해서만 꺼낼 수 있다**는 것. 요청 → 대기 → 전달. 이 병목이 의사결정을 막습니다.
- **광고 마케터가 매일 막히는 4가지 순간**:
  | # | 상황 | 고통 |
  |---|------|------|
  | 1 | 갑자기 데이터가 궁금하다 | SQL 모르면 데이터팀 요청 → 수 시간 대기 |
  | 2 | 매주 같은 리포트를 만든다 | 수기 엑셀 집계 → 반복 작업에 수 시간 낭비 |
  | 3 | 내가 안 보는 사이 로그가 이상해졌다 | 뒤늦게 발견 → 예산 이미 낭비됨 |
  | 4 | 늘 봐야 하는 지표가 있다 | 매번 요청 반복 → 리소스 낭비 |
- **발표자 멘트**: "이 4가지가 광고 도메인에서 매일 반복됩니다. 저희 CAPA는 이 4가지를 각각 해결했습니다."

### **Slide 12: 데이터 수집 및 저장 (Kinesis + S3/Athena)** (시현)
- **왜 데이터를 3개 스트림으로 분리했나? (Impression / Click / Conversion)**:
  - 노출, 클릭, 전환은 발생하는 시점(수 초 ~ 길게는 수 일)과 트래픽 볼륨이 완전히 다릅니다. 이들을 하나의 테이블로 억지로 묶지 않고 3개의 독립된 Kinesis Stream으로 분리 수집하여, 실제 프로덕션 광고 시스템의 비동기적 퍼널 발생 구조를 정확하게 모사했습니다.
- **수집 — Kinesis Data Stream + Firehose**:
  - Fan-out 아키텍처: 실시간 이상치 탐지용 (Lambda/Prophet)과 장기 보관용(S3)에 동시 송출하여 트래픽 스파이크 시에도 로그 유실을 방지합니다.
- **저장 — 풀스캔(Full Scan) 요금 폭탄을 막는 3가지 최적화 전략 (S3 + Parquet + Airflow ETL)**:
  - 데이터는 저렴한 S3에 담고 서버리스 쿼리 엔진인 Athena를 사용하지만, Athena는 **조회 스캔 용량만큼 과금**되므로 막연한 쿼리는 요금 폭탄을 부릅니다.
  - **① 파티셔닝(Partition Pruning)**: `year=.../month=.../day=.../hour=...` 구조로 폴더를 쪼개어 단일 날짜 조회 시의 버킷 전체 스캔을 방지합니다.
  - **② Parquet 형식 저장**: 기존 CSV/JSON 대신 컬럼 기반 특화 포맷인 Parquet으로 압축 저장하여, 디스크 공간 절약은 물론 Athena의 스캔 비용을 획기적으로 낮췄습니다.
  - **③ Airflow 요약(Summary) 테이블 생성**: 마케터가 쿼리할 때마다 방대한 3개의 원본(Raw) 로그를 매번 조인하면 비용·시간이 극심합니다. 이를 막기 위해 **Airflow ETL이 1시간/1일 단위로 로그를 가공해 Summary 테이블을 사전 생성**해두어, 대시보드와 LLM은 가볍고 빠른 요약 연산만 수행하도록 성능을 극대화했습니다.

### **Slide 13: 11단계 Text-to-SQL 파이프라인**
- **왜 일반 LLM으로는 안 되나?**: 일반 LLM은 사내 DB 스키마를 모릅니다 → 없는 테이블·컬럼명을 지어냅니다 (할루시네이션).
- **해법**: ChromaDB에 사내 DDL + 마케터 예시 SQL을 학습시키고, 질문마다 관련 문맥만 골라 LLM에 주입합니다 (RAG).
- **11단계 파이프라인** — 사용자 질문 1개가 이 단계를 순서대로 통과합니다:

| 단계 | 컴포넌트 | 역할 |
|------|----------|------|
| Step 0 | Conversation Retriever | DynamoDB에서 이전 대화 이력 불러오기 |
| Step 1 | Intent Classifier | 광고 도메인 질문인지 / 범위 외 질문인지 분류 |
| Step 2 | Question Refiner | 인사말 제거, 지표 키워드 중심으로 질문 정제 |
| Step 3 | Keyword Extractor | 도메인 핵심 키워드 추출 (CTR, 기기, 캠페인 등) |
| Step 4 | RAG Retriever | ChromaDB에서 유사 SQL 예제 + DDL 검색 |
| Step 5 | SQL Generator | Claude CoT(Chain-of-Thought)로 Athena SQL 생성 |
| Step 6 | SQL Validator | AST 파싱 + EXPLAIN 실행으로 문법·보안 검증 |
| Step 6.5 | Self-Correction | 검증 실패 시 에러 피드백으로 최대 3회 자동 재생성 |
| Step 7~9 | Query Execution | Redash 경유 → Athena 실행 → 결과 수집 (비동기 모드 지원) |
| Step 10 | AI Analyzer | 결과 해석, 인사이트 도출, 차트 유형 결정 |
| Step 11 | History Recorder | 전체 트랜잭션 DynamoDB에 기록 |

- **멀티턴 대화**: Slack 스레드의 `thread_ts`를 `session_id`로 사용 → DynamoDB에서 최근 5턴 이력 조회 → Question Refiner·SQL Generator 프롬프트에 동적 주입. "그 다음은?", "기기별로 보여줘" 같은 후속 질문도 맥락을 유지하며 처리 가능.
- **비동기 쿼리 응답**: Athena 쿼리가 수십 초 이상 걸릴 경우 즉시 `task_id` 반환 → 클라이언트가 폴링 → DynamoDB `async-tasks` 테이블에서 상태 추적 → 완료 시 결과 전달. Slack API 타임아웃(3초) 초과 없이 대용량 쿼리 처리 가능.

### **Slide 14: SQL 정확도를 높인 핵심 기술들**
- **발표자 멘트**: "파이프라인을 만들었다고 끝이 아닙니다. 틀린 SQL이 나오면 무용지물입니다. 프롬프트와 시딩을 계속 고치면서 정확도를 끌어올렸습니다."

**[검증 레이어]**

| 기술 | 무엇을 하나 | 왜 필요했나 |
|------|------------|------------|
| **3중 SQL 검증** | ① 차단 키워드 필터(DROP·DELETE·INSERT·UPDATE·DDL 10종) → ② sqlglot AST 파싱 + SELECT 전용 + 허용 테이블 화이트리스트 → ③ EXPLAIN 실행 | LLM이 악의적 Prompt Injection으로 DROP/DELETE를 생성해도 실행 전 차단. 허용된 2개 테이블 외 접근 불가. |
| **Self-Correction Loop** | 검증 실패 시 에러 메시지를 LLM에 돌려줘 최대 3회 자동 재생성 | 오류 원인을 알려주면 LLM이 스스로 고침 |

**[프롬프트 엔지니어링]** — "LLM에게 우리 Athena의 규칙을 직접 가르치는 작업"

| 고민한 것 | 적용 방법 |
|----------|----------|
| **Athena 전용 문법 강제** | DATE() 함수 금지, BETWEEN 금지, 파티션(`year`/`month`/`day`) STRING 형식 필수, NULLIF로 0나누기 방지 — 21가지 Negative Rules로 명시 |
| **지표 공식 내재화** | CTR = clicks/impressions, CVR = conversions/clicks, ROAS = revenue/cost 등 공식을 프롬프트에 직접 삽입 |
| **CoT 6단계 사고 순서** | "어떤 테이블? → 필요한 컬럼? → 집계 방식? → 날짜 파티션? → 보안 검증? → 최종 SQL" 순서로 유도 |
| **프롬프트 인젝션 방지** | AI Analyzer에서 시스템 지시문과 Athena 쿼리 결과를 별도 content block으로 완전 분리 → 데이터 안에 숨어있는 악의적 지시가 시스템 역할을 탈취하는 것 방지 |

**[RAG 시딩 최적화]** — "벡터 DB에 무엇을, 어떻게 넣느냐가 정확도를 결정"

| 고민한 것 | 적용 방법 |
|----------|----------|
| **동적 DDL 주입** | DDL을 시딩하지 않고, QA Pair의 메타데이터(`tables` 필드)에서 실제 사용 테이블을 역추적해 해당 DDL만 주입 (Metadata Backtracking) |
| **cosine 거리 지표 전환** | 임베딩 모델(ko-sroberta)이 cosine 유사도 기준인데 ChromaDB 기본값은 L2 거리 → 불일치로 검색 품질 저조 → cosine으로 강제 전환 |
| **문장형 Documentation 전환** | 섹션형 구조화 문서는 임베딩 매칭 성능 저하 → "CTR이 낮을 때는 창의적인 소재를 검토해야 한다"처럼 완성된 문장으로 전면 재작성 |
| **Negative Examples 시딩** | 오답 패턴(NULLIF 미사용 CTR 계산, 파티션 하드코딩, conversion 테이블 오용 등 6가지)을 ChromaDB에 별도 시딩 → LLM이 자주 하는 실수를 사전 학습 |
| **학습 데이터 삭제 관리** | `DELETE /training-data/{id}` 엔드포인트로 잘못된 QA Pair·오류 시딩 데이터를 직접 삭제 가능 → 비즈니스 용어·Athena 규칙·정책 데이터 CRUD 완성. 나쁜 예시가 누적되면 정확도가 역으로 저하되는 문제 방지 |

### **Slide 15: 정량 검증 — Exec 정확도 4라운드 개선**
- **평가 지표**: **Exec (Execution Accuracy)** — 생성된 SQL의 실행 결과값이 정답과 일치하는가. 목표 ≥ 90%.
  - ※ EM(Exact Match, SQL 문자열 일치)은 같은 결과를 내는 다양한 표현이 존재해 지표로 활용 불가 판단, 중간에 제외.
- **4라운드 반복 개선**:

| 회차 | 주요 변경 | Exec | 발견한 문제 |
|------|----------|------|------------|
| **Round 1** | E2E 최초 실행 | **0%** | DDL이 LLM에 전달되지 않는 버그 발견 |
| **Round 2** | DDL 버그 수정 + Reranker 다국어 모델 교체 (jina-multilingual) | 베이스라인 확보 | 기존 Reranker가 DDL을 저평가 → 모델 교체로 해소 |
| **Round 3** | CoT 6단계 + Negative Rules + 시딩 재설계 (55개 QA, 패러프레이징) | **~91%** ✅ | GROUP BY 누락, DATE() 오남용 등 반복 패턴 → 프롬프트로 차단 |
| **Round 4** | cosine 거리 지표 전환 + Metadata Backtracking DDL 역추적 | **91%+ 안정화** ✅ | 임베딩 모델(ko-sroberta) 특성과 ChromaDB 지표 불일치 해소 |

- **발표자 멘트**: "0%에서 시작해 4번의 원인 추적과 구조 개선으로 목표치를 달성했습니다. 단순 프롬프트 튜닝이 아니라, 매 라운드마다 '왜 틀렸는가'를 파고들었습니다."

### **Slide 16: 대시보드 구축 (Why Redash?)**(시현)
- **Why?**:
  - Athena 데이터소스 네이티브 지원.
  - 마케터들이 자주 보는 핵심 지표(CTR, CVR, ROAS)를 한눈에 볼 수 있는 친화적 UI 제공.
- **S3 파티셔닝 구조 vs 대시보드 UX의 딜레마 극복**:
  - **문제**: 앞서 비용을 줄이기 위해 데이터 구조를 `year`, `month`, `day` 컬럼으로 조각냈습니다. 하지만 마케터는 Redash에서 직관적인 달력(Date Range UI)으로 "이번 주 월~일 데이터"를 조회하길 원합니다. 파티션을 타려면 쿼리 조건이 복잡해집니다.
  - **해결**: Redash 쿼리 안에서 `date_parse()` 함수와 CTE(Common Table Expression)를 조합해 UI의 날짜 파라미터(`{{start_date}}`)와 S3의 파티션 구조를 매핑하는 쿼리 템플릿을 개발했습니다. 마케터는 캘린더만 누르고, DB 내부에선 자동으로 파티션 프루닝이 동작하여 **성능(비용/속도)과 마케터의 UX 둘 다 타협 없이 제공**합니다.

### **Slide 18: 실시간 광고 이상 탐지 및 Slack 알림**(병훈)
- **핵심 메커니즘**: Kinesis로 들어오는 방대한 로그 데이터량을 **Airflow가 5분마다 자동 모니터링**합니다. 분석 알고리즘이 설정한 '정상 범위'를 벗어나는 즉시 **Slack으로 이상 징후 알림**을 보내 마케터가 24시간 대응할 수 있게 합니다.
- **이상 탐지 알고리즘 (Prophet + Isolation Forest 앙상블)**:
  - **Prophet (시계열 예측)**: 페이스북에서 오픈소스로 공개한 알고리즘으로, 트렌드와 계절성(매일의 점심/저녁 피크, 매주의 주말 효과)을 분해하여 학습합니다. 이를 통해 '지금 이 시점에 데이터가 있어야 할 정상 범위'를 예측합니다.
  - **Isolation Forest (아웃라이어 탐지)**: 데이터를 무작위로 분할했을 때, 정상 데이터보다 훨씬 '빨리 고립되는(Isolation)' 값이 이상치라는 원리를 이용합니다. 
  - **왜 앙상블인가?**: Prophet이 놓칠 수 있는 단기 급변값은 Isolation Forest가 잡아주고, Isolation Forest가 놓칠 수 있는 완만한 추세 변동은 Prophet이 잡아내어 **복합적인 이상 징후를 누락 없이 포착**합니다.
- **26년 2월 실제 데이터 기반 학습**: 배달앱 특유의 점심·저녁 피크 트래픽 패턴을 학습하여, 단순 수치 비교가 아닌 정교한 시계열 기반 이상을 판별합니다.
- **안정적 운영을 위한 기술 디테일**:
  - **Airflow KPO 실행**: 격리된 K8s 파드(Pod)에서 실행하여 라이브러리 간섭 없이 안정적으로 동작합니다.
  - **실전 데이터 보정**: 음수 예측값 방지 및 CloudWatch 수집 지연(2분 버퍼) 대응 로직을 적용해 탐지 정확도를 높였습니다.
- **최종 결과물**: 
  - **Slack 알림**: 이상치 발생 시 즉시 알림 발송.
  - **시각화 리포트**: 5분마다 갱신되는 시계열 그래프(`png`)와 상세 분석용 인터랙티브 차트(`html`) 자동 생성.

### **Slide 19: 인프라 코드화 + 트래픽 대응 (Terraform + EKS + Karpenter)**(현우)
- **왜 Terraform + EKS?**: 
  - **Terraform(IaC)**: 클릭 0회, 25분 만에 전체 인프라+앱 배포. 코드 기반 환경 재현성 100% 보장.
  - **EKS(Orchestration)**: 성격이 다른 5개 서비스(Airflow, Vanna, Bot 등)를 단일 클러스터에서 독립적으로 격리·운영하고, Karpenter와 결합하여 워크로드 기반 지능형 스케일링을 가능케 하는 핵심 엔진.
- **비용 최적화 — Karpenter (On-Demand + Spot 하이브리드)**:
  - On-Demand 고정 1대: Karpenter Controller, Airflow Scheduler 등 핵심 파드 전용.
  - Spot 노드 (Karpenter 자동 관리): Vanna API, Slack Bot 등 워커 파드 → On-Demand 대비 **최대 70% 저렴**.
  - Pending Pod 감지 후 **20~60초** 안에 신규 Spot 노드 자동 프로비저닝. 부하 해소 시 자동 반납(Consolidation).
- **2계층 Auto-Scaling**: HPA(파드 수평 확장, CPU 70%) → Karpenter(노드 자동 증설) 순서로 연쇄 동작.

### **Slide 20: AS-IS vs TO-BE 도입 전후 비교**
- **정량 성과 비교**:
  | 항목 | AS-IS (도입 전) | TO-BE (CAPA 도입 후) |
  | --- | --- | --- |
  | 데이터 요청~응답 시간 | 최소 1시간 ~ 3일 | **약 수십 초 내** |
  | 데이터팀 개입 여부 | 매 요청마다 필수 | **불필요** |
  | 리포트 작성 | 수기 엑셀 (수시간) | **자동 생성** |
  | 이상치 감지 | 사람이 직접 확인 | **실시간 자동 알림** |
  | 마케터 데이터 접근성 | SQL 지식 필요 | **자연어로 즉시 조회** |
- **결론**: 데이터 추출 업무 소요 시간 **약 99% 단축** (AS-IS 최소 1시간 = 3,600초 vs TO-BE 수십 초 기준). 마케터는 즉각적인 지표 확인을 통해 최적의 광고 집행 타이밍을 확보합니다.
- **한 줄 슬로건**: **"마케터는 오직 마케팅 본질에만 집중하십시오. 데이터 조회는 CAPA가 맡겠습니다."**

### **Slide 23: 시행착오 & 아쉬운 점 (Lessons Learned)**(모두)
- **시행착오 1 - EKS StorageClass 이슈**: EBS CSI Driver 버전 차이로 인한 PVC Pending 문제 발생 → Terraform Add-on 코드화로 자동화 해결.
- **시행착오 2 - Nested 쿼리 SQL 오류**: 복잡한 하위 질의 생성 시 할루시네이션 발생 → 집계용 뷰(View) 테이블 사전 생성 및 Prompt Engineering 강화로 정확도 개선.
- **시행착오 3 - Reranker 도입과 롤백 (정확도 vs 속도)**:
  - **상황**: 초기 정답률 0% 해결을 위해 고성능 Reranker(Jina) 도입 → 정확도는 대폭 상승(90%대)했으나 CPU 환경에서 리랭킹에만 57초 소요.
  - **결단**: 임베딩 모델 및 시딩 데이터 최적화로 리랭커 없이도 유사도 0.88 확보. 미미한 정확도 이득(2~3%)보다 응답 속도 45초 단축이 UX 측면에서 월등히 가치 있다고 판단하여 최종 롤백.
- **아쉬운 점**:
  - 평가 체계(Spider EM/Exec) 구축이 일정상 완전히 마무리되지 못함.
  - 시뮬레이션 로그 기반 검증이라 실 서비스 적용 시 추가 튜닝 필요.
  - **통계 집계 주기 한계**: CTR/CVR 등 지표 조회에 사용하는 Summary 테이블이 1시간~하루 단위로 갱신됩니다. 로그 수집 파이프라인 자체는 준실시간이지만, 마케터가 조회하는 통계 지표는 최신 집계 기준이 아닐 수 있습니다. 집계 주기 단축은 향후 과제로 남아 있습니다.
  - **RAG 성능 한계**: 복잡한 비즈니스 로직이나 예외 케이스를 포함한 질문의 경우, LLM이 잘못된 SQL을 생성할 확률이 여전히 존재합니다. (Exec 기준 평균 91%, 복잡 질문 유형에서 저하)
  - **프롬프트·시딩의 본질적 한계**: Negative Rules와 CoT로 반복 실수 패턴은 차단할 수 있지만, 규칙에 없는 새로운 오류 유형에는 여전히 취약합니다. 서브쿼리·HAVING 조합처럼 복잡한 SQL 패턴은 프롬프트 제약만으로 커버하기 어렵습니다.
  - **시딩 데이터 편향**: 학습 QA Pair가 많은 영역(단순 지표 조회, CTR/ROAS 중심)에서는 정확도가 높지만, 시딩 커버리지 밖의 질문 유형은 성능을 보장할 수 없습니다. 결국 시딩 데이터의 품질과 다양성이 시스템 상한선을 결정합니다.
  - **비용**: Claude API 사용으로 인해 비용이 발생합니다. (평균 0.002$/query)
