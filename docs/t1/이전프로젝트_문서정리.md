    ### **1. Helm + Kubernetes Provider 통합 ⭐⭐⭐**
    
    ```
    markdown
    
    **문제**: Milvus, Airflow 같은 복잡한 Kubernetes 앱 배포
    
    **Terraform 없었다면**:
    - Helm 수동:`helm install milvus ...`
    - kubectl 수동:`kubectl apply -f consumer.yaml`
    - 인프라(AWS)와 앱(K8s) 관리 도구 분리
    - 배포 순서 수동 관리 (EKS 먼저, 그다음 Helm)
    
    **Terraform으로 해결**:
    -`helm_release` 리소스로 Helm Chart 자동 배포
    -`kubernetes_deployment`로 Pod 자동 생성
    - EKS 생성 → Helm 배포 의존성 자동
    - 단일`terraform apply`로 끝
    
    ```
    
    → **우리 프로젝트 핵심**: 인프라 + 앱을 하나의 코드로 관리
    
    ---
    
    ---
    
    ### **3. 멱등성 + Plan 미리보기 🔍**
    
    ```
    markdown
    
    **문제**: 배포 실패 시 복구, 실수 방지
    
    **수동 AWS 콘솔이었다면**:
    - 실패 시 어디까지 만들어졌는지 모름
    - 재시도하면 중복 생성 에러
    - 삭제할지 말지 판단 어려움
    - 실수로 Production 삭제 위험
    
    **Terraform으로 해결**:
    -`terraform plan`으로 변경사항 미리 확인
    - apply 실패 시 다시 apply만 하면 됨 (멱등성)
    - State로 현재 상태 정확히 파악
    - 삭제할 리소스 빨간색으로 경고
    
    ```
    
    → **우리 프로젝트 핵심**: 안전한 배포, 실수 방지

- EKS 상세 설명
    
    **데몬셋 이란? 클러스터의 모든 노드(또는 일부 노드)에 파드(Pod)의 복제본을 하나씩 실행하도록 보장하는 컨트롤러**입니다. 새로운 노드가 클러스터에 추가되면 자동으로 파드가 배포되고, 노드가 제거되면 파드도 함께 삭제되는 특징이 있어 로깅, 모니터링, 네트워킹과 같이 노드당 하나만 필요한 백그라운드 작업에 주로 사용
    
    ### **1. IRSA (IAM Roles for Service Accounts) ⭐⭐⭐**
    
    ```
    markdown
    
    **문제**: Consumer, Airflow가 Kinesis, S3에 접근해야 함
    
    **EKS 없었다면**:
    - AWS Access Key를 코드에 하드코딩
    - Git에 Credentials 노출 위험
    - 모든 Pod가 같은 권한 사용
    
    **EKS IRSA로 해결**:
    - Consumer는 Kinesis 읽기만
    - Airflow는 S3 읽기/쓰기만
    - Credentials 노출 0건
    
    ```
    
    → **우리 프로젝트 핵심**: 로그 파이프라인이 AWS 서비스와 안전하게 통신
    
    ---
    
    ### **2. Helm Chart 지원 🎯**
    
    ```
    markdown
    
    **문제**: Milvus, Airflow 같은 복잡한 앱 배포
    
    **YAML 직접 작성했다면**:
    - Milvus Deployment, Service, PVC, ConfigMap 등 500줄+
    - Airflow Scheduler, Webserver, Worker, DB 등 1000줄+
    - 설정 하나 바꾸려면 여러 파일 수정
    
    **Helm으로 해결**:
    -`helm_release` 리소스 10줄
    - Values 파일만 수정하면 끝
    - Terraform으로 자동 배포
    
    ```
    
    → **우리 프로젝트 핵심**: 5일 안에 복잡한 앱 4개 배포 가능
    
    ---
    
    ### **3. DaemonSet (모든 노드 자동 배포) 📡**
    
    ```
    markdown
    
    **문제**: 모든 노드에서 로그 수집 필요
    
    **일반 EC2였다면**:
    - 노드 2개면 Fluent Bit 수동 설치 2번
    - 노드 추가되면 또 수동 설치
    - 설정 변경 시 모든 노드에 재배포
    
    **DaemonSet으로 해결**:
    - 노드 추가되면 Fluent Bit 자동 실행
    - 설정은 ConfigMap 1번만 수정
    - 모든 노드에 동시 적용
    
    ```
    
    → **우리 프로젝트 핵심**: 로그 수집기가 자동으로 모든 노드에 배포

## **3. EKS Cluster 배포(필수)**

### **제목**

**"EKS 인프라 자동 배포: 06-iam.tf → 05-eks.tf 순차 생성"**

### **슬라이드 내용**

**EKS Terraform 배포 순서 (`05-eks.tf`)**

| 단계 | 리소스 | 생성 내용 | 배포 방식 |
| --- | --- | --- | --- |
| **1단계** | VPC/Subnet (Data Source) | 기본 VPC 및 서브넷 조회 | 기존 리소스 참조 |
| **2단계** | `aws_eks_cluster.main` | EKS Cluster 1.29- API/Audit/Authenticator 로깅- Public + Private 엔드포인트 | **~8분 소요**IAM Role 의존 |
| **3단계** | `aws_eks_node_group.main` | Node Group (t3.medium)- Min/Desired/Max: 2/2/4- On-Demand 인스턴스 | **~4분 소요**Cluster 생성 후 |
| **4단계** | `aws_eks_access_entry` | 팀원 EKS 등록- IAM ARN → EKS 연결 | for_each로 병렬 생성 |
| **5단계** | `aws_eks_access_policy_association` | 팀원별 Admin 권한 부여- AmazonEKSClusterAdminPolicy 연결 | Access Entry 생성 후 |
| **6단계** | `aws_iam_openid_connect_provider` | OIDC Provider 생성- **IRSA 기반 구축**- EKS ↔ IAM 신뢰 관계 설정 | **모든 IRSA의 전제조건**06-iam.tf 7개 Role이 참조 |
| **7단계** | `aws_eks_addon.ebs_csi` | EBS CSI Driver Addon- PVC/StorageClass 지원 | Node Group 준비 후 |

**의존성 체인 (Terraform `depends_on`)**

```
06-iam.tf (IAM Roles)
    ↓
05-eks.tf (EKS Cluster) ← 8분
    ↓
Node Group ← 4분
    ↓
OIDC Provider → 모든 IRSA Role 연동
    ↓
Access Entry (팀원 접근)
    ↓
EBS CSI Addon (스토리지)

```

**핵심 수치**

- **총 배포 시간**: **~12분** (EKS Cluster 8분 + Node Group 4분)
- **리소스 개수**: 7개 (Cluster, Node Group, OIDC, Access Entry×2, Policy×2, Addon)
- **Auto-scaling 범위**: 2 → 4 노드 (Cluster Autoscaler 연동)
- **팀원 추가**: `terraform.tfvars`에 ARN 추가 → `apply` 1번

**배포 후 즉시 사용 가능**

```bash
# 1. kubectl 설정 (1회)
aws eks update-kubeconfig --name cali-cluster --region ap-northeast-2

# 2. 즉시 확인
kubectl get nodes
# NAME                          STATUS   AGE
# ip-172-31-47-63...internal    Ready    5d
# ip-172-31-62-32...internal    Ready    5d

```

- **EKS Cluster 생성 (8분)**:

## **3. IAM 권한 설계(빼기)**

### **제목**

**"IRSA와 Least Privilege: 보안 사고를 사전에 차단"**

### **슬라이드 내용**

**왜 IRSA와 Least Privilege인가?**

| 기술 | 해결하는 문제 | 사용 안 하면? (위험) |
| --- | --- | --- |
| **IRSA** | Pod마다 다른 AWS 권한 필요(Consumer는 Kinesis 읽기Airflow는 S3 쓰기) | ❌ AWS Access Key 하드코딩❌ Git 노출 위험❌ 모든 Pod 동일 권한 사용 |
| **Least Privilege** | 해킹/실수 시 피해 최소화(Firehose는 S3 쓰기만) | ❌ 관리자 권한 남발❌ 실수로 전체 S3 삭제 가능❌ 해킹 시 모든 리소스 접근 |

**IAM Role 구성 (Least Privilege 적용)**

| Pod / Service | ServiceAccount | IAM Role | 권한 범위 |
| --- | --- | --- | --- |
| **Consumer** | `consumer-sa` | `cali-app-role` | Kinesis 읽기, S3 쓰기(리소스: `cali-*`만) |
| **Airflow** | `airflow-*` (5개) | `cali-airflow-role` | S3 읽기/쓰기(버킷: `cali-logs-*`만) |
| **Grafana** | `grafana` | `cali-grafana-role` | OpenSearch 읽기 전용 |
| **Cluster Autoscaler** | `cluster-autoscaler` | `cali-autoscaler-role` | Auto Scaling 제어(태그: `cali-*`만) |
| **Firehose** | (AWS Service) | `cali-firehose-role` | S3 `PutObject`만(읽기/삭제 불가) |

**핵심 수치**

- **IAM Role 개수**: 7개 (역할별 완전 분리)
- **하드코딩된 AWS Credentials**: **0개** (IRSA 100% 적용)
- **관리자 권한 사용**: **0건** (Least Privilege 100%)
- **리소스 제한**: 모든 정책에 `cali-*` ARN 명시

### **발표자 노트 (3분)**

- **IRSA 강조**: "Pod에 AWS Key를 환경 변수로 넣지 않습니다. OIDC를 통해 임시 자격 증명을 자동 발급받습니다"
- **실제 예시**: "만약 Consumer Pod이 해킹당해도, Kinesis와 S3만 접근 가능하고 EC2나 RDS는 건드릴 수 없습니다"

## 5. 문제 해결 전략(개인별 에필로그에 넣기)

- 설명 어려운 버전
    
    ### 1. EBS CSI Driver 이슈
    
    **이슈**: Milvus와 Airflow Pod가 `Pending` 상태에서 벗어나지 못함
    
    **원인**: PersistentVolumeClaim(PVC) 생성 시 EBS 볼륨을 자동으로 프로비저닝할 수 있는 CSI Driver가 설치되지 않음
    
    **해결**: EKS Addon으로 EBS CSI Driver를 설치하고, EKS Node에 `AmazonEBSCSIDriverPolicy` IAM 권한 부여
    
    **교훈**: Stateful 워크로드는 스토리지 프로비저너가 필수이며, EKS에서는 CSI Driver를 명시적으로 설치해야 함
    
    ### 2. OpenSearch FGAC 이슈
    
    **이슈**: Firehose에서 OpenSearch로 데이터 전송 시 `403 security_exception` 에러 발생
    
    **원인**: AWS IAM `access_policies`와 별개로 OpenSearch Security Plugin의 내부 Role Mapping 설정이 필요함 (2단계 보안 계층)
    
    **해결**: Terraform Provisioner를 사용하여 kubectl로 임시 Pod를 생성하고, OpenSearch API를 호출하여 Firehose IAM Role을 `all_access` 그룹에 매핑
    
    **교훈**: 관리형 서비스도 내부 보안 메커니즘을 이해해야 하며, Terraform만으로 해결 안 될 때는 API 직접 호출 등 유연한 접근 필요
    
    ### 3. EKS 클러스터 접근 권한 관리
    
    **이슈**: 팀원들이 `kubectl` 명령어 실행 시 `Unauthorized` 또는 `Access Denied` 에러 발생
    
    **원인**: EKS 클러스터는 기본적으로 생성한 IAM 사용자/Role만 접근 가능하며, 추가 팀원의 IAM Principal을 명시적으로 등록해야 함
    
    **해결**: Terraform의 `aws_eks_access_entry` 리소스를 사용하여 팀원들의 IAM User ARN을 클러스터에 등록하고, `AmazonEKSClusterAdminPolicy`를 부여하여 전체 관리 권한 부여
    
    **교훈**: EKS는 AWS IAM과 Kubernetes RBAC이 통합되어 있어, 팀 협업 시 Access Entry를 코드로 관리하면 수동 설정 없이 자동으로 권한 부여 가능
    

### 1. EBS CSI Driver 이슈

**이슈**: Milvus와 Airflow가 계속 "준비 중" 상태로 멈춤

**원인**: 데이터 저장할 디스크(EBS)를 자동으로 만들어주는 프로그램이 설치 안 됨

**해결**: EBS 자동 생성 프로그램(CSI Driver) 설치 + 권한 부여

**교훈**: 데이터를 저장하는 앱은 디스크가 필요한데, EKS는 자동으로 안 만들어줘서 직접 설정 필요

### 2. OpenSearch FGAC 이슈

**이슈**: Firehose가 OpenSearch에 데이터 전송할 때 "접근 거부됨" 에러

**원인**: AWS 권한 설정과 OpenSearch 내부 권한 설정이 따로 있는데, 둘 다 설정해야 함 (문 2개를 다 통과해야 함)

**해결**: Terraform으로 안 되는 부분은 직접 명령어로 OpenSearch에 권한 추가

**교훈**: AWS 서비스도 내부에 자체 보안 시스템이 있을 수 있어서, AWS 권한만으론 부족할 수 있음

### 3. 팀원 EKS 접근 권한 설정

**이슈**: 팀원이 `kubectl get pods` 명령어를 실행하면 "권한 없음" 에러

**원인**: EKS는 처음 만든 사람만 접근 가능하고, 다른 팀원은 자동으로 접근 불가

**해결**: Terraform 코드에 팀원 AWS 계정 정보를 추가해서 자동으로 권한 부여

**교훈**: 팀원 추가할 때마다 수동으로 설정하지 않고, 코드에 한 줄만 추가하면 끝
