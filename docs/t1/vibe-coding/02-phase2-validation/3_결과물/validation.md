## 2. 상세 검증 결과

### 2.1 목표 및 범위 검증

#### ✅ 적합한 부분
- **Data Foundation**: `project_concept_v3.md`에서 강조한 "신뢰성"을 위해 Core Pipeline을 보수적으로 설계한 점은 컨셉과 완벽히 일치합니다.
- **AI-Assisted Analytics**: "분석 경험 개선"을 위해 Text-to-SQL(Vanna)을 도입하고, 이를 격리(Lab)하여 리스크를 관리한 전략은 매우 논리적입니다.

#### ⚠️ 개선 필요 (불일치)
- **Report & Dashboard 누락**: `project_concept_v3.md`의 MVP 범위에는 **📝 Report (자동 리포트)**와 **📊 Dashboard (Redash)**가 포함되어 있으나, `plan-draft.md`의 WBS와 범위(In-Scope)에서는 이들이 명시적으로 보이지 않거나 제외되었습니다. 의도된 범위 축소인지 확인이 필요합니다.

### 2.2 기술적 타당성 검증

#### 기술 스택 평가

| 기술 | 적합성 | 리스크 | 대안 검토 |

## 3. 필수 수정 사항 (Major Issues)

### 🟠 Major Issues (강력 권고)

1.  **[MI-001] MVP 범위 불일치 (Report/Dashboard)**
    -   **문제**: 기반 문서(`project_concept_v3.md`)의 MVP 4대 기능 중 'Alert', 'Report', 'Dashboard'가 계획서(`plan-draft.md`)의 실행 계획(WBS)에서 구체화되지 않음. 특히 **Redash 대시보드**와 **주간 자동 리포트** 구현 태스크가 누락됨.
    -   **영향**: 프로젝트 완료 시, 이해관계자가 기대한 MVP 기능 세트와 실제 산출물 간의 괴리 발생 가능.
    -   **수정 방안**:
        -   안 1: MVP 범위에서 해당 기능을 명시적으로 제외(Out of Scope)한다고 선언.
        -   안 2: WBS에 'Redash 구성' 및 'Basic Reporting' 태스크 추가 (약 3~4일 소요 예상).

2.  **[MI-002] 이상 탐지(Alert) 구현 구체성 부족**
    -   **문제**: 컨셉 문서에서는 CloudWatch 기반의 비즈니스 알림(로그 급감, CTR 급락 등)을 MVP로 정의했으나, 계획서에는 'Disaster Recovery' 관점의 시스템 알림만 강조됨.
    -   **영향**: 비즈니스 담당자가 기대하는 "선제적 대응" 가치 전달 실패.
    -   **수정 방안**: 'F-03' 또는 별도 태스크에 **CloudWatch Business Metrics & Alarm 설정**을 명시.
