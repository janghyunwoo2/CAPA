## 1. 목표 정의

### 1.1 핵심 목표
본 프로젝트의 초기 단계에서는 **안정성**과 **신뢰성**을 최우선으로 고려하여, 데이터 파이프라인의 견고한 기반을 구축하는 것에 집중합니다. AI 기능 도입 전, 데이터 정합성과 시스템 가용성을 100% 확보하는 것을 목표로 합니다.

```
[목표 요약]
- **안정적 데이터 수집**: Kinesis 및 Firehose를 활용한 무중단 로그 수집 파이프라인 구축 (데이터 손실 0%)
- **데이터 정합성 보장**: S3 및 Glue Catalog 기반의 명확한 데이터 스키마 및 적재 프로세스 확립
- **기본 연동 검증**: Slack Bot과 백엔드 간의 단순 연결성 확인 (복잡한 AI 로직 제외)
```

### 1.2 성공 기준 (Definition of Done)
모든 성공 기준은 정량적으로 측정 가능해야 하며, 보수적으로 설정합니다.

| 기준 | 측정 방법 | 목표값 (보수적) |
|------|----------|--------|
| 데이터 유실률 | Kinesis Producer/Consumer 메트릭 비교 | 0% (완전 무결) |
| 파이프라인 지연 | 로그 생성 시점 ~ Athena 조회 가능 시점 | < 5분 (P99) |
| 시스템 가용성 | 주요 컴포넌트(Kinesis, Athena, API) Uptime | 99.9% 이상 |
| 쿼리 정확도 | 샘플 데이터 100건에 대한 Manual SQL 검증 | 100% 일치 |
| 롤백 테스트 | 배포 실패 시 이전 버전 복구 시간 | < 10분 |

### 1.3 범위 (Scope)
불확실성이 높은 AI/ML 요소는 초기 범위에서 배제하거나 격리하여 리스크를 통제합니다.

**포함 (In Scope)**:
- [ ] Kinesis Data Stream (Provisioned Mode) 및 Firehose 설정
- [ ] S3 Data Lake (Parquet 포맷) 및 파티셔닝 구조 설계
- [ ] AWS Glue Crawler 및 Catalog 테이블 정의
- [ ] AWS Athena 기본 쿼리 환경 구성
- [ ] Slack Bot 기본 응답 (Echo) 및 Health Check 기능
- [ ] CloudWatch 대시보드 및 필수 알람(Error Rate, Lag) 구성

**제외 (Out of Scope)**:
- [ ] Vanna/LLM 기반 Text-to-SQL (검증 단계로 이관)
- [ ] Prophet 기반 시계열 예측 (단순 임계값 알람으로 대체)
- [ ] 자동화된 리포트 생성 (수동 쿼리로 대체 후 점진적 도입)
- [ ] 복잡한 사용자 세션 관리

## 2. 배경 및 문제 정의

### 2.1 현재 상황
현재 광고 로그 데이터는 대규모로 생성되고 있으나, 분석을 위한 접근성이 낮고 수동 작업에 의존하고 있습니다. 이는 데이터 팀의 업무 과부하와 휴먼 에러 가능성을 내포하고 있습니다.

### 2.2 해결해야 할 문제
급격한 신기술 도입보다는, 현재의 비효율을 **안전하게** 제거하는 것이 우선입니다.

| 문제 | 영향 | 우선순위 |
|------|------|----------|
| 데이터 접근 지연 | 의사결정 속도 저하, 비즈니스 기회 손실 | 높음 |
| 수동 SQL 작성 오류 | 데이터 신뢰도 하락, 잘못된 의사결정 유발 | 높음 |
| 시스템 불안정성 우려 | 신규 시스템 도입 시 장애 발생 가능성 | 중간 |

## 3. 기술 스택 및 아키텍처 방향

### 3.1 사용 기술
최소 3년 이상 검증된 기술과 관리형 서비스를 우선 선택하여 운영 부담과 장애 가능성을 최소화합니다.

| 영역 | 기술 | 버전 | 선택 이유 |
|------|------|------|-----------|
| 언어 | Python | 3.11 (LTS) | 최신 기능보다 안정성 및 라이브러리 호환성 우선 |
| 데이터 수집 | Kinesis Data Stream | Provisioned | On-demand의 예측 불가능한 스케일링 지연 방지 및 비용 예측 용이 |
| 데이터 적재 | Kinesis Firehose | Standard | 서버리스, 완전 관리형으로 운영 복잡도 최소화 |
| 저장소 | Amazon S3 | Standard | 내구성 99.999999999% 보장, 버전 관리 활성화 필수 |
| 메타데이터 | AWS Glue | Catalog | Hive Metastore 호환 및 Athena 통합 최적화 |
| 분석 | Amazon Athena | V3 | 서버리스 SQL 엔진, 검증된 안정성 |
| 인터페이스 | Slack Bolt | Stable | 널리 사용되는 안정적인 Slack SDK |

### 3.2 아키텍처 개요
복잡도를 낮추고 데이터 흐름을 명확히 하기 위해 단방향 파이프라인을 채택합니다.

```
[보수적 데이터 파이프라인 아키텍처]

[Log Generator] 
      │ (Reliable Ingestion)
      ▼
[Kinesis Data Stream] (Shard 모니터링 필수)
      │
      ▼
[Kinesis Firehose] (Buffer: 5분/128MB)
      │
      ▼
[Amazon S3] (Raw/Parquet) <── [Glue Catalog]
      │
      ▼
[Amazon Athena] (Standard SQL)
      │
      ▼
[Slack Bot / API] (Read-Only 접근 원칙)
```

## 5. 리스크 및 대응 방안
발생 가능한 모든 기술적, 운영적 리스크를 사전에 식별하고 구체적인 대응책을 마련합니다.

| 리스크 | 영향도 | 발생 확률 | 대응 방안 (Mitigation) |
|--------|--------|----------|-----------|
| **데이터 스키마 변경** | 높음 | 높음 | Glue Schema Registry 도입, 하위 호환성 유지 원칙 준수 |
| **Kinesis 샤드 스로틀링** | 중간 | 중간 | 적절한 파티션 키 설계, 과부하 시 알람 및 Auto-scaling 정책 보수적 설정 |
| **S3 비용 증가** | 낮음 | 중간 | Lifecycle Policy 조기 적용, 불필요한 버전 관리 주기 단축 |
| **쿼리 성능 저하** | 중간 | 중간 | 파티셔닝(일/시간) 철저 적용, 쿼리 스캔 제한 설정 |
| **API 키 노출** | 치명적 | 낮음 | AWS Secrets Manager 연동, 코드 내 하드코딩 절대 금지 |
