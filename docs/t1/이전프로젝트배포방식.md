# CALI 인프라: Terraform 기반 EKS & Helm 배포 구현 방식

네, 기억하신 대로 **별도의 쉘 스크립트 없이 Terraform만으로 EKS 생성부터 애플리케이션(Helm Chart) 배포까지 모두 자동화**되어 있습니다.

핵심 원리는 **`providers.tf`에서 EKS 클러스터 정보를 동적으로 가져와 `helm` 프로바이더에 주입**하고, **`10-helm-releases.tf` 파일에서 `helm_release` 리소스**를 사용하는 것입니다.

## 1. 구현 원리 요약

1.  **`05-eks.tf`**: EKS 클러스터를 생성합니다.
2.  **`providers.tf`**: 생성된 EKS 클러스터의 엔드포인트와 인증 토큰을 `kubernetes` 및 `helm` 프로바이더에게 전달합니다.
3.  **`10-helm-releases.tf`**: 설정된 `helm` 프로바이더를 사용하여 Terraform이 직접 Helm Chart를 EKS에 배포합니다.

이 방식 덕분에 `kubectl`이나 `helm` 명령어를 따로 실행할 필요 없이, `terraform apply` 한 번으로 인프라와 앱이 모두 배포됩니다.

---

## 2. 파일별 상세 구현 내용

### A. `infra/terraform/providers.tf` (핵심 연결 고리)
Terraform이 AWS 리소스(EKS)를 생성한 후, 그 정보를 이용해 바로 Kubernetes/Helm 프로바이더를 초기화하는 부분입니다.

```hcl
# EKS 클러스터 정보를 데이터 소스로 가져옴
data "aws_eks_cluster" "cluster" {
  name = aws_eks_cluster.main.name
}

data "aws_eks_cluster_auth" "cluster" {
  name = aws_eks_cluster.main.name
}

# Helm 프로바이더 설정 (EKS 접속 정보 주입)
provider "helm" {
  kubernetes {
    host                   = data.aws_eks_cluster.cluster.endpoint
    cluster_ca_certificate = base64decode(data.aws_eks_cluster.cluster.certificate_authority[0].data)
    token                  = data.aws_eks_cluster_auth.cluster.token
  }
}
```

### B. `infra/terraform/10-helm-releases.tf` (배포 정의)
`helm install` 명령어 대신 Terraform 리소스인 `helm_release`를 사용합니다.

```hcl
resource "helm_release" "airflow" {
  name             = "airflow"
  repository       = "https://airflow.apache.org"
  chart            = "airflow"
  version          = "1.15.0"
  namespace        = "airflow"
  create_namespace = true

  # values.yaml 파일을 읽어서 설정을 주입
  values = [
    file("${path.module}/../helm-values/airflow.yaml")
  ]

  # EKS 클러스터 생성이 완료될 때까지, 또는 특정 리소스가 생성될 때까지 대기
  depends_on = [
    kubernetes_storage_class.gp2,
    kubernetes_config_map.airflow_requirements
    # (필요 시 EKS 모듈 자체를 depends_on으로 걸 수도 있음)
  ]
}
```

### C. `infra/terraform/11-k8s-apps.tf` (K8s Manifest)
Helm 차트 외에 직접 작성한 K8s Manifest(예: Secret, Custom Resource 등)도 Terraform으로 관리합니다.

```hcl
resource "kubernetes_manifest" "example" {
  manifest = yamldecode(file("${path.module}/../k8s/example.yaml"))
}
```

## 3. 이 방식의 장점
1.  **단일 진실 공급원 (Single Source of Truth)**: 인프라와 애플리케이션 상태가 모두 Terraform State(`tfstate`)에 저장됨.
2.  **의존성 관리**: Terraform의 `depends_on`을 통해 "EKS 생성 -> Node Group 생성 -> Helm 배포" 순서를 명확히 제어 가능.
3.  **멱등성 (Idempotency)**: 여러 번 실행해도 변경된 부분만 적용되므로 안전함. 스크립트는 매번 실행 여부를 체크하는 로직을 짜야 함.

다른 프로젝트에 적용하실 때도 이 구조(`providers.tf` 설정 + `helm_release` 리소스)를 그대로 참고하시면 됩니다.
